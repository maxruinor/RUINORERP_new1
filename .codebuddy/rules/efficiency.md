# 开发效率与成本控制规则

## 核心原则

1. **直接解决问题，避免绕圈子**
   - 不要反复尝试无效的解决方案
   - 如果某个方案失败2次以上，立即换一种思路
   - 相信用户的反馈，如果用户说某个方案无效，不要重复尝试

2. **节省 Token 和信用点数**
   - 避免输出冗长的解释和说明
   - 代码修改后，只报告关键结果
   - 不要重复已经确认的信息
   - 优先使用简洁的技术术语

3. **任务完成后，不要输出一堆没用的内容**
   - 只输出必要的信息
   - 不需要总结、回顾或多余说明
   - 用户只关心结果，不关心过程

4. **代码修改原则**
   - 保持代码简洁，删除不必要的注释
   - 移除调试代码和测试代码
   - 不要保留被注释掉的代码块
   - 一次修改到位，避免多次小修小改

5. **问题排查原则**
   - 先理解问题的根本原因
   - 不要猜测，要基于事实和错误信息
   - 如果用户提供了有效的解决方案，直接采用
   - 避免"可能"、"也许"等不确定的表述

## 具体实践

### 日志系统调试
- 遇到数据库类型转换错误时，优先考虑源头处理
- 在设置参数时直接使用 DBNull.Value 而不是 null
- 简化自定义 Layout，避免过度设计
- 配置文件修改后，验证一次即可

### 代码提交规范
- 提交信息简洁明了，说明改动了什么
- 不要提交调试代码
- 保持代码整洁，删除无用注释

### 沟通效率
- 用户说"不要输出一堆没用的"，就立即停止详细说明
- 用户说"不要浪费我的 token"，就直接给出解决方案
- 相信用户的判断，不要质疑用户的反馈
