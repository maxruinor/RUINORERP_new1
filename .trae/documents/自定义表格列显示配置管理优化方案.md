# 自定义表格列显示配置管理优化方案

## 一、现状分析

### 1.1 功能概述

本系统提供两个列显示配置入口，用于管理DataGridView表格列的显示/隐藏、排序等个性化设置：

1. **入口一**：NewSumDataGridView右键菜单"自定义显示列"功能
2. **入口二**：工具栏"属性"按钮（frmGridViewColSetting窗体）

### 1.2 相关文件清单

| 文件路径 | 说明 |
|---------|------|
| `RUINORERP.UI\UControls\NewSumDataGridView.cs` | 自定义DataGridView控件，包含右键菜单 |
| `RUINORERP.UI\UControls\CustomizeGrid.cs` | 列自定义配置按钮控件 |
| `RUINORERP.UI\ForCustomizeGrid\frmColumnsSets.cs` | 列设置窗体（入口一） |
| `RUINORERP.UI\UserPersonalized\frmGridViewColSetting.cs` | 列设置窗体（入口二） |
| `RUINORERP.Model\tb_UIGridSetting.cs` | 数据库配置实体 |
| `RUINORERP.Model\tb_UIMenuPersonalization.cs` | 菜单个性化设置实体 |
| `RUINORERP.Business\tb_UIGridSettingController.cs` | 配置业务控制器 |
| `RUINORERP.IServices\Itb_UIGridSettingServices.cs` | 配置服务接口 |
| `RUINORERP.Services\tb_UIGridSettingServices.cs` | 配置服务实现 |

### 1.3 入口一实现逻辑（XML存储）

**调用流程**：
```
NewSumDataGridView右键菜单"自定义显示列"
    → NewSumDataGridView_自定义列事件
    → CustomizeGrid.SetColumns()
    → frmColumnsSets窗体（显示/隐藏复选框）
    → 确定保存 → SaveColumnsList() → XML文件
```

**关键代码**：
- `CustomizeGrid.NeedSaveColumnsXml`属性（默认false）决定是否持久化
- `CustomizeGrid.XmlFileNamecdc`属性指定XML文件名
- 存储路径：`Application.StartupPath + "\\ColumnsConfig\\"`

### 1.4 入口二实现逻辑（数据库存储）

**调用流程**：
```
工具栏"属性"按钮
    → frmGridViewColSetting窗体
    → "初始化列"按钮 → UIBizService.InitDataGridViewColumnDisplays()
    → "保存"按钮 → UIBizService.SaveGridSettingsAsync()
    → tb_UIGridSetting表
```

**关键代码**：
- `UIBizService.InitDataGridViewColumnDisplays()` - 初始化列配置
- `UIBizService.SaveGridSettingsAsync()` - 异步保存配置到数据库

## 二、详细实现分析

### 2.1 配置加载流程

#### 入口一（NewSumDataGridView + XML）加载流程：
```
1. NewSumDataGridView初始化 → BindColumnStyle()方法
2. 如果NeedSaveColumnsXml=true → 调用customizeGrid.LoadColumnsListByCdc()
3. 从XML文件加载配置 → 反序列化为List<ColDisplayController>
4. 调用BindColumnStyle(ColumnDisplays)应用配置到表格
```

**关键代码位置**：
- `NewSumDataGridView.cs` 第2321行：`if (NeedSaveColumnsXml) { ColumnDisplays = customizeGrid.LoadColumnsListByCdc(); }`
- `CustomizeGrid.cs` 第160-217行：`LoadColumnsListByCdc()`方法

#### 入口二（SourceGridHelper + 数据库）加载流程：
```
1. 窗体初始化 → UIBizService.SetCustomSourceGrid()
2. 从tb_UIGridSetting表读取ColsSetting字段
3. 反序列化为List<SGColDisplayHandler>（JSON格式）
4. 转换为List<ColDisplayController>应用到SourceGrid
```

**关键代码位置**：
- `UIBizService.cs` 第1754-1900行：SetCustomSourceGrid方法
- 读取配置：`JsonConvert.DeserializeObject<List<SGColDisplayHandler>>(GridSetting.ColsSetting)`

### 2.2 配置保存流程

#### 入口一保存流程：
```
1. 用户在frmColumnsSets中修改列配置
2. 点击确定 → btnOK_Click事件
3. 更新ColumnDisplays集合的Visible、ColDisplayIndex属性
4. 调用customizeGrid.SaveColumnsList(ColumnDisplays)
5. 序列化List<ColDisplayController>为XML并保存
```

**问题**：`NeedSaveColumnsXml`默认为false，导致配置仅存内存，重启丢失！

#### 入口二保存流程：
```
1. 用户在frmGridViewColSetting中修改列配置
2. 点击保存 → UIBizService.SaveGridSettingsAsync()
3. 序列化List<SGColDisplayHandler>为JSON
4. 调用SqlSugar更新tb_UIGridSetting.ColsSetting字段
```

### 2.3 ColDisplayController与SGColDisplayHandler关系

| 类名 | 用途 | 存储格式 |
|------|------|---------|
| `ColDisplayController` | NewSumDataGridView(KryptonDataGridView)使用 | XML序列化 |
| `SGColDisplayHandler` | SourceGrid(旧版表格)使用 | JSON序列化到数据库 |

**两者关系**：
- 功能相同，都是存储列显示配置信息
- 属性几乎相同（ColName、ColDisplayText、Visible、ColWidth等）
- 但类名不同，需要相互转换
- 转换代码在UIBizService中通过JSON序列化实现

### 2.4 菜单信息获取

**关键发现**：NewSumDataGridView本身没有MenuID属性，需要从父窗体获取。

获取方式（入口二示例）：
```csharp
// 从当前用户的个性化设置中获取
tb_UserPersonalized userPersonalized = MainForm.Instance.AppContext.CurrentUser_Role_Personalized;
tb_UIMenuPersonalization menuPersonalization = userPersonalized.tb_UIMenuPersonalizations
    .FirstOrDefault(t => t.MenuID == CurMenuInfo.MenuID);
```

### 2.5 两个入口的核心差异

| 差异项 | 入口一(NewSumDataGridView) | 入口二(SourceGrid) |
|--------|--------------------------|------------------|
| 控件类型 | KryptonDataGridView | SourceGrid.Grid |
| 配置类 | ColDisplayController | SGColDisplayHandler |
| 存储方式 | XML文件(可配置) | 数据库tb_UIGridSetting表 |
| 持久化开关 | NeedSaveColumnsXml属性 | 默认true |
| 菜单关联 | 无(仅用文件名) | 通过UIMenuPID关联 |
| 恢复默认 | 删除XML文件重建 | 调用InitDataGridViewColumnDisplays() |

### 2.6 存在问题分析

#### 2.6.1 双体系运行问题

| 特性 | 入口一(XML) | 入口二(数据库) |
|------|------------|--------------|
| 存储方式 | XML文件 | tb_UIGridSetting表 |
| 恢复默认 | 删除XML重建 | 调用InitDataGridViewColumnDisplays() |
| 持久化 | NeedSaveColumnsXml=false时仅内存 | 始终持久化 |

**问题**：两个入口使用完全独立的存储机制，修改一个不会影响另一个，导致配置不同步。

#### 2.6.2 配置丢失问题

1. **内存配置丢失**：当`NeedSaveColumnsXml=false`时，配置仅保存在内存中，应用重启后丢失
2. **列名匹配失败**：`ShowColumns()`方法通过`FieldNameList`匹配列名，未匹配到的列默认隐藏
3. **加载时机问题**：`DataSourceChanged`事件中重新加载配置可能覆盖用户设置

#### 2.6.3 代码冗余问题

1. 两套独立的列配置管理逻辑
2. 重复的初始化/恢复默认值实现
3. XML序列化/反序列化代码与数据库操作代码并存

## 三、优化目标

### 3.1 功能保留

- 两个入口均保持可用，不影响用户现有操作习惯

### 3.2 业务逻辑统一

- 建立统一的核心业务处理流程
- 消除逻辑分支和实现差异

### 3.3 系统简化

- 移除XML存储相关代码（作为兼容保留）
- 简化配置管理复杂度
- 提高代码可维护性

### 3.4 性能优化

- 实现批量、延时性的配置保存机制
- 减少频繁IO操作

## 四、技术方案

### 4.1 统一配置管理服务

创建`ColumnConfigManager`统一配置服务，核心功能：

```csharp
public interface IColumnConfigService
{
    // 加载配置（优先数据库，兼容XML）
    Task<List<ColDisplayController>> LoadColumnConfigAsync(string gridKeyName, long menuId);
    
    // 保存配置（延时批量提交）
    void SaveColumnConfig(string gridKeyName, long menuId, List<ColDisplayController> config);
    
    // 批量提交缓冲
    Task FlushChangesAsync();
    
    // 恢复默认
    Task ResetToDefaultAsync(string gridKeyName, long menuId);
}
```

### 4.4 性能优化设计

#### 4.4.1 缓存机制

```csharp
public class ColumnConfigCache
{
    private static readonly ConcurrentDictionary<string, List<ColDisplayController>> _cache = new();
    
    // 缓存Key格式: "{MenuID}_{GridKeyName}"
    public static List<ColDisplayController> Get(string key) => _cache.GetOrAdd(key, LoadFromDb);
    
    public static void Invalidate(string menuId) 
    {
        var keys = _cache.Keys.Where(k => k.StartsWith($"{menuId}_")).ToList();
        foreach (var key in keys) _cache.TryRemove(key, out _);
    }
}
```

#### 4.4.2 批量延时保存

```csharp
public class ColumnConfigManager
{
    private readonly ConcurrentQueue<ConfigChange> _pendingChanges = new();
    private readonly Timer _autoFlushTimer;
    
    public void SaveColumnConfig(...)
    {
        _pendingChanges.Enqueue(new ConfigChange(...));
        
        // 达到阈值立即保存
        if (_pendingChanges.Count >= 10)
            FlushChangesAsync().ConfigureAwait(false);
    }
}
```

## 六、影响评估

### 6.1 代码架构影响

| 影响项 | 影响程度 | 说明 |
|-------|---------|------|
| CustomizeGrid | 中 | 需修改SaveColumnsList/LoadColumnsListByCdc方法，改用数据库存储 |
| frmColumnsSets | 低 | 入口保持不变，底层存储方式变更 |
| frmGridViewColSetting | 低 | 已经是数据库存储，可复用统一服务 |
| UIBizService | 中 | 现有SaveGridSettingsAsync方法可迁移到统一服务 |
| NewSumDataGridView | 低 | ColumnDisplays属性保持不变，仅加载逻辑调整 |

### 6.2 性能影响

| 场景 | 当前实现 | 优化后 | 影响 |
|------|---------|--------|------|
| 首次加载 | XML文件IO | 数据库查询+缓存 | 略有增加(可忽略) |
| 配置保存 | 直接写入(XML) | 延时批量提交 | **显著减少IO** |
| 多用户并发 | 无冲突处理 | 数据库事务保护 | 更安全 |
| 配置查询 | 文件系统 | 内存缓存 | **更快** |

### 6.3 功能兼容性

- **向后兼容**：保留XML读取逻辑，现有配置不丢失
- **入口兼容**：两个配置入口保持不变，用户无感知
- **菜单关联**：XML方式改为数据库菜单关联，支持多用户个性化

### 6.4 数据迁移

### 6.5 前端渲染逻辑影响

| 影响点 | 说明 |
|-------|------|
| 列顺序 | 通过ColDisplayIndex控制，与现有逻辑一致 |
