# 单据状态流转与按钮控制功能完整规范 Spec

## Why
需要全面检查和验证现有状态流转功能的正确性，确保业务流程逻辑完整、权限控制严密、边界情况处理完善。

## What Changes
- **状态流转规则检查**：验证DataStatus所有状态之间的转换是否正确
- **按钮权限控制检查**：验证每个状态下按钮的Enabled/Visible状态
- **边界情况验证**：检查可能存在的逻辑漏洞
- **逆向操作验证**：确认反审核、反结案等逆向操作的正确性

## 当前系统实现分析

```csharp
public enum DataStatus
{
    草稿 = 1,    // 未提交状态
```

## 发现的问题

### 问题1: 完结状态的定义重复
**位置**: GlobalStateRulesManager.cs:268-272
**问题**: 代码中完结状态被定义了两次：
```csharp
// 完结态（已执行或已结案）：可反执行（转确认）、可反结案（转确认）
[DataStatus.完结] = new List<object> { DataStatus.确认 },

// 完结态：无任何转换可能, 终身锁定
[DataStatus.完结] = new List<object> { },
```
**影响**: 第二次定义会覆盖第一次，导致完结状态不能反结案
**建议**: 删除第二个定义，保留第一个

### 问题2: 完结状态缺少UI按钮规则
**位置**: InitializeDataStatusUIButtonRules
**问题**: 没有为完结状态定义按钮规则
**建议**: 添加完结状态的按钮规则

### 问题3: 撤销提交操作名称不一致
**位置**: GlobalStateRulesManager
**问题**: 按钮规则中使用的是`cancelSubmitEnabled`，但操作枚举中可能是`撤销提交`
**建议**: 确认操作枚举和按钮规则的命名一致性

## ADDED Requirements

### Requirement: 完结状态的完整支持
**描述**: 系统应正确支持完结状态的转换和按钮控制

#### Scenario: 完结状态的反结案操作
- **WHEN** 单据处于完结状态
- **THEN** 反结案按钮应启用
- **WHEN** 用户点击反结案
- **THEN** 单据状态变更为确认状态
- **THEN** 反结案按钮禁用，结案按钮启用

### Requirement: 保存按钮的数据变更检测
**描述**: 保存按钮应根据实体HasChanged属性动态调整启用状态

#### Scenario: 保存成功后的按钮状态
- **WHEN** 用户点击保存按钮
- **AND** 保存操作成功
- **THEN** HasChanged被重置为false
- **THEN** 保存按钮立即禁用（变灰色）

#### Scenario: 数据变更后的保存按钮状态
- **WHEN** 用户修改单据数据
- **AND** HasChanged变为true
- **AND** 单据处于可修改状态
- **THEN** 保存按钮自动启用

## 技术实现要点

### 1. 状态管理架构
- 使用GlobalStateRulesManager统一管理所有状态规则
- 通过UnifiedStateManager执行状态转换验证
- 利用IUnifiedStateManager接口进行状态控制

### 2. 按钮状态控制
- 权限控制：按钮的Enabled由用户权限决定
- 状态控制：在权限基础上，根据单据状态进一步控制
- 数据变更控制：保存按钮额外受HasChanged属性影响

### 3. 事件处理流程
- PropertyChanged事件 → 更新UI状态 → 调用UpdateAllUIStates
- UpdateAllUIStates → UpdateAllButtonStates → UpdateToolStripButtons
- UpdateToolStripButtons → 根据状态规则设置按钮Enabled

## 边界情况处理

### 1. 并发操作
- 防止多个用户同时修改同一单据
- 使用锁机制确保数据一致性

### 2. 网络延迟
- 保存操作添加loading状态
- 超时处理机制

### 3. 异常恢复
- 保存失败时保持保存按钮启用
- 状态栏显示错误信息
