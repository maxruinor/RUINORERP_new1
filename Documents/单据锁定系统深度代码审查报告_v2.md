```csharp
// ✅ 完善的缓存更新判断
private bool ShouldUpdateCache(LockInfo oldValue, LockInfo newValue)
{
    // 1. 旧值已过期 → 更新
    if (oldValue.IsExpired) return true;
    
    // 2. 新值更新时间更晚 → 更新
    if (newValue.LastUpdateTime > oldValue.LastUpdateTime) return true;
    
    // 3. 锁定状态变化 → 更新
    if (newValue.IsLocked != oldValue.IsLocked) return true;
    
    // 4. 锁定人变化 → 更新
    if (newValue.IsLocked && 
        (newValue.LockedUserId != oldValue.LockedUserId || 
         !string.Equals(newValue.LockedUserName, oldValue.LockedUserName)))
        return true;
    
    // 5. 关键业务信息变化 → 更新
    if (!string.Equals(newValue.BillNo, oldValue.BillNo) ||
        newValue.MenuID != oldValue.MenuID)
        return true;
    
    return false;
}
```

## 🔍 七、并发安全分析

### 7.1 锁竞争场景

#### 场景1: 同一单据被多个用户同时打开

**流程**:
1. 用户A打开单据 → `LockBillAsync` → 获取细粒度锁 → 锁定成功
2. 用户B打开同一单据 → `LockBillAsync` → 等待细粒度锁 → 超时或失败

**安全性**: ✅ 安全
- 细粒度锁确保同一时刻只有一个线程能操作该单据的锁状态
- 超时机制防止无限等待

#### 场景2: 解锁后立即查询

**流程** (修复前):
1. 线程A调用`UnlockBillAsync` → 移除`_billLocks[billId]` → 释放锁
2. 线程B调用`CheckLockStatusAsync` → `GetOrAdd`创建新锁对象 → 尝试获取
3. **问题**: 线程A的`finally`块可能还在执行，导致状态混乱

**流程** (修复后):
1. 线程A调用`UnlockBillAsync` → 释放锁 → `CleanupUnusedLock`延迟清理
2. 线程B调用`CheckLockStatusAsync` → 获取同一锁对象 → 正常执行
3. **结果**: ✅ 状态一致，无竞态条件

### 7.2 死锁风险评估

**潜在死锁点**:
1. ❌ ~~全局信号量嵌套细粒度锁~~ (已避免)
2. ❌ ~~锁对象在持有期间被移除~~ (已修复)
3. ✅ 定时器回调中的异步操作 (有超时保护)

**结论**: 修复后死锁风险极低。
