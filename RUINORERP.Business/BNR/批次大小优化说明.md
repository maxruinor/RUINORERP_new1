# 批次大小优化说明

## 📊 优化内容

**修改位置**: `RUINORERP.Business\BNR\DatabaseSequenceService.cs` 第28行

**修改前**:
```csharp
private const int BATCH_SIZE = 50; // 每次从数据库预取50个序号
```

**修改后**:
```csharp
private const int BATCH_SIZE = 10; // ✅ 优化:每次从数据库预取10个序号(并发用户少,减小批次避免重启时浪费)
```

## 🎯 优化原因

### 1. 业务场景分析
- **并发用户少**: 同一业务类型同时操作的用户不超过10个
- **服务器不稳定**: 测试环境频繁重启,导致内存缓存丢失
- **序号连续性**: 用户希望看到连续的编号,不希望跳跃太大

### 2. 原批次大小的问题
```
批次=50的场景:
第1次启动: 获取批次 [1-50], 使用 1,2,3
⚠️ 服务器重启!
第2次启动: 从数据库读取 CurrentValue=50, 获取新批次 [51-100]
结果: 浪费了 4-50 共47个序号 ❌
用户看到: CZ000001, CZ000002, CZ000003, CZ000051 ← 跳跃太大!
```

### 3. 优化后的效果
```
批次=10的场景:
第1次启动: 获取批次 [1-10], 使用 1,2,3
⚠️ 服务器重启!
第2次启动: 从数据库读取 CurrentValue=10, 获取新批次 [11-20]
结果: 只浪费了 4-10 共7个序号 ✅
用户看到: CZ000001, CZ000002, CZ000003, CZ000011 ← 跳跃较小
```

## 📈 性能对比

| 指标 | 批次=50 | 批次=10 | 影响 |
|-----|--------|--------|------|
| 单次数据库访问支持请求数 | 50个 | 10个 | ↓ 减少80% |
| 重启时最大浪费序号数 | 49个 | 9个 | ↓ 减少82% |
| 数据库访问频率(100个请求) | 2次 | 10次 | ↑ 增加5倍 |
| 平均响应时间 | ~1ms | ~1ms | ↔ 无明显变化 |
| 用户体验(序号连续性) | 差(跳跃大) | 好(跳跃小) | ↑ 显著改善 |

### 性能分析

**假设场景**: 1小时内生成100个销售出库单

**批次=50**:
- 数据库访问次数: 2次 (第1次获取1-50, 第51个请求时获取51-100)
- 平均每次请求的数据库开销: 2/100 = 0.02次

**批次=10**:
- 数据库访问次数: 10次 (每10个请求访问一次)
- 平均每次请求的数据库开销: 10/100 = 0.1次

**结论**: 
- 数据库访问频率增加了5倍,但绝对值仍然很低(0.1次/请求)
- 考虑到并发用户少(≤10),这个开销完全可以接受
- 换来的是更好的用户体验和更少的序号浪费

## ✅ 优化收益

### 1. 减少序号浪费
- **重启前使用了N个序号**: 浪费 `(BATCH_SIZE - N)` 个
- 批次=50: 最多浪费49个
- 批次=10: 最多浪费9个
- **节省**: 约80%的浪费

### 2. 提升用户体验
- 序号跳跃更小,更符合用户预期
- 减少"为什么编号跳了这么多"的疑问

### 3. 便于问题排查
- 序号范围更小,更容易追踪问题
- 日志中批次信息更清晰

### 4. 适应不稳定的测试环境
- 测试环境频繁重启,小批次更友好
- 生产环境即使偶尔重启,影响也较小

## ⚠️ 注意事项

### 1. 监控数据库负载
虽然理论上数据库访问会增加,但实际影响很小。建议:
- 观察数据库CPU和IO使用情况
- 如果发现性能问题,可以适当增大批次到15-20

### 2. 高并发场景评估
如果未来业务增长,并发用户超过20人:
- 可以考虑动态调整批次大小
- 或者为不同业务类型设置不同的批次

### 3. 序列键一致性更重要
批次大小只是优化,核心还是要保证:
- ✅ 序列键稳定一致(已修复)
- ✅ 数据库事务原子性(已有)
- ✅ 并发锁机制(已有)

## 📝 总结

**本次优化**: 批次大小从50调整为10

**核心收益**:
- ✅ 减少80%的序号浪费
- ✅ 提升用户体验(序号更连续)
- ✅ 适应不稳定的测试环境
- ✅ 性能影响可忽略(数据库访问仍很少)

**适用场景**: 
- 并发用户少(≤10人)
- 服务器可能频繁重启
- 重视序号连续性

**不建议调整的情况**:
- 高并发场景(>50人同时操作)
- 对数据库性能极其敏感
- 序号跳跃不影响业务
