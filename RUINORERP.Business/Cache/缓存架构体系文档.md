## 一、架构概述

### 1.1 设计目标

RUINORERP 缓存架构旨在提供：
- **高效的数据缓存**: 减少数据库访问，提高响应速度
- **多端数据一致性**: 确保服务器与多个客户端缓存同步
- **可扩展性**: 支持新缓存类型的灵活扩展
- **可靠性**: 网络波动情况下的缓存完整性保障

### 1.2 核心原则：谁先发送谁请求

**重要**：在本架构中，`CacheRequest` 和 `CacheResponse` 不是固定的“客户端请求/服务器响应”角色，而是遵循以下原则：

- **谁先发送谁就是请求方**：使用 `CacheRequest` 作为数据载体
- **接收方的回复**：使用 `CacheResponse` 作为回复载体
- **服务器主动推送**：服务器是发起方，因此使用 `CacheRequest` 而非 `CacheResponse`

这种设计避免了传统 C/S 架构中“客户端=请求、服务器=响应”的固化思维，使得双向通信更加灵活。

### 4.2 双命令设计思路：CacheSync vs CacheMetadataSync

**核心设计理念**：采用"轻量元数据协调 + 重量数据传输"的分层同步策略

#### 为什么需要两套命令？

```
┌─────────────────────────────────────────────────────────────┐
│                   问题场景                                    │
│                                                             │
│  用户登录时，服务器有 100 个可缓存的表                         │
│  如果直接同步所有数据：                                       │
│  ❌ 传输量大（可能几百MB）                                    │
│  ❌ 登录慢（用户体验差）                                      │
│  ❌ 网络拥堵（影响其他业务）                                   │
│  ❌ 内存爆炸（客户端压力大）                                   │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│                   解决方案                                    │
│                                                             │
│  阶段1：元数据同步（CacheMetadataSync）- 几KB                │
│  ├─ 服务器发送：每个表的统计信息                              │
│  │   { TableName, DataCount, VersionStamp, LastUpdateTime } │
│  └─ 客户端决策：比对本地版本，标记需更新的表                   │
│                                                             │
│  阶段2：按需数据同步（CacheSync）- 仅需要的表                 │
│  ├─ 客户端请求：只请求版本不一致的表                          │
│  └─ 服务器响应：发送完整的实体数据                            │
│                                                             │
│  结果：                                                       │
│  ✅ 登录快（元数据仅几KB）                                    │
│  ✅ 流量省（只同步变化的表）                                   │
│  ✅ 体验好（智能按需加载）                                     │
└─────────────────────────────────────────────────────────────┘
```

#### 两个命令的对比

| 特性 | CacheSync (0x0202) | CacheMetadataSync (0x0204) |
|------|-------------------|---------------------------|
| **本质** | 数据管道 | 协调协议 |
| **数据量** | 大（MB级） | 小（KB级） |
| **频率** | 高（实时） | 低（登录/定时） |
| **目的** | 传输内容 | 决策依据 |
| **必要性** | ✅ 必须 | ✅ 必须（否则无法智能同步） |
| **类比** | 书的正文内容 | 书的目录和出版信息 |

#### 使用场景示例

**场景1：用户登录**
```csharp
// 第1步：获取元数据（快速）
var metadata = await RequestCacheMetadataAsync();
// 返回：{ tb_Company: v100, tb_Product: v200, ... }

// 第2步：比对本地，发现 tb_Product 版本旧
if (localVersion < serverVersion)
{
    // 第3步：只请求需要更新的表（精准）
    await RequestCacheDataAsync("tb_Product");
}
```

**场景2：运行时数据变更**
```csharp
// 客户端A修改数据
await SyncCacheUpdateAsync("tb_Product", updatedProduct);

// 服务器广播给其他订阅者（使用 CacheSync）
await BroadcastCacheChangeAsync(pushRequest);
```

### 5.3 登录时缓存同步

**流程说明**:

1. **用户登录成功后**，服务器调用 `CacheMetadataSyncService.SyncCacheMetadataToClientAsync()`
2. **服务器收集**所有缓存元数据 (`CacheSyncMetadataManager.GetAllTableSyncInfo()`)
3. **创建同步请求** (`CacheMetadataSyncRequest`)，包含所有表的同步信息
4. **发送到客户端**，客户端接收后比对本地缓存状态
5. **客户端请求**需要更新的缓存数据
6. **客户端订阅**需要监控的表变更通知

#### 6.1.2 空表缓存优化

```csharp
public bool ValidateTableCacheIntegrity(string tableName)
{
    var syncInfo = _tableSyncInfo.GetOrAdd(tableName, _ => null);
    // 修改为：只要有LastUpdateTime就视为有效缓存
    return syncInfo != null && syncInfo.LastUpdateTime > DateTime.MinValue;
}
```

**效果**:
- 减少数据库查询次数
- 提高响应速度
- 减轻数据库压力

#### 6.1.3 缓存完整性验证

**新增功能**:
- `ValidateTableCacheIntegrity()` - 验证表缓存完整性
- `GetTablesWithIncompleteCache()` - 获取缓存不完整的表
- `RefreshIncompleteTables()` - 自动刷新不完整的缓存

#### 6.1.4 版本控制与冲突检测机制 (2026-04-18)

**问题**: 缺少版本控制，难以处理并发更新场景；客户端和服务器数据不一致时没有明确的解决机制

**解决方案**:
1. **在 `CacheSyncInfo` 中增加 `VersionStamp` 字段**:
   ```csharp
   public long VersionStamp { get; set; } = 1;
   ```

2. **在 `CacheData` 中增加 `VersionStamp` 字段**:
   ```csharp
   [JsonProperty]
   public long VersionStamp { get; set; } = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds();
   ```

3. **在 `CacheSyncMetadataManager.UpdateTableSyncInfo()` 中自动递增版本戳**:
   ```csharp
   if (_syncMetadata.TryGetValue(tableName, out var existingInfo))
   {
       newSyncInfo.VersionStamp = existingInfo.VersionStamp + 1;
   }
   else
   {
       newSyncInfo.VersionStamp = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds();
   }
   ```

4. **在 `CacheClientService.ProcessServerPushedRequest()` 中增加版本戳校验**:
   ```csharp
   if (localSyncInfo.VersionStamp > request.CacheData.VersionStamp)
   {
       _log.LogDebug("跳过过期的服务器推送: 表={0}, 本地版本={1}, 推送版本={2}", ...);
       return;
   }
   ```

5. **在 `CacheMetadataSyncService.SyncCacheMetadataToClientAsync()` 中为元数据打上最新版本号**:
   ```csharp
   kvp.Value.VersionStamp = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds();
   ```

6. **在 `CacheCommandHandler.BroadcastCacheChangeAsync()` 中同步版本戳到广播请求**:
   ```csharp
   var syncInfo = _cacheSyncMetadataManager.GetTableSyncInfo(request.TableName);
   if (syncInfo != null && pushRequest.CacheData != null)
   {
       pushRequest.CacheData.VersionStamp = syncInfo.VersionStamp;
   }
   ```

**效果**:
- 实现了基于时间戳的版本控制机制
- 避免了旧数据覆盖新数据的冲突问题
- 提高了多端数据一致性保障能力
- 符合文档第 7.4 节关于“版本控制”的要求

#### 6.1.5 架构原则修正：谁先发送谁请求 (2026-04-18)

**问题**: 之前错误地认为服务器推送应该使用 `CacheResponse`，导致类型不匹配和编译错误

**根本原因**: 误解了 `CacheRequest` 和 `CacheResponse` 的设计原则

**正确理解**:
- **核心原则**：谁先发送谁就是请求方，没有固定的“客户端=请求、服务器=响应”绑定关系
- **服务器主动推送**：服务器是发起方，应使用 `CacheRequest` 作为载体
- **客户端回复**：客户端收到服务器的推送后，可以用 `CacheResponse` 回复确认

**修正内容**:
1. **CacheRequest.cs**: 更新注释，明确服务器推送也使用 `CacheRequest`
2. **CacheResponse.cs**: 更新注释，明确 `CacheResponse` 仅用于对 `CacheRequest` 的回复
3. **CacheCommandHandler.cs**: 将 `BroadcastCacheChangeAsync` 中的 `CacheResponse` 改为 `CacheRequest`
4. **CacheClientService.cs**: 
   - 优先处理 `CacheRequest` 格式的服务器推送
   - 兼容处理 `CacheResponse` 格式的旧版推送
   - 新增 `ProcessServerPushedRequest` 方法处理标准推送

**效果**:
- 符合“谁先发送谁请求”的架构设计原则
- 解决了 `SendPacketCoreAsync<TRequest>` 要求 `IRequest` 接口的类型约束问题
- 消除了编译错误（CS0103、CS0121）
- 使双向通信逻辑更加清晰和一致

## 七、现存问题

### 7.1 通信机制不完善

**问题**: `CacheSubscriptionManager` 中订阅/取消订阅的通信部分代码被注释掉

**影响**: 实际同步功能受限，订阅关系可能无法正确传播

**建议**: 启用被注释的通信代码，完善订阅管理

### 7.2 缺少冲突解决策略

**问题**: 当客户端和服务器数据不一致时没有明确的解决机制

**影响**: 可能导致数据冲突，特别是在离线恢复后

**建议**: 
- 实现乐观并发控制
- 使用版本号或时间戳比较
- 建立明确的冲突检测和解决机制

### 7.3 网络故障恢复能力弱

**问题**: 缺少可靠的重连和消息重发机制

**影响**: 在网络不稳定情况下同步容易失败

**建议**:
- 实现消息持久化
- 设计指数退避算法
- 实现消息发送确认和接收确认机制

### 7.4 缺少版本控制 (✅ 已解决)

**问题**: 没有使用版本号或时间戳来确保数据一致性

**影响**: 难以处理并发更新场景

**解决方案**: ✅ 已在 v1.1 中实现基于时间戳的版本控制机制
- 为每次缓存更新添加时间戳（VersionStamp）
- 实现版本控制机制（CacheSyncInfo.VersionStamp、CacheData.VersionStamp）
- 建立基于规则的冲突检测策略（客户端推送时校验版本戳）

### 7.5 单向同步为主

**问题**: 主要支持服务器向客户端的同步，客户端变化同步到服务器的机制不完善

**影响**: 客户端数据变更无法及时同步到其他客户端

**建议**:
- 实现双向同步机制
- 客户端变化时自动同步到服务器
- 服务器验证后广播到所有订阅客户端

## 八、使用指南

### 8.1 基本缓存操作

```csharp
// 获取实体列表缓存
var entityList = _cacheManager.GetEntityList<MyEntity>();

// 更新实体列表缓存
_cacheManager.UpdateEntityList(entityList);

// 删除实体缓存
_cacheManager.DeleteEntity<MyEntity>(idValue);

// 删除整个表的缓存
_cacheManager.DeleteEntityList<MyEntity>();
```

### 8.2 缓存同步操作

```csharp
// 更新实体列表并同步元数据
_cacheManager.UpdateEntityListWithSync(entityList);

// 或者使用表名版本
_cacheManager.UpdateEntityListWithSync("MyEntity", entityList);

// 检查缓存是否需要同步
bool isExpired = _cacheSyncMetadata.IsTableExpired("MyEntity");

// 获取同步元数据
var syncInfo = _cacheSyncMetadata.GetTableSyncInfo("MyEntity");

// 验证缓存完整性
bool isCacheValid = _cacheSyncMetadata.ValidateTableCacheIntegrity("MyEntity");

// 如果缓存不完整，自动刷新
if (!isCacheValid)
{
    _cacheSyncMetadata.RefreshIncompleteTables(tableName => {
        LoadAndUpdateCache(tableName);
    });
}
```

### 8.3 缓存订阅操作

```csharp
// 订阅表的缓存变更
_subscriptionManager.Subscribe(sessionId, "MyEntity");

// 取消订阅
_subscriptionManager.Unsubscribe(sessionId, "MyEntity");

// 获取订阅了特定表的所有会话
var subscribers = _subscriptionManager.GetSubscribers("MyEntity");
```

### 8.4 依赖注入配置

```csharp
// 注册缓存服务
services.AddSingleton<IEntityCacheManager, EntityCacheManager>();
services.AddSingleton<ICacheSyncMetadata, CacheSyncMetadataManager>();
services.AddSingleton<CacheSubscriptionManager>();
services.AddSingleton<EventDrivenCacheManager>();

// 初始化表结构
var schemaManager = services.GetRequiredService<ITableSchemaManager>();
schemaManager.RegisterTableSchema<MyEntity>(
    e => e.Id,            // 主键表达式
    e => e.DisplayName,   // 显示字段表达式
    isView: false,        // 是否为视图
    isCacheable: true,    // 是否可缓存
    description: "我的实体" // 描述
);
```

### 8.5 使用 ITableSchemaManager (推荐)

```csharp
// 初始化表结构 - 直接使用 ITableSchemaManager
public class MyInitializer
{
    private readonly ITableSchemaManager _schemaManager;
    
    public MyInitializer(ITableSchemaManager schemaManager)
    {
        _schemaManager = schemaManager;
    }
    
    public void Initialize()
    {
        _schemaManager.RegisterTableSchema<tb_Company>(
            k => k.ID,
            v => v.CNName,
            isCacheable: true);
    }
}

// 管理缓存 - 使用 IEntityCacheManager
public class MyService
{
    private readonly IEntityCacheManager _cacheManager;
    
    public MyService(IEntityCacheManager cacheManager)
    {
        _cacheManager = cacheManager;
    }
    
    public void DoWork()
    {
        var list = _cacheManager.GetEntityList<tb_Company>();
        _cacheManager.UpdateEntityList(list);
    }
}
```
