## 三、死锁风险点详细分析

### 3.1 风险点1：事务内循环查询（最严重）

**技术原因**：
- 每个 `IsExistEntityAsync` 调用都会在数据库中执行 SELECT 语句
- 在事务内执行会获取共享锁(S锁)
- 循环执行多次查询延长了锁持有时间

**死锁场景**：

```
时间线：
┌─────────────────────────────────────────────────────────────────────┐
│ 事务A: 采购入库审核(循环查询)        事务B: 销售出库审核(循环查询)   │
├─────────────────────────────────────────────────────────────────────┤
│ BEGIN TRAN                          BEGIN TRAN                       │
│ SELECT tb_Inventory WHERE... (S锁)   SELECT tb_Inventory WHERE...     │
│ 【等待B释放锁】                     【等待A释放锁】                  │
│ ...循环10次...                       ...循环10次...                  │
│ UPDATE tb_Inventory (X锁)           UPDATE tb_Inventory (X锁)        │
│ 死锁！                                                       │
└─────────────────────────────────────────────────────────────────────┘
```

### 3.3 风险点3：表访问顺序不一致

**问题描述**：
- 不同业务模块访问 `tb_Inventory` 的顺序不同
- 没有统一的全局访问规范
- 导致循环等待死锁

**各模块访问顺序现状**：

| 模块 | 访问顺序 | 问题 |
|------|---------|------|
| 销售出库 | 库存→订单→流水 | 顺序不统一 |
| 采购入库 | 库存→订单→流水 | 与销售相同 |
| 调拨单 | 源库存→目标库存 | 特殊顺序 |
| 形态转换 | 源库存→目标库存 | 特殊顺序 |

## 四、UnitOfWorkManage 事务管理分析

### 4.1 当前实现机制

**关键代码** (`UnitOfWorkManage.cs` 行74-115)：

```csharp
public void BeginTran()
{
    var context = GetOrCreateTransactionContext();
    var dbClient = GetDbClient();

    // 锁住当前线程的连接
    lock (dbClient)  // ⚠️ 关键：每个连接有自己的锁
    {
        if (context.Depth == 1)
        {
            dbClient.Ado.BeginTran();  // 开启物理事务
        }
        else
        {
            // 嵌套事务使用保存点
            dbClient.Ado.ExecuteCommand($"SAVE TRANSACTION {savePointName}");
        }
    }
}
```

**建议2：添加死锁自动重试机制**

```csharp
public bool TryCommitWithRetry(int maxRetries = 3)
{
    for (int i = 0; i < maxRetries; i++)
    {
        try
        {
            CommitTran();
            return true;
        }
        catch (SqlException ex) when (ex.Number == 1205) // 死锁错误码
        {
            // 等待后重试
            Thread.Sleep(100 * (i + 1));
        }
    }
    return false;
}
```

### 5.2 P0-立即执行方案

#### 5.2.1 创建全局表访问顺序规范类

```csharp
// 文件位置: RUINORERP.Business/CommService/DeadlockPrevention.cs

/// <summary>
/// 全局表访问顺序规范（防止死锁）
/// 所有库存相关业务必须按此顺序访问表
/// </summary>
public static class TableAccessOrder
{
    /// <summary>
    /// 库存模块表访问优先级（数字越小越先访问）
    /// </summary>
    public static readonly Dictionary<string, int> InventoryModulePriority = new()
    {
        { "tb_Inventory", 1 },           // 核心资源，优先访问
        { "tb_InventoryTransaction", 2 }, // 流水记录
        { "tb_SaleOrder", 10 },          // 销售订单
        { "tb_SaleOut", 11 },            // 销售出库
        { "tb_PurOrder", 12 },           // 采购订单
        { "tb_PurEntry", 13 },           // 采购入库
    };

    /// <summary>
    /// 验证访问顺序是否正确
    /// </summary>
    public static bool ValidateAccessOrder(string tableName, HashSet<string> accessedTables)
    {
        if (!InventoryModulePriority.TryGetValue(tableName, out int priority))
            return true; // 不在管控范围内的表

        foreach (var accessed in accessedTables)
        {
            if (InventoryModulePriority.TryGetValue(accessed, out int accessedPriority))
            {
                if (priority < accessedPriority)
                {
                    // 违反了访问顺序
                    return false;
                }
            }
        }
        return true;
    }
}
```

### 5.3 P1-短期优化方案

#### 5.3.1 创建批量预加载扩展方法

```csharp
// 文件位置: RUINORERP.Business/CommService/InventoryQueryHelper.cs

/// <summary>
/// 库存查询辅助类（死锁优化）
/// </summary>
public static class InventoryQueryHelper
{
    /// <summary>
    /// 批量预加载库存（1次查询替代N次）
    /// </summary>
    public static async Task<Dictionary<(long ProdDetailID, long LocationID), tb_Inventory>> 
        BatchLoadInventoryAsync(
            this IUnitOfWorkManage unitOfWorkManage,
            IEnumerable<(long ProdDetailID, long LocationID)> keys)
    {
        var keyList = keys.ToList();
        if (keyList.Count == 0)
            return new Dictionary<(long, long), tb_Inventory>();

        var inventoryList = await unitOfWorkManage.GetDbClient()
            .Queryable<tb_Inventory>()
            .Where(i => keyList.Any(k => k.ProdDetailID == i.ProdDetailID && k.LocationID == i.LocationID))
            .ToListAsync();

        return inventoryList.ToDictionary(
            i => (i.ProdDetailID, i.LocationID)
        );
    }
}
```

### 5.4 P2-中期优化方案

#### 5.4.1 增强 UnitOfWorkManage 死锁处理

```csharp
// 建议在 UnitOfWorkManage 中添加

/// <summary>
/// 带重试的事务执行
/// </summary>
public void ExecuteWithRetry(Action action, int maxRetries = 3)
{
    for (int i = 0; i < maxRetries; i++)
    {
        try
        {
            action();
            return;
        }
        catch (SqlException ex) when (IsDeadlockException(ex))
        {
            if (i == maxRetries - 1) throw;
            
            // 指数退避等待
            int waitMs = (int)Math.Pow(2, i) * 100;
            Thread.Sleep(waitMs);
            
            // 确保事务状态正确
            if (CurrentTransactionContext?.Status == TransactionStatus.Active)
            {
                RollbackTran();
            }
        }
    }
}

private bool IsDeadlockException(SqlException ex)
{
    return ex.Number == 1205 || // SQL Server 死锁
           ex.Message.Contains("deadlock");
}
```

## 八、总结

### 8.1 核心发现

1. **事务内循环查询是最严重的死锁风险**：90%以上的死锁源于此
2. **缺乏统一的表访问顺序**：不同模块访问顺序不统一导致循环等待
3. **UnitOfWorkManage 缺乏高级特性**：无超时控制、无死锁重试机制
