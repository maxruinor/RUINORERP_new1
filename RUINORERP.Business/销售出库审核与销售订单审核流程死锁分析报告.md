## 二、为什么查询操作被包含在事务中？

### 2.1 技术原因：SqlSugar 连接管理与事务绑定

**关键代码**：`RUINORERP.Repository\UnitOfWorks\UnitOfWorkManage.cs`

```csharp
public SqlSugarScope GetDbClient()
{
    // 为每个异步上下文创建独立的数据库连接实例
    if (_asyncLocalClient.Value == null)
    {
        _asyncLocalClient.Value = new SqlSugarScope(newConfig);
    }
    return _asyncLocalClient.Value as SqlSugarScope;
}

public void BeginTran()
{
    var dbClient = GetDbClient();
    lock (dbClient)  // 锁住当前线程的连接
    {
        dbClient.Ado.BeginTran();  // 开启物理事务
    }
}
```

**关键点**：
- 事务与特定的数据库连接绑定
- 开启事务后，所有通过该连接执行的查询都在事务上下文中运行
- 这导致开发者在事务内执行查询时，不会考虑额外开销

## 三、死锁产生的技术原因

### 3.1 场景一：循环等待死锁（最常见）

```
时间线：
┌─────────────────────────────────────────────────────────────────────┐
│ 事务A: 销售订单审核                    事务B: 销售出库审核          │
├─────────────────────────────────────────────────────────────────────┤
│ BEGIN TRAN                          BEGIN TRAN                      │
│                                     (预查询库存)                     │
│ UPDATE tb_Inventory                 UPDATE tb_Inventory              
│ SET Sale_Qty = Sale_Qty + 10       SET Quantity = Quantity - 5    
│ WHERE ProdDetailID=1               WHERE ProdDetailID=1             
│ ───────────────────────            ─────────────────────────       
│ 【X锁: (1,100)】                   【X锁: (1,100)】                
│                                     UPDATE tb_SaleOrder              
│                                     SET DataStatus = 8               
│                                     ───────────────────────          
│ UPDATE tb_SaleOrder                 【X锁: SOrder_ID=200】          
│ WHERE SOrder_ID = 200                                            │
│ ───────────────────────            ...等待...                      
│ 【等待: SOrder_ID=200】            死锁！                           
└─────────────────────────────────────────────────────────────────────┘
```

**死锁原因**：
1. 事务A先锁定库存，再锁定订单
2. 事务B先锁定库存，再锁定订单
3. 访问顺序不一致导致循环等待

---

### 3.2 场景二：索引缺失导致锁升级

```
┌─────────────────────────────────────────────────────────────────────┐
│ 死锁原因分析：                                                      │
│                                                                      │
│ 1. 无适当索引支持 → 表扫描 → 全表锁定                               │
│ 2. 锁升级阈值达到 → 行锁 → 页锁 → 表锁                              │
│ 3. 不同事务访问顺序不同 → 循环等待                                  │
└─────────────────────────────────────────────────────────────────────┘
```

**当前已实施的索引优化**：
- SQL Server 2008: `SQLScripts\Database_Indexes_SQLServer2008.sql`
- SQL Server 2016: `SQLScripts\Database_Indexes_SQLServer2016.sql`

## 四、事务设计的改进建议

### 4.1 方案一：缩小事务范围（推荐）

**原则**：只将真正需要原子性的操作放在事务内

```
改进前：
┌─────────────────────────────────────────┐
│         整个审核流程都在事务内           │
│  ┌─────────────────────────────────┐   │
│  │ 预查询 → 计算 → 更新库存 → ...  │   │
│  └─────────────────────────────────┘   │
└─────────────────────────────────────────┘

改进后：
┌─────────────────────────────────────────┐
│     预处理(无事务)  │  核心更新(事务)   │
│  ┌──────────────┐  │  ┌─────────────┐  │
│  │ 预查询       │  │  │ 更新库存   │  │
│  │ 计算         │  │  │ 更新状态   │  │
│  │ 验证         │  │  │ 记录流水   │  │
│  └──────────────┘  │  └─────────────┘  │
└─────────────────────────────────────────┘
```

**具体修改清单**：

| 序号 | 文件 | 当前状态 | 建议修改 | 优先级 |
|------|------|---------|---------|--------|
| 1 | tb_SaleOrderControllerPartial.cs | 已优化 | 保持现状 | 低 |
| 2 | tb_SaleOutControllerPartial.cs | 部分优化 | 进一步缩小事务范围 | 高 |

### 4.3 方案三：乐观锁优化

对于库存更新，使用乐观锁替代悲观锁：

```csharp
/// <summary>
/// 乐观锁更新库存
/// </summary>
public async Task<bool> UpdateInventoryWithOptimisticLockAsync(
    tb_Inventory inv, 
    int originalVersion)
{
    int affected = await _dbClient.Updateable(inv)
        .Where(x => x.Inventory_ID == inv.Inventory_ID)
        .Where(x => x.Version == originalVersion)  // 版本检查
        .SetValue(x => x.Version, x => x.Version + 1)
        .ExecuteCommandAsync();
    
    if (affected == 0)
    {
        throw new ConcurrencyException("库存数据已被其他操作修改，请刷新后重试");
    }
    
    return true;
}
```

**实施要求**：
1. 为 tb_Inventory 表添加 Version 字段
2. 修改库存更新逻辑使用乐观锁

## 七、总结

### 7.1 当前问题根因

1. **事务范围过大**：将大量查询操作包含在事务内，延长了锁持有时间
2. **缺乏统一的表访问顺序**：不同事务以不同顺序访问表，导致循环等待
3. **缺少乐观锁机制**：过度依赖悲观锁，高并发下死锁频发

### 7.2 已实施的优化

| 优化项 | 状态 | 说明 |
|--------|------|------|
| 死锁重试机制 | ✅ 已完成 | BatchRecordTransactionsWithRetry |
| 索引优化 | ✅ 已完成 | SQL Server 2008/2016 脚本 |
| 销售订单事务分离 | ✅ 已完成 | 财务处理已分离到事务外 |
