# 帮助内容维护机制

## 概述

本文档定义了RUINOR ERP帮助内容的维护机制，确保帮助文档与系统功能保持同步，为用户提供准确、有效的帮助信息。

## 维护职责

### 内容负责人

| 角色 | 职责 |
|------|------|
| 业务顾问 | 负责业务模块帮助内容的准确性和完整性 |
| 开发工程师 | 负责技术相关内容和新功能的帮助文档 |
| 产品经理 | 负责整体帮助内容规划和优先级排序 |
| 文档管理员 | 负责帮助文档的格式规范和发布管理 |

### 维护周期

| 内容类型 | 更新频率 | 触发条件 |
|----------|----------|----------|
| 核心业务功能 | 随版本发布 | 功能变更、BUG修复 |
| 常用操作指南 | 季度 review | 用户反馈、流程优化 |
| 界面截图 | 半年度更新 | 界面较大改版 |
| 字段说明 | 随版本发布 | 字段变更 |

## 更新触发机制

### 1. 功能变更触发

当系统功能发生以下变更时，必须更新帮助文档：

- **新增功能**：新功能上线前完成帮助文档
- **功能修改**：功能逻辑变化时同步更新
- **界面调整**：界面较大变化时更新截图和说明
- **字段变更**：字段名称、属性变化时更新字段帮助

### 2. 用户反馈触发

用户反馈帮助内容问题时，按以下流程处理：

1. 记录问题：标题、问题描述、涉及文档
2. 分类处理：
   - 错误类：2个工作日内修复
   - 不清晰：5个工作日内优化
   - 缺失类：纳入下个版本计划
3. 验证修复：测试确认问题已解决

### 3. 定期审查

每季度进行帮助内容审查：

- 检查功能描述是否与实际一致
- 检查截图是否过时
- 检查是否有遗漏的重要功能
- 收集用户使用反馈

## 内容管理流程

### 新增帮助内容流程

```
需求识别 → 编写初稿 → 格式审查 → 测试验证 → 发布上线 → 归档维护
```

1. **需求识别**：识别需要创建帮助文档的功能点
2. **编写初稿**：按模板编写帮助文档内容
3. **格式审查**：检查格式规范性（参考下面的格式规范）
4. **测试验证**：在实际系统中验证帮助内容准确性
5. **发布上线**：将文档部署到帮助系统
6. **归档维护**：记录文档版本，更新索引

### 更新帮助内容流程

```
发现问题 → 评估影响 → 修改内容 → 验证测试 → 发布更新 → 记录变更
```

1. **发现问题**：通过审查或反馈发现需要更新的内容
2. **评估影响**：评估问题的影响范围和紧急程度
3. **修改内容**：按模板修改相关帮助文档
4. **验证测试**：确认修改后的内容正确无误
5. **发布更新**：更新帮助系统中的内容
6. **记录变更**：记录更新的时间、内容和原因

## 格式规范

### 文档命名规范

```
[类型]-[模块]-[功能].md

类型：Forms（窗体）、Fields（字段）、Modules（模块）
模块：销售管理、采购管理、库存管理等
功能：具体功能名称
```

示例：
- `Forms-UCSaleOrder.md` - 销售订单窗体
- `Fields-tb_SaleOrder-OrderNo.md` - 销售订单编号字段
- `Modules-SalesManagement.md` - 销售管理模块

### 文档结构规范

每个帮助文档应包含以下章节：

1. **标题**：功能/字段的名称
2. **说明**：简要描述用途
3. **业务意义**：详细说明业务含义
4. **使用场景**：典型使用场景
5. **相关字段**：关联的字段或控件
6. **常见问题**：FAQ

### 内容质量标准

- **准确性**：内容描述与系统功能一致
- **完整性**：覆盖主要使用场景和功能点
- **可读性**：语言清晰，结构合理
- **时效性**：与当前系统版本同步

## 版本管理

### 版本标识

帮助文档使用以下版本标识：

- **版本号**：格式为 `v主版本.次版本`
- **更新日期**：格式为 `YYYY-MM-DD`
- **更新说明**：简要说明本次更新内容

### 版本记录

每个文档头部应包含版本信息：

```markdown
---
title: 销售订单窗体
version: v1.2
last_update: 2024-03-15
update_log:
  - v1.2: 添加平台订单功能说明
  - v1.1: 补充定制订单场景
  - v1.0: 初始版本
---
```

## 发布管理

### 发布流程

1. **开发环境测试**：在开发环境验证帮助内容
2. **预发布验证**：在预发布环境进行最终验证
3. **正式发布**：发布到生产环境
4. **用户通知**：通过系统公告通知用户

### 回滚机制

当发布出现问题时：

1. 立即停止问题版本的访问
2. 恢复前一版本的内容
3. 修复问题后重新发布

## 质量保障

### 检查清单

发布前确认以下项目：

- [ ] 文档格式符合规范
- [ ] 内容与系统功能一致
- [ ] 截图清晰且为最新版本
- [ ] 链接跳转正确
- [ ] 无错别字和语法错误
- [ ] 长度适中（核心内容300-500字）

### 质量指标

| 指标 | 目标值 |
|------|--------|
| 内容准确率 | >= 98% |
| 用户满意度 | >= 90% |
| 问题响应时间 | <= 2工作日 |
| 文档覆盖率 | >= 80%核心功能 |
