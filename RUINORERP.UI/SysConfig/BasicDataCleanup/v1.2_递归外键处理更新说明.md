# v1.2 更新说明 - 递归外键冲突处理

## 📅 更新日期
2026-04-24

## 🎯 更新目标

解决多层级外键约束冲突问题，特别是当删除操作涉及3层或更多层的关联关系时。

---

## 🔍 问题分析

### 错误日志示例

```
DELETE 语句与 REFERENCE 约束"FK_PURENTRY_PURENTRYDETAIL"冲突。
该冲突发生于数据库"erpnew"，表"dbo.tb_PurEntryDetail", column 'PurEntryID'。

DELETE 语句与 REFERENCE 约束"FK_PURORDER_REF_FM_PAYEEINFO"冲突。
该冲突发生于数据库"erpnew"，表"dbo.tb_PurOrder", column 'PayeeInfoID'。

DELETE 语句与 REFERENCE 约束"FK_PURORDTAIL_REF_PURORDER"冲突。
该冲突发生于数据库"erpnew"，表"dbo.tb_PurOrderDetail", column 'PurOrder_ID'。
```

### 根本原因

**多层级外键依赖链**：
```
tb_CustomerVendor (客户供应商)
  ↓ OneToMany: tb_PurOrders
tb_PurOrder (采购订单)
  ↓ OneToMany: tb_PurEntries  
tb_PurEntry (采购入库单)
  ↓ OneToMany: tb_PurEntryDetails
tb_PurEntryDetail (采购入库单明细) ← 外键冲突!
```

**问题流程**：
1. `DeleteDeepLevelRelationsByNavigationAsync` 尝试删除 `tb_PurOrder` 的关联数据
2. 成功删除 `tb_PurEntry`（采购入库单）
3. 但 `tb_PurEntryDetail`（明细）仍然引用 `tb_PurEntry`
4. 删除 `tb_PurEntry` 时触发外键约束冲突
5. 异常被 BaseController 捕获并记录警告，但事务已处于不一致状态
6. 后续调用 `BaseDeleteByNavAsync` 时再次失败
7. 最终事务回滚，无法提交

## ✅ 修复方案

### 1. 增强 DeleteDeepLevelRelationsByNavigationAsync 异常处理

**修复前**：
```csharp
// 正式模式:执行真正的深层级删除
await controller.DeleteDeepLevelRelationsByNavigationAsync(entity);
Log($"已清理 {typeof(T).Name} ID={id} 的所有深层级关联数据");
```

**修复后**：
```csharp
// 正式模式:执行真正的深层级删除
try
{
    await controller.DeleteDeepLevelRelationsByNavigationAsync(entity);
    Log($"已清理 {typeof(T).Name} ID={id} 的所有深层级关联数据");
}
catch (SqlException navEx)
{
    // DeleteDeepLevelRelationsByNavigationAsync 也可能遇到外键冲突
    _logger?.LogWarning(navEx, $"DeleteDeepLevelRelationsByNavigationAsync 遇到外键冲突: {navEx.Message}");
    
    // 尝试解析并处理外键冲突
    var fkInfo = ParseForeignKeyError(navEx.Message);
    if (fkInfo != null)
    {
        Log($"[深层级删除] 检测到外键约束冲突: 表={fkInfo.Value.TableName}, 字段={fkInfo.Value.ColumnName}");
        var deleted = await DeleteReferencingTableAsync<T>(id, fkInfo.Value.TableName, fkInfo.Value.ColumnName);
        
        if (deleted)
        {
            // 重试深层级删除
            await controller.DeleteDeepLevelRelationsByNavigationAsync(entity);
            Log($"✓ 解决外键冲突后成功清理深层级关联数据");
        }
    }
}
```

### 2. 递归外键冲突处理

**核心改进**：`DeleteReferencingTableAsync` 现在支持递归处理多层级外键

```csharp
private async Task<bool> DeleteReferencingTableAsync<T>(
    long mainId, 
    string referencingTableName, 
    string referencingColumnName, 
    int depth = 0)  // ← 新增深度参数
{
    if (depth > 3) // 限制递归深度为3层
    {
        _logger?.LogWarning($"递归深度超过限制,停止处理: {referencingTableName}");
        return false;
    }

    try
    {
        // 尝试删除关联表
        var deletedCount = await db.Deleteable<object>()
            .AS(referencingTableName)
            .Where($"{referencingColumnName} = @id", new { id = mainId })
            .ExecuteCommandAsync();
        
        return true;
    }
    catch (SqlException ex)
    {
        // 删除时也遇到外键冲突,递归处理
        var subFkInfo = ParseForeignKeyError(ex.Message);
        if (subFkInfo != null)
        {
            Log($"[递归] 检测到 {referencingTableName} 的外键冲突: 表={subFkInfo.Value.TableName}");
            
            // 递归删除更深层级的关联数据
            var subDeleted = await DeleteReferencingTableAsync<T>(
                mainId, 
                subFkInfo.Value.TableName, 
                subFkInfo.Value.ColumnName, 
                depth + 1  // ← 深度+1
            );
            
            if (subDeleted)
            {
                // 重试删除当前表
                var retryDeleted = await db.Deleteable<object>()
                    .AS(referencingTableName)
                    .Where($"{referencingColumnName} = @id", new { id = mainId })
                    .ExecuteCommandAsync();
                
                return retryDeleted >= 0;
            }
        }
        
        return false;
    }
}
```

## ⚠️ 注意事项

### 1. 递归深度限制

当前限制为 **3层**，原因：
- 避免无限递归导致栈溢出
- 大多数业务场景3层足够
- 超过3层说明数据库设计可能存在问题

如需调整，修改 `DeleteReferencingTableAsync` 方法中的：
```csharp
if (depth > 3) // 改为需要的深度
```

### 2. 性能考虑

递归处理会增加数据库调用次数：
- 每层外键冲突至少2次SQL调用（删除子表 + 重试删除父表）
- 3层递归最多可能需要 2³ = 8 次调用

**优化建议**：
- 大数据量时分批处理
- 监控日志中的递归深度
- 考虑在数据库中设置级联删除（ON DELETE CASCADE）

### 3. 事务一致性

- 所有递归操作都在同一个事务中
- 任何一层失败都会导致整个事务回滚
- 确保数据库连接和事务上下文正确传递
