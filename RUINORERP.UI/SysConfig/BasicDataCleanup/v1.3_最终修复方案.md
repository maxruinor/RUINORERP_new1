# v1.3 最终修复方案 - 简化外键处理策略

## 📅 更新日期
2026-04-24

## 🎯 核心问题

删除供应商时遇到多层级外键冲突：
```
tb_CustomerVendor (供应商)
  → tb_PurOrder (采购订单)
    → tb_PurEntry (采购入库单)
      → tb_PurEntryDetail (入库明细) ← 外键冲突!
```

## ❌ 之前的问题方案

### v1.2 的递归外键处理存在的问题

1. **参数传递错误**：用供应商ID去删除入库明细表（应该用入库单ID）
2. **逻辑过于复杂**：递归处理容易导致死循环和事务超时
3. **治标不治本**：没有解决根本问题（Controller 的 Include 不完整）

### 错误示例

```csharp
// ❌ 错误: 用供应商ID删除入库明细
await DeleteReferencingTableAsync<T>(
    supplierId,           // ← 这是供应商ID!
    "tb_PurEntryDetail",  // ← 但这是入库明细表
    "PurEntryID"          // ← 字段需要的是入库单ID
);
```

## ✅ v1.3 正确方案

### 核心策略

**依赖现有的 BaseController 基础设施，而不是手动处理外键**

1. **第一步**：调用 `DeleteDeepLevelRelationsByNavigationAsync` 清理所有深层级数据
2. **第二步**：调用 `BaseDeleteByNavAsync` 删除直接关联的数据和主表
3. **如果失败**：提供详细的错误诊断，提示用户修复 Controller

### 代码实现

```csharp
// 正式模式:执行真正的深层级删除
try
{
    await controller.DeleteDeepLevelRelationsByNavigationAsync(entity);
    Log($"已清理 {typeof(T).Name} ID={id} 的所有深层级关联数据");
}
catch (Exception deepEx)
{
    // DeleteDeepLevelRelationsByNavigationAsync 遇到异常
    _logger?.LogError(deepEx, $"DeleteDeepLevelRelationsByNavigationAsync 失败: {deepEx.Message}");
    Log($"✗ 深层级删除失败: {deepEx.Message}");
    
    // 记录失败但继续尝试主表删除(可能部分数据已被删除)
    result.FailedIds ??= new List<long>();
    result.FailedIds.Add(id);
    continue;
}

// 执行主表删除
try
{
    await controller.BaseDeleteByNavAsync(entity);
    totalDeleted++;
    Log($"✓ 成功删除 ID={id} 及其关联数据");
}
catch (SqlException ex)
{
    // 提供详细的错误诊断信息
    string diagnosticMsg = $"\n[外键约束冲突诊断]\n" +

        $"  • 目标 ID: {id}\n" +
        $"  • 错误信息: {ex.Message}\n\n" +
        $"[可能原因]\n" +
        $"  1. Controller.BaseDeleteByNavAsync 中遗漏了某些 Include\n" +
        $"  2. DeleteDeepLevelRelationsByNavigationAsync 未能完全清理深层级数据\n" +
        $"  3. 数据库中存在未被导航属性覆盖的外键关系\n\n" +
        $"[解决方案]\n" +
        $"  1. 检查 {typeof(T).Name}Controller.BaseDeleteByNavAsync 方法\n" +
        $"  2. 确保所有 OneToMany 导航属性都已 Include\n" +
        $"  3. 参考以下错误信息中的表和字段名定位缺失的关联\n";
    
    var fkInfo = ParseForeignKeyError(ex.Message);
    if (fkInfo != null)
    {
        diagnosticMsg += $"\n[冲突详情]\n" +
            $"  • 冲突表: {fkInfo.Value.TableName}\n" +
            $"  • 冲突字段: {fkInfo.Value.ColumnName}\n";
    }
    
    Log(diagnosticMsg);
    
    // 记录失败并继续处理下一条
    result.FailedIds ??= new List<long>();
    result.FailedIds.Add(id);
    continue;
}
```

## 🔧 根本解决方案

### 修复 tb_CustomerVendorController.BaseDeleteByNavAsync

**问题**：当前的 Include 只包含一层关系，没有递归删除子表的子表

```csharp
// ❌ 当前实现 - 只删除一层
bool rs = await _db.DeleteNav<tb_CustomerVendor>(m => m.CustomerVendor_ID == entity.CustomerVendor_ID)
    .Include(m => m.tb_PurOrders)  // ← 只删除采购订单
    // ... 其他 Include
    .ExecuteCommandAsync();
```

**正确做法**：在 Include 之前先调用 `DeleteDeepLevelRelationsByNavigationAsync`

```csharp
public async override Task<bool> BaseDeleteByNavAsync(T model)
{
    tb_CustomerVendor entity = model as tb_CustomerVendor;
    
    // ✅ 先删除所有深层级关联数据
    await DeleteDeepLevelRelationsByNavigationAsync(entity);
    
    // ✅ 再删除直接关联的数据和主表
    bool rs = await _unitOfWorkManage.GetDbClient().DeleteNav<tb_CustomerVendor>(m => m.CustomerVendor_ID == entity.CustomerVendor_ID)
        .Include(m => m.tb_FM_Invoices)
        .Include(m => m.tb_PurOrders)
        .Include(m => m.tb_SaleOrders)
        // ... 其他 Include
        .ExecuteCommandAsync();
        
    return rs;
}
```

### 或者修改 DeleteDeepLevelRelationsByNavigationAsync

确保它能够正确处理所有层级的关系。检查 `BaseControllerGeneric.cs` 中的实现，确保它：

1. 递归扫描所有 OneToMany 导航属性
2. 对每个子表也调用其 `DeleteDeepLevelRelationsByNavigationAsync`
3. 最后才删除子表本身

## ⚠️ 注意事项

### 1. 不要手动处理外键冲突

❌ **错误做法**：
```csharp
// 不要用供应商ID去删除入库明细
await DeleteReferencingTableAsync(supplierId, "tb_PurEntryDetail", "PurEntryID");
```

✅ **正确做法**：
```csharp
// 依赖 DeleteDeepLevelRelationsByNavigationAsync 处理
await controller.DeleteDeepLevelRelationsByNavigationAsync(entity);
```

### 2. 事务超时问题

如果出现 60 秒超时，说明：
- 数据量过大
- 存在死循环
- 数据库锁等待

**解决方案**：
- 分批删除（每次 10-50 条）
- 检查是否有无限递归
- 优化数据库索引

### 3. Controller 自动生成

如果 Controller 是自动生成的，修改后可能会被覆盖。建议：

- 方案A：修改生成模板
- 方案B：创建部分类（partial class）扩展方法
- 方案C：在 DataCleanupEngine 中统一处理
