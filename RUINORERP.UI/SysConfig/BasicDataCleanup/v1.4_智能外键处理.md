### 智能处理方法

#### 方法1: SmartHandleForeignKeyConflictAsync

```csharp
private async Task<bool> SmartHandleForeignKeyConflictAsync<T>(
    long mainId, 
    string referencingTableName, 
    string referencingColumnName)
{
    var db = _unitOfWork.GetDbClient();
    
    try
    {
        // 策略1: 直接删除引用表中所有关联的记录
        var deletedCount = await db.Deleteable<object>()
            .AS(referencingTableName)
            .Where($"EXISTS (SELECT 1 FROM [{typeof(T).Name}] WHERE ...)", 
                new { id = mainId })
            .ExecuteCommandAsync();
        
        if (deletedCount > 0)
        {
            Log($"[智能处理] ✓ 成功删除 {referencingTableName} {deletedCount} 条记录");
            return true;
        }
        
        // 策略2: 通过导航属性查找
        return await DeleteViaNavigationPropertyAsync<T>(
            mainId, 
            referencingTableName, 
            referencingColumnName
        );
    }
    catch (Exception ex)
    {
        _logger?.LogError(ex, $"[智能处理] 处理外键冲突失败");
        return false;
    }
}
```

#### 方法2: DeleteViaNavigationPropertyAsync

```csharp
private async Task<bool> DeleteViaNavigationPropertyAsync<T>(
    long mainId, 
    string referencingTableName, 
    string referencingColumnName)
{
    var entityType = typeof(T);
    
    // 查找指向目标表的导航属性
    var navProperties = entityType.GetProperties()
        .Where(p => 
        {
            var navigateAttr = p.GetCustomAttribute<SqlSugar.Navigate>();
            return navigateAttr != null && 
                   navigateAttr.GetNavigateType() == SqlSugar.NavigateType.OneToMany;
        })
        .ToList();
    
    foreach (var navProp in navProperties)
    {
        var childType = navProp.PropertyType.IsGenericType
            ? navProp.PropertyType.GetGenericArguments()[0]
            : navProp.PropertyType;
        var childTableName = childType.Name;
        
        // 如果匹配,直接删除
        if (string.Equals(childTableName, referencingTableName, StringComparison.OrdinalIgnoreCase))
        {
            var navigateAttr = navProp.GetCustomAttribute<SqlSugar.Navigate>();
            var fkFieldName = navigateAttr.GetName();
            
            var deletedCount = await db.Deleteable<object>()
                .AS(childTableName)
                .Where($"{fkFieldName} = @id", new { id = mainId })
                .ExecuteCommandAsync();
            
            Log($"[导航属性] ✓ 删除 {childTableName} {deletedCount} 条记录");
            return true;
        }
    }
    
    // 如果没找到导航属性,尝试直接删除
    var directDeleted = await db.Deleteable<object>()
        .AS(referencingTableName)
        .Where($"{referencingColumnName} IN (SELECT PrimaryKeyID FROM ...)", 
            new { id = mainId })
        .ExecuteCommandAsync();
    
    return directDeleted > 0;
}
```

## ⚠️ 注意事项

### 1. 递归深度限制

当前实现可能会触发多次外键冲突处理，形成递归调用。建议：

- 监控日志中的 `[智能处理]` 标记
- 如果出现超过 5 层的递归，考虑优化数据库设计
- 添加递归深度计数器防止无限循环

### 2. 性能影响

每次外键冲突都需要额外的 SQL 查询和删除操作：

- 小数据量 (< 100 条): 影响可忽略
- 中等数据量 (100-1000 条): 可能增加 1-2 秒
- 大数据量 (> 1000 条): 建议分批处理

### 3. 事务一致性

所有操作都在同一个事务中：

- ✅ 任何一步失败都会回滚整个事务
- ✅ 保证数据一致性
- ⚠️ 长时间运行可能导致事务超时

**解决方案**：
- 增加事务超时时间（当前 60 秒）
- 或者分批删除（每次 10-50 条）

### 4. 边界情况

以下情况可能无法自动处理：

- 循环外键引用（A → B → A）
- 软删除标记（IsDeleted 字段）
- 触发器阻止删除
- 权限限制

## 📝 总结

**v1.4 的核心思想**：

1. ✅ **能解析就自动处理**：不再仅仅报错，而是尝试自动解决
2. ✅ **多策略智能处理**：直接删除 → 导航属性 → SQL 子查询
3. ✅ **保持事务一致性**：所有操作在同一事务中，要么全成功，要么全回滚
4. ✅ **详细日志记录**：便于追踪和调试

**适用场景**：
- ✅ 管理员手动清理数据
- ✅ 测试环境数据重置
- ✅ 偶尔的数据维护操作

**不适用场景**：
- ❌ 高频批量删除（性能考虑）
- ❌ 有复杂业务逻辑的删除（需自定义处理）
- ❌ 有审计要求的删除（需保留历史记录）
