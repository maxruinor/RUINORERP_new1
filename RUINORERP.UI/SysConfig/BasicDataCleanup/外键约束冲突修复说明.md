# 外键约束冲突修复说明

## 📋 问题描述

在删除 `tb_ProdCategories`（产品分类）时出现外键约束冲突错误：

```
DELETE 语句与 REFERENCE 约束"FK_TB_PROD_REFERENCE_TB_PRODC"冲突。
该冲突发生于数据库"erpnew"，表"dbo.tb_Prod", column 'Category_ID'。
```

**根本原因**：
1. **删除顺序错误**：先删除主表（分类），但子表（产品）仍有引用
2. **测试模式跳过深层级删除**：导致无法准确统计影响范围
3. **多层级关系未处理**：如 `tb_ManufacturingOrderDetail` → `tb_ProdDetail` → `tb_ProdBase`

---

## ✅ 修复方案

### 1. 修改删除逻辑（DataCleanupEngine.cs）

#### 修复前的问题代码：
```csharp
// ❌ 测试模式直接跳过，不执行任何删除操作
if (isTestMode)
{
    Log($"[测试模式] 模拟删除 ID={id}");
    totalDeleted++;
    continue; // 跳过后面的所有逻辑
}

// 正式模式才执行深层级删除
await controller.DeleteDeepLevelRelationsByNavigationAsync(entity);
await controller.BaseDeleteByNavAsync(entity);
```

#### 修复后的正确逻辑：
```csharp
// ✅ 无论测试还是正式模式，都必须先清理深层级关联数据
if (!isTestMode)
{
    // 正式模式:执行真正的深层级删除
    await controller.DeleteDeepLevelRelationsByNavigationAsync(entity);
    Log($"已清理 {typeof(T).Name} ID={id} 的所有深层级关联数据");
}
else
{
    // 测试模式:也需要模拟深层级删除以准确统计
    var navDeletedCount = await SimulateDeepLevelDeleteAsync<T>(entity);
    totalDeleted += navDeletedCount;
    Log($"[测试模式] 模拟清理深层级关联数据,预计影响 {navDeletedCount} 条记录");
}

// 执行主表删除
if (isTestMode)
{
    // 测试模式:仅统计,不实际删除
    totalDeleted++;
    Log($"[测试模式] 模拟删除主记录 ID={id}");
}
else
{
    // 正式模式:执行真正的主表删除
    await controller.BaseDeleteByNavAsync(entity);
    totalDeleted++;
    Log($"✓ 成功删除 ID={id} 及其关联数据");
}
```

### 2. 新增模拟深层级删除方法

添加了两个新方法用于测试模式下的准确统计：

#### `SimulateDeepLevelDeleteAsync<T>()`
- **功能**：通过反射扫描实体的所有 OneToMany 导航属性
- **作用**：统计每个子表的记录数，递归检查最多2层深度
- **返回**：预计影响的总记录数

#### `SimulateSubTableDeletesAsync()`
- **功能**：递归模拟子表的深层级删除
- **参数**：父表类型、外键字段、父ID、当前深度
- **限制**：最大递归深度为2层，避免性能问题

### 3. 修复外键冲突重试逻辑

在外键冲突处理中也区分测试/正式模式：

```csharp
catch (SqlException ex)
{
    var fkInfo = ParseForeignKeyError(ex.Message);
    if (fkInfo != null)
    {
        var deleted = await DeleteReferencingTableAsync<T>(id, fkInfo.Value.TableName, fkInfo.Value.ColumnName);
        if (deleted)
        {
            try
            {
                if (isTestMode)
                {
                    totalDeleted++;
                    Log($"[测试模式] 解决外键冲突后模拟删除 ID={id}");
                }
                else
                {
                    await controller.BaseDeleteByNavAsync(entity);
                    totalDeleted++;
                    Log($"✓ 解决外键冲突后成功删除 ID={id}");
                }
            }
            catch (Exception retryEx)
            {
                // 记录失败并继续
            }
        }
    }
}
```

## 🎯 修复效果

### 测试模式（预览/测试执行）
- ✅ 准确统计所有深层级关联数据的影响范围
- ✅ 不会实际删除任何数据（事务回滚）
- ✅ 提供详细的预计删除记录数

### 正式模式（正式删除）
- ✅ 先删除所有深层级关联数据（通过 `DeleteDeepLevelRelationsByNavigationAsync`）
- ✅ 再删除主表记录（通过 `BaseDeleteByNavAsync`）
- ✅ 如果仍有外键冲突，自动解析并删除冲突的关联数据
- ✅ 事务保证原子性，要么全部成功，要么全部回滚

## 📊 删除流程对比

### 修复前的错误流程：
```
用户选择分类A (ID=100)
  ↓
测试模式: 直接跳过 ❌
正式模式: 
  ├─ 调用 BaseDeleteByNavAsync
  │   └─ 尝试删除 tb_ProdCategories WHERE Category_ID=100
  │       └─ ❌ 外键冲突! tb_Prod 表仍有引用
  └─ 删除失败
```

### 修复后的正确流程：
```
用户选择分类A (ID=100)
  ↓
测试模式:
  ├─ SimulateDeepLevelDeleteAsync
  │   ├─ 扫描 tb_ProdCategories 的导航属性
  │   ├─ 统计 tb_Prods (Category_ID=100): 50条
  │   ├─ 递归统计 tb_ProdDetail (ProdBaseID in ...): 200条
  │   └─ 递归统计 tb_ManufacturingOrderDetail (ProdDetailID in ...): 500条
  ├─ 预计总删除: 751条 (50+200+500+1主记录)
  └─ 事务回滚，数据未删除 ✅

正式模式:
  ├─ DeleteDeepLevelRelationsByNavigationAsync
  │   ├─ 删除 tb_ManufacturingOrderDetail (ProdDetailID in ...)
  │   ├─ 删除 tb_ProdDetail (ProdBaseID in ...)
  │   └─ 删除 tb_Prods (Category_ID=100)
  ├─ BaseDeleteByNavAsync
  │   └─ 删除 tb_ProdCategories WHERE Category_ID=100 ✅
  └─ 事务提交，数据永久删除
```

## 🔍 关键改进点

| 改进项 | 修复前 | 修复后 |
|--------|--------|--------|
| **测试模式准确性** | 跳过所有逻辑，统计不准确 | 模拟深层级删除，准确统计 |
| **删除顺序** | 直接删除主表，可能外键冲突 | 先删子表，再删主表 |
| **外键冲突处理** | 解析后重试但未区分模式 | 区分测试/正式模式正确处理 |
| **多层级关系** | 仅处理1层 | 递归处理最多2层 |
| **事务管理** | TransactionScope + IUnitOfWorkManage | 仅使用 IUnitOfWorkManage,简化管理员工具处理 |

## 📝 注意事项

1. **数据安全**：正式删除前务必备份数据库
2. **性能考虑**：深层级删除可能耗时较长，建议分批处理
3. **递归深度**：当前限制为2层，如需更深可调整 `SimulateSubTableDeletesAsync` 中的深度限制
4. **日志记录**：所有操作都有详细日志，便于排查问题
