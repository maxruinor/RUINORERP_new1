# 程序卡死问题诊断与修复

## 📅 日期
2026-04-25

## ❌ 问题现象

从日志看，程序在依赖收集阶段突然中断：
```
[16:37:19.815] [依赖收集] → 正在分析表: tb_FM...
```

之后没有任何输出，程序似乎卡死或崩溃。

## 🔍 问题分析

### 可能的原因

#### 1. StackOverflowException（栈溢出）- **最可能**

虽然添加了 `visited` 集合防止循环引用，但可能存在以下情况：

**场景A：递归深度过大**
```
tb_BOM_S (1条记录)
  └─ tb_ProdDetail (13条记录)
     └─ 55个子表需要分析
        └─ 每个子表可能有自己的子表
           └─ ... (递归深度可能超过100层)
```

即使没有循环引用，如果外键关系树非常深，也可能导致栈溢出。

**场景B：async/await的上下文切换问题**
```csharp
// 异步递归调用
await CollectAllForeignKeysRecursiveAsync(...);
```

在某些情况下，async/await可能导致上下文累积，增加栈深度。

#### 2. 数据库连接超时或断开

COUNT(*) 查询可能在某个表上卡住：
- 表上有锁
- 索引损坏
- 网络问题

#### 3. UI线程阻塞

AppendRealTimeLog 可能在大量日志输出时阻塞UI线程。

## ✅ 已实施的修复

### 修复1：添加异常捕获

在 `CollectAllForeignKeysRecursiveAsync` 的递归调用处添加try-catch：

```csharp
foreach (var fk in foreignKeys)
{
    try
    {
        cancellationToken.ThrowIfCancellationRequested();
        await CollectAllForeignKeysRecursiveAsync(fk.ReferencingTableName, dependencies, visited, cancellationToken);
    }
    catch (Exception ex)
    {
        Log($"[依赖收集] ❌ 递归分析 {fk.ReferencingTableName} 时出错: {ex.Message}");
        throw; // 重新抛出，让上层处理
    }
}
```

**作用**：
- 捕获具体哪个表导致异常
- 记录详细的错误信息
- 便于定位问题

### 修复2：外层已有异常处理

UI层的 `KbtnDeleteSelected_Click` 已有catch块：
```csharp
catch (Exception ex)
{
    AppendRealTimeLog($"[异常] 删除操作异常: {ex.Message}");
```

## 🔧 进一步修复建议

### 方案1：将递归改为迭代（推荐）

如果确实是栈溢出问题，最好的解决方案是将递归改为迭代：

```csharp
private async Task CollectAllForeignKeysIterativeAsync(
    string rootTableName,
    Dictionary<string, List<DbForeignKeyInfo>> dependencies,
    CancellationToken cancellationToken = default)
{
    var visited = new HashSet<string>();
    var queue = new Queue<string>();
    
    queue.Enqueue(rootTableName);
    
    while (queue.Count > 0)
    {
        cancellationToken.ThrowIfCancellationRequested();
        
        var tableName = queue.Dequeue();
        
        if (visited.Contains(tableName))
            continue;
        
        visited.Add(tableName);
        Log($"[依赖收集] → 正在分析表: {tableName}");
        
        var db = _unitOfWork.GetDbClient();
        
        // 检查记录数
        var recordCount = await db.Queryable<object>()
            .AS(tableName)
            .CountAsync();
        
        if (recordCount == 0)
        {
            Log($"[依赖收集]   ℹ️ 表 {tableName} 无数据，跳过");
            dependencies[tableName] = new List<DbForeignKeyInfo>();
            continue;
        }
        
        // 查询外键
        var foreignKeys = await db.Ado.SqlQueryAsync<DbForeignKeyInfo>(@"...", new { tableName });
        
        dependencies[tableName] = foreignKeys.ToList();
        
        // 将子表加入队列
        foreach (var fk in foreignKeys)
        {
            if (!visited.Contains(fk.ReferencingTableName))
            {
                queue.Enqueue(fk.ReferencingTableName);
            }
        }
    }
}
```

**优点**：
- 不会栈溢出
- 可以控制遍历顺序（BFS）
- 更容易监控进度

**缺点**：
- 代码复杂度增加
- 需要重构现有逻辑

### 方案2：限制递归深度

```csharp
private const int MAX_RECURSION_DEPTH = 50;

private async Task CollectAllForeignKeysRecursiveAsync(
    string tableName,
    Dictionary<string, List<DbForeignKeyInfo>> dependencies,
    HashSet<string> visited,
    int depth = 0,
    CancellationToken cancellationToken = default)
{
    if (depth > MAX_RECURSION_DEPTH)
    {
        Log($"[依赖收集] ⚠️ 递归深度超过 {MAX_RECURSION_DEPTH}，停止深入");
        return;
    }
    
    // ... 原有逻辑 ...
    
    foreach (var fk in foreignKeys)
    {
        await CollectAllForeignKeysRecursiveAsync(
            fk.ReferencingTableName, 
            dependencies, 
            visited, 
            depth + 1,  // 传递深度
            cancellationToken);
    }
}
```

## 🧪 调试步骤

### 步骤1：确认是否是栈溢出

运行程序，观察是否有以下现象：
- 程序突然退出，无任何错误提示
- Windows事件查看器中有 ".NET Runtime" 错误
- 错误代码包含 `0x800703E9` (StackOverflowException)

### 步骤2：查看完整日志

如果程序没有完全崩溃，查看日志文件中是否有：
```
[依赖收集] ❌ 递归分析 xxx 时出错: ...
```

这会告诉我们哪个表导致了问题。

### 步骤3：简化测试

选择一个外键关系较少的表进行测试：
1. 选择只有2-3个子表的表
2. 执行删除操作
3. 观察是否正常完成

如果小表正常，大表失败，说明确实是递归深度问题。

### 步骤4：使用Profiler工具

使用Visual Studio的性能分析器：
1. 启动性能分析
2. 执行删除操作
3. 查看调用堆栈
4. 确认是否有无限递归
