# 空表跳过优化 - 性能提升方案

## 📅 更新日期
2026-04-25

## ✅ 优化内容

### 1. **启动时预加载表记录数**
- ✅ 在工具打开时，异步查询所有表的记录数
- ✅ 将记录数缓存到 `_tableRecordCounts` 字典中
- ✅ 不阻塞UI，后台异步执行
- ✅ 每50个表更新一次进度状态

### 2. **TreeView显示记录数**
- ✅ 在TreeNode的文本中显示记录数，格式：`表名 (1,234)`
- ✅ 使用 `TreeNodeData` 封装类存储 EntityType、RecordCount、TableName
- ✅ 数字使用千分位格式化（N0），便于阅读

### 3. **依赖收集时跳过空表**
- ✅ 在 `CollectAllForeignKeysRecursiveAsync` 方法中，先检查表是否有数据
- ✅ 如果记录数为0，直接跳过外键查询和递归分析
- ✅ 日志中标记为 `ℹ️ 表 xxx 无数据，跳过依赖分析`
- ✅ 大幅减少不必要的数据库查询

### 4. **预查询ID时跳过空表**
- ✅ 在 `QueryChildIds` 递归函数中，先检查子表是否有数据
- ✅ 如果记录数为0，跳过主键ID查询
- ✅ 日志中标记为 `(无数据，跳过)`
- ✅ 避免对空表执行 SELECT 查询

## 🔧 技术实现

### UCBasicDataCleanup.cs 修改

#### 1. 添加字段
```csharp
private Dictionary<string, long> _tableRecordCounts; // 表名 -> 记录数的缓存
```

#### 2. 异步加载记录数
```csharp
private async Task LoadTableRecordCountsAsync()
{
    _tableRecordCounts = new Dictionary<string, long>();
    var db = MainForm.Instance.AppContext.Db;
    
    foreach (var entity in _allEntities)
    {
        var count = await db.Queryable<object>()
            .AS(entity.TableName)
            .CountAsync();
        
        _tableRecordCounts[entity.TableName] = count;
    }
    
    // 更新TreeView显示
    UpdateTreeViewWithRecordCounts();
}
```

#### 3. 更新TreeView节点
```csharp
private void UpdateTreeViewWithRecordCounts()
{
    foreach (TreeNode entityNode in categoryNode.Nodes)
    {
        if (entityNode.Tag is Type entityType)
        {
            var metadata = EntityRegistry.Entities.FirstOrDefault(e => e.EntityType == entityType);
            long recordCount = _tableRecordCounts[metadata.TableName];
            
            // 更新显示文本
            entityNode.Text = $"{metadata.DisplayName} ({recordCount:N0})";
            
            // 保存封装数据
            entityNode.Tag = new TreeNodeData
            {
                EntityType = entityType,
                RecordCount = recordCount,
                TableName = metadata.TableName
            };
        }
    }
}
```

#### 4. TreeNodeData 封装类
```csharp
private class TreeNodeData
{
    public Type EntityType { get; set; }
    public long RecordCount { get; set; }
    public string TableName { get; set; }
}
```

#### 5. 修改 GetSelectedEntityNodes
```csharp
private List<TreeNode> GetSelectedEntityNodes()
{
    var result = new List<TreeNode>();
    
    foreach (TreeNode entityNode in categoryNode.Nodes)
    {
        // 支持两种Tag类型：Type 或 TreeNodeData
        if (entityNode.Checked && (entityNode.Tag is Type || entityNode.Tag is TreeNodeData))
        {
            result.Add(entityNode);
        }
    }
    
    return result;
}
```

### DataCleanupEngine.cs 修改

#### 1. 依赖收集时跳过空表
```csharp
private async Task CollectAllForeignKeysRecursiveAsync(...)
{
    visited.Add(tableName);
    
    // ✅ 优化：先检查表是否有数据
    var recordCount = await db.Queryable<object>()
        .AS(tableName)
        .CountAsync();
    
    if (recordCount == 0)
    {
        Log($"[依赖收集]   ℹ️ 表 {tableName} 无数据，跳过依赖分析");
        dependencies[tableName] = new List<DbForeignKeyInfo>();
        return;
    }
    
    Log($"[依赖收集]   📊 表 {tableName} 有 {recordCount:N0} 条记录，继续分析");
    
    // 继续执行外键查询...
}
```

#### 2. 预查询ID时跳过空表
```csharp
async Task QueryChildIds(string parentTable, List<long> parentIds, string parentFkColumn)

            .CountAsync();

        {
            Log($"[预查询] {parentTable}.{parentFkColumn} → {childTable}.{childFkColumn} (无数据，跳过)");
            idMap[childTable] = new List<long>();
            continue;
        }
        
        // 继续执行主键ID查询...
    }
}
```

## 📊 性能提升效果

### 优化前的问题
从用户提供的日志可以看到：
```
[15:30:23.959] [依赖收集] → 正在分析表: tb_BOMChangeLog
[15:30:23.972] [依赖收集]   执行SQL: 查询 sys.foreign_keys WHERE referenced_table = tb_BOMChangeLog
[15:30:23.985] [依赖收集]   查询结果: 找到 0 个外键引用
[15:30:23.986] [依赖收集] → 正在分析表: tb_BOM_SDetail
...
[15:30:24.069] [依赖收集]   查询结果: 找到 55 个外键引用
[15:30:24.133] [依赖收集] → 正在分析表: tb_ManufacturingOrderDetail
[15:30:24.139] [依赖收集]   查询结果: 找到 0 个外键引用
```

**问题分析**：
- ❌ tb_BOMChangeLog 可能没有数据，但仍然执行了外键查询
- ❌ tb_ManufacturingOrderDetail 可能没有数据，但仍然执行了外键查询
- ❌ 每个空表都要执行一次 `sys.foreign_keys` 查询
- ❌ 如果有100个空表，就要执行100次无意义的查询

#### 场景2：预查询ID阶段

**优化前**：
```sql
-- 对每个子表都执行
SELECT [PrimaryKeyID] FROM [tb_EmptyTable1] WHERE [FK_ID] IN (@ids)
SELECT [PrimaryKeyID] FROM [tb_EmptyTable2] WHERE [FK_ID] IN (@ids)
SELECT [PrimaryKeyID] FROM [tb_EmptyTable3] WHERE [FK_ID] IN (@ids)
-- ... 即使这些表没有数据
```

**优化后**：
```sql
-- 先检查记录数
SELECT COUNT(*) FROM [tb_EmptyTable1] -- 返回0，跳过
SELECT COUNT(*) FROM [tb_EmptyTable2] -- 返回0，跳过
SELECT COUNT(*) FROM [tb_EmptyTable3] -- 返回0，跳过

-- 只对非空表执行
SELECT [PrimaryKeyID] FROM [tb_HasDataTable] WHERE [FK_ID] IN (@ids)
```

## 📝 注意事项

### 1. 启动时间
- ⚠️ 首次打开工具时，需要查询所有表的记录数
- ⚠️ 如果有200个表，可能需要10-20秒
- ✅ 异步执行，不阻塞UI
- ✅ 可以继续使用其他功能

### 2. 数据实时性
- ⚠️ 记录数是启动时查询的，不是实时的
- ⚠️ 如果在工具打开期间有其他操作修改了数据，记录数可能不准确
- ✅ 对于数据清理场景，这个误差可以接受
- ✅ 如果需要实时数据，可以添加"刷新记录数"按钮

### 3. COUNT查询性能
- ✅ COUNT(*) 是快速聚合查询
- ✅ 比外键查询（sys.foreign_keys JOIN）快得多
- ✅ 即使有200个表，总耗时也在可接受范围内

### 4. 空表判断
- ✅ 使用 `recordCount == 0` 判断空表
- ✅ 空表不会执行外键查询和ID查询
- ✅ 大幅减少数据库负载
