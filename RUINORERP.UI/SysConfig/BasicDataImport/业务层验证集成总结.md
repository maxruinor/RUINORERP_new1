public class ImportValidationAdapter
{

## 🎯 优势与价值

### 对用户的价值

1. **提前发现配置错误** - 在保存配置时就发现问题,避免导入时失败
2. **清晰的错误提示** - 明确指出哪个字段配置有问题,如何修正
3. **数据质量保障** - 导入时自动验证业务规则,确保数据合法性
4. **减少返工** - 避免因配置错误或数据问题导致的重复导入

### 对开发者的价值

1. **复用现有验证器** - 无需为导入功能单独编写验证逻辑
2. **集中管理规则** - 所有验证规则在 Validator 类中统一管理
3. **灵活扩展** - 新增验证规则只需修改对应 Validator 类
4. **低耦合** - 导入代码与验证逻辑解耦,便于维护

## 🔄 扩展示例

### 示例1: 为新实体添加验证器

假设要为 `tb_Product` 实体添加导入验证:

```csharp
namespace RUINORERP.Business.Validator
{
    public partial class tb_ProductValidator : BaseValidatorGeneric<tb_Product>
    {
        public tb_ProductValidator(ApplicationContext appContext = null) : base(appContext)
        {
            // 产品编码验证
            RuleFor(x => x.ProdCode)
                .NotEmpty().WithMessage("产品编码不能为空")
                .MaximumLength(50).WithMessage("产品编码不能超过50个字符");
            
            // 产品名称验证
            RuleFor(x => x.ProdName)
                .NotEmpty().WithMessage("产品名称不能为空")
                .MaximumLength(200).WithMessage("产品名称不能超过200个字符");
            
            // 产品价格验证
            RuleFor(x => x.Price)
                .GreaterThanOrEqualTo(0).WithMessage("产品价格不能为负数");
        }
    }
}
```

**效果**: 导入产品数据时,会自动调用上述验证规则,无需修改导入代码。

### 示例2: 扩展现有验证器

假设要增强供应商验证规则:

```csharp
// 在 tb_CustomerVendorValidator.cs 中添加
RuleFor(x => x.VendorTaxNo)
    .Must(BeValidTaxNo).WithMessage("税号格式不正确")
    .When(x => !string.IsNullOrEmpty(x.VendorTaxNo));

private bool BeValidTaxNo(string taxNo)
{
    // 税号验证逻辑
    return Regex.IsMatch(taxNo, @"^\d{15}$|^\d{18}$");
}
```

**效果**: 导入供应商数据时,会自动验证税号格式。

## ⚠️ 注意事项

1. **验证器命名规范**
   - 实体类名: `tb_CustomerVendor`
   - 验证器名: `tb_CustomerVendorValidator`
   - 命名空间: `RUINORERP.Business.Validator`

2. **构造函数要求**
   ```csharp
   public tb_CustomerVendorValidator(ApplicationContext appContext = null) 
       : base(appContext)
   {
       // 验证规则
   }
   ```

3. **没有验证器的情况**
   - 如果实体没有对应的 Validator 类,导入时会跳过验证
   - 不会报错,保证向后兼容性

4. **验证失败处理**
   - 配置保存时: 阻止保存,显示错误消息
   - 数据导入时: 记录失败行,继续处理其他行
