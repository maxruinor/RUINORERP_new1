# 基础数据导入 - 业务层验证集成说明

## 概述

本次更新在基础数据导入功能中集成了现有的 FluentValidation 业务验证架构体系,确保在**配置保存**和**数据导入**两个关键环节都能进行有效的数据验证。

## 核心设计原则

1. **不修改现有验证体系** - 完全复用 `RUINORERP.Business.Validator` 命名空间下的验证器
2. **最小化改动** - 仅添加适配层,不改变原有导入流程
3. **轻量级集成** - 通过 `ImportValidationAdapter` 统一调用验证逻辑

## 新增文件

### ImportValidationAdapter.cs
**位置**: `RUINORERP.UI.SysConfig.BasicDataImport\ImportValidationAdapter.cs`

**职责**:
- 验证列映射配置的完整性
- 验证整个导入配置的合法性
- 调用现有 FluentValidation 验证器验证实体对象

**主要方法**:
```csharp
// 验证单个列映射配置
bool ValidateColumnMapping(ColumnMapping mapping, out List<string> errors)

// 验证整个导入配置
bool ValidateImportConfiguration(ImportConfiguration config, out List<string> errors)

// 使用实体验证器验证实体对象
string ValidateEntity<T>(T entity) where T : class
```

## 集成点

### 1. 配置保存时验证

**位置**: `FrmColumnPropertyConfig.kbtnOK_Click`

**验证内容**:
- 系统字段不能为空
- 外键关联配置完整性(表名、字段名)
- 列拼接配置(至少2个源列)
- 枚举默认值配置
- 图片配置(命名规则)
- 业务键配置合理性

**示例**:
```csharp
// 构建临时映射对象用于验证
var tempMapping = new ColumnMapping { ... };

// 使用验证适配器进行配置验证
var validator = new ImportValidationAdapter();
if (!validator.ValidateColumnMapping(tempMapping, out List<string> validationErrors))
{
    MessageBox.Show("列配置验证失败:\n" + string.Join("\n", validationErrors));
    return;
}
```

### 2. 导入配置保存验证

**位置**: `frmColumnMappingConfig.kbtnSaveMapping_Click`

**验证内容**:
- 配置名称不能为空
- 至少需要一个列映射
- 所有列映射的单独验证
- 去重配置合理性
- 业务键存在性策略检查

**示例**:
```csharp
var validator = new ImportValidationAdapter();
if (!validator.ValidateImportConfiguration(ImportConfig, out List<string> validationErrors))
{
    MessageBox.Show("导入配置验证失败:\n" + string.Join("\n", validationErrors));
    return;
}
```

### 3. 数据导入时验证

**位置**: `DynamicImporter.ImportAsync` → `ValidateEntityWithValidator`

**验证内容**:
- 自动查找对应的 Validator 类(如 `tb_CustomerVendorValidator`)
- 调用 FluentValidation 的 Validate 方法
- 收集所有验证错误消息

**示例**:
```csharp
// 在导入循环中验证每个实体
var entity = CreateEntityFromRow(row, mappings, entityType, i + 2);

// 使用Validator进行业务验证
string validationError = ValidateEntityWithValidator(entity, entityType);
if (!string.IsNullOrEmpty(validationError))
{
    result.FailedRecords.Add(new FailedRecord { ... });
    result.FailedCount++;
    continue;
}
```

## 工作流程

```
用户操作                    验证环节                  验证结果
─────────────────────────────────────────────────────────────
配置列属性  ───────────→  ImportValidationAdapter  ──→  阻止保存并提示错误
保存导入配置 ──────────→  ImportValidationAdapter  ──→  阻止保存并提示错误
执行数据导入 ──────────→  DynamicImporter          ──→  记录失败行并继续
                         ↓
                     ValidateEntityWithValidator
                         ↓
                     查找对应 Validator 类
                         ↓
                     调用 FluentValidation
                         ↓
                     返回错误消息或空字符串
```

## 扩展现有验证器

如果需要在导入时使用新的验证规则,只需在对应的 Validator 类中添加规则即可:

**示例**: `tb_CustomerVendorValidator.cs`
```csharp
public partial class tb_CustomerVendorValidator : BaseValidatorGeneric<tb_CustomerVendor>
{
    public tb_CustomerVendorValidator(ApplicationContext appContext = null) : base(appContext)
    {
        // 添加新的验证规则
        RuleFor(x => x.VendorName)
            .NotEmpty().WithMessage("供应商名称不能为空")
            .MaximumLength(100).WithMessage("供应商名称不能超过100个字符");
        
        RuleFor(x => x.VendorCode)
            .NotEmpty().WithMessage("供应商编码不能为空")
            .Must(BeUniqueVendorCode).WithMessage("供应商编码已存在");
    }
}
```

导入时会自动调用这些验证规则,无需修改导入代码。

## 注意事项

1. **验证器命名规范**: 实体类名为 `tb_CustomerVendor`,验证器必须为 `tb_CustomerVendorValidator`
2. **命名空间**: 验证器必须在 `RUINORERP.Business.Validator` 命名空间下
3. **构造函数**: 验证器构造函数接受 `ApplicationContext` 参数(可为 null)
4. **性能考虑**: 验证在导入前逐行执行,大数据量时可能影响性能,建议先预览验证

## 优势

✅ **复用现有验证体系** - 无需重复编写验证逻辑  
✅ **集中管理验证规则** - 所有验证规则在 Validator 类中统一管理  
✅ **灵活扩展** - 新增验证规则只需修改对应 Validator 类  
✅ **不影响现有功能** - 没有验证器的实体类型会跳过验证  
✅ **友好错误提示** - 清晰的错误消息帮助用户修正数据  

## 后续优化建议

1. **批量验证**: 对于大数据量导入,可以考虑批量验证而非逐行验证
2. **异步验证**: 将验证过程改为异步,提升UI响应速度
3. **验证缓存**: 缓存验证结果,避免重复验证相同数据
4. **验证报告**: 生成详细的验证报告,包含所有错误和警告
