### 1. ValidationResult 类（新建）

**文件位置**：`RUINORERP.Business/Common/ValidationResult.cs`

**功能说明**：
- 用于业务层向UI层传递验证结果
- 避免在业务层直接弹出 MessageBox
- 支持三种验证状态：通过、失败、需要用户确认

**核心方法**：
```csharp
// 创建通过的验证结果
ValidationResult.Pass()

// 创建失败的验证结果
ValidationResult.Fail(string errorMessage)

// 创建需要用户确认的验证结果
ValidationResult.NeedConfirm(
    string title,
    string message,
    string confirmButtonText = "确定",
    string cancelButtonText = "取消"
)
```

#### 2.2 ApprovalAsync 方法调用处优化

**修改前**：
```csharp
if (!ValidatePaymentDetails(existingPendingPayments, entity, TotalOrderAmount, rmrs))
{
    rmrs.ErrorMsg = "验证预收付款明细的合法性不通过!审核失败。";
    return rmrs;
}
```

**修改后**：
```csharp
var validationResult = ValidatePaymentDetails(existingPendingPayments, entity, TotalOrderAmount, rmrs);

// 如果验证需要用户确认，返回特殊错误码，由UI层处理
if (validationResult.NeedUserConfirm)
{
    rmrs.ErrorMsg = $"NEED_CONFIRM:{validationResult.ConfirmTitle}:{validationResult.ConfirmMessage}";
    rmrs.Succeeded = false;
    rmrs.ReturnObject = entity as T;
    return rmrs;
}

// 如果验证失败，直接返回错误
if (!validationResult.IsValid)
{
    rmrs.ErrorMsg = validationResult.ErrorMessage ?? "验证预收付款明细的合法性不通过!审核失败。";
    rmrs.Succeeded = false;
    rmrs.ReturnObject = entity as T;
    return rmrs;
}
```

**关键改进**：
1. ✅ 使用特殊前缀 `NEED_CONFIRM:` 标识需要用户确认的情况
2. ✅ UI层可以根据此前缀决定是否弹出确认框
3. ✅ 保持业务层纯净，不包含任何UI逻辑

## 🎯 设计原则

### 1. 职责分离原则

| 层级 | 职责 | 禁止行为 |
|------|------|----------|
| **业务层** (Business) | 数据验证、业务规则检查 | ❌ 弹出 MessageBox<br>❌ 显示 Form<br>❌ 访问 UI 控件 |
| **UI层** (UI) | 用户交互、结果显示 | ✅ 根据业务层返回结果决定如何提示 |

### 2. 验证结果传递模式

```
┌─────────────────────┐
│   业务层 (Business)  │
│                     │
│  ValidatePayment... │──► 返回 ValidationResult
│                     │    - NeedUserConfirm
│                     │    - ConfirmTitle
│                     │    - ConfirmMessage
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│   Controller 层      │
│                     │
│  ApprovalAsync      │──► 封装为 NEED_CONFIRM: 前缀
│                     │    的错误消息
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│    UI 层            │
│                     │
│  解析错误消息        │──► 如果需要确认：
│                     │    弹出 MessageBox
│                     │   否则：
│                     │    显示错误提示
└─────────────────────┘
```

### 3. 错误消息格式规范

**需要用户确认的消息格式**：
```
NEED_CONFIRM:{标题}:{详细消息}
```

**示例**：
```
NEED_CONFIRM:超额确认:预收款单总金额1000(包含当前金额500)，超过了订单总金额800
业务类型：销售订单
来源单号：SO20240101

累计预收款金额：1000
订单总金额：800
超额金额：200

建议：
1. 检查是否有重复的预收款单
2. 调整预收款金额，使其不超过订单总额
3. 如确需超额，请与财务部门确认
```

**UI层解析示例**：
```csharp
if (result.ErrorMsg.StartsWith("NEED_CONFIRM:"))
{
    var parts = result.ErrorMsg.Split(new[] { ':' }, 3);
    string title = parts[1];
    string message = parts[2];
    
    if (MessageBox.Show(message, title, MessageBoxButtons.YesNo) == DialogResult.Yes)
    {
        // 用户确认，继续执行
        await ContinueApprovalWithConfirmation();
    }
}
```

// 在 UCSaleOrder 或相关UI界面中
var result = await controller.ApprovalAsync(entity);

}

## ✨ 优势总结

### 1. 符合架构原则
- ✅ 业务层不再依赖 UI 层
- ✅ 职责清晰，易于维护
- ✅ 便于单元测试（无需模拟 MessageBox）

### 2. 提升用户体验
- ✅ UI层可以自定义提示方式（MessageBox、Toast、Dialog等）
- ✅ 支持国际化（不同语言使用不同的提示文本）
- ✅ 可以记录用户的选择历史

### 3. 增强系统稳定性
- ✅ 避免在事务中弹出对话框导致的锁表问题
- ✅ 验证逻辑与展示逻辑分离，降低耦合
- ✅ 更容易实现批量操作和自动化流程

### 4. 便于扩展
- ✅ 可以轻松添加新的验证类型
- ✅ 支持异步验证
- ✅ 可以集成到工作流引擎中
