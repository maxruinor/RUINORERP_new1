### 【P0-2】简化心跳机制 ⭐⭐⭐⭐⭐

#### 问题分析
**当前架构**:
```
Server端:
  SessionInfo.HeartbeatCount/FailedCount (183-188行)
  SessionService.CleanupAndHeartbeatCallback (定时检查)
  _heartbeatTimeoutMultiplier (内存压力时调整)

Client端:
  ClientCommunicationService._heartbeatTask (独立线程)
  _heartbeatFailedAttempts (101行)
  HEARTBEAT_FAILURE_THRESHOLD = 5 (193行)
  OnConnectionStateChangedForHeartbeat (事件订阅)
```

**问题**:
1. Server和Client各自维护心跳状态,无法协调
2. 内存压力时动态调整超时倍数增加了复杂度
3. Client端心跳任务与ConnectionManager职责重叠

#### 优化方案

**方案A: 统一到ConnectionManager (推荐)**
```csharp
// ConnectionManager.cs
public class HeartbeatManager : IDisposable
{
    private Timer _heartbeatTimer;
    private int _failedCount = 0;
    private readonly int _maxFailures;
    
    public event Action<int> HeartbeatFailed; // 参数:失败次数
    public event Action HeartbeatRecovered;
    public event Action ThresholdReached;
    
    public void Start(TimeSpan interval)
    {
        _heartbeatTimer = new Timer(SendHeartbeat, null, interval, interval);
    }
    
    private async void SendHeartbeat(object state)
    {
        try
        {
            var success = await SendHeartbeatPacket();
            if (success)
            {
                if (_failedCount > 0)
                {
                    _failedCount = 0;
                    HeartbeatRecovered?.Invoke();
                }
            }
            else
            {
                _failedCount++;
                HeartbeatFailed?.Invoke(_failedCount);
                
                if (_failedCount >= _maxFailures)
                {
                    ThresholdReached?.Invoke();
                }
            }
        }
        catch (Exception ex)
        {
            _logger.LogError(ex, "发送心跳失败");
            _failedCount++;
        }
    }
}
```

**方案B: 保留现有架构但简化逻辑**
- 移除 `SessionService` 中的心跳计数 (由SuperSocket底层处理)
- Client端直接使用 `ConnectionManager.IsConnected` 判断
- 移除 `_heartbeatTimeoutMultiplier` 动态调整

**推荐**: 方案A,彻底解决分散问题

#### 重构步骤
1. 在 `ConnectionManager` 中新增 `HeartbeatManager` 内部类
2. 将 `ClientCommunicationService` 的心跳逻辑迁移过去
3. 移除 `SessionService` 中的心跳相关字段
4. 更新事件订阅关系

**预期收益**:
- 心跳逻辑集中管理
- 减少 ~200行分散代码
- 更容易调试心跳问题

### 【P0-3】移除PacketBuilder.BuildAndClone()重复 ⭐⭐⭐⭐

#### 问题分析
**重复功能**:

**问题**:

2. `PacketModel.Clone()` 几乎未被使用
3. 如果新增字段需要同步修改两处

#### 优化方案

**步骤1**: 在 `PacketModel` 中实现标准克隆方法
```csharp
// PacketModel.cs
public PacketModel CloneForResponse()
{
    var cloned = PacketModelPool.Rent();
    cloned.PacketId = IdGenerator.GenerateResponseId(this.PacketId);
    cloned.CommandId = this.CommandId;
    cloned.Direction = this.Direction == PacketDirection.ClientRequest 
        ? PacketDirection.ServerResponse 
        : this.Direction;
    cloned.Status = this.Status;
    cloned.SessionId = this.SessionId;
    cloned.CreatedTime = DateTime.Now;
    cloned.TimestampUtc = DateTime.UtcNow;
    
    // 复制执行上下文
    if (this.ExecutionContext != null)
    {
        cloned.ExecutionContext = new CommandContext
        {
            RequestId = this.ExecutionContext.RequestId,
            Token = this.ExecutionContext.Token,
            SessionId = this.ExecutionContext.SessionId
        };
    }
    
    // 复制扩展属性
    if (this.Extensions != null && this.Extensions.HasValues)
    {
        cloned.Extensions = new JObject(this.Extensions);
    }
    
    return cloned;
}
```

**步骤2**: 简化 `PacketBuilder.BuildAndClone()`
```csharp
// PacketBuilder.cs
public PacketModel BuildAndClone()
{
    var original = Build();
    return original.CloneForResponse(); // 委托给PacketModel
}
```

**步骤3**: 删除未使用的 `PacketModel.Clone()` (如果存在)

**预期收益**:
- 消除重复逻辑
- 单一数据源,易于维护
- 减少 ~40行代码

### 【P1-2】拆分锁管理服务 ⭐⭐⭐⭐

**当前职责** (`ClientLockManagementService`, 2102行):
1. 锁获取/释放 (核心)
2. 本地缓存管理 (`_clientCache`)
3. 锁恢复管理 (`_recoveryManager`)
4. 状态通知 (`_notificationService`)

**问题**:

```csharp
public interface ILockService
```

**重构步骤**:
1. 提取 `LockCacheService` (原 `ClientLocalLockCacheService`)
2. 提取 `LockRecoveryService` (原 `LockRecoveryManager`)
3. 提取 `LockNotificationService` (原 `LockStatusNotificationService`)
4. 简化 `LockService` 为核心锁操作
5. 使用依赖注入组装

**预期收益**:
- 每个类 < 500行

- 更容易单元测试
- 消除循环依赖

### 【P1-3】移除MemoryMonitoringService自动GC ⭐⭐⭐

#### 问题分析
**矛盾的设计**:
```csharp
// 注释说"避免高频GC导致CPU升高"
private long _autoGCThreshold = 2560; // 但又设置自动触发
private const int GC_COOLDOWN_SECONDS = 900;
private const int MAX_GC_ATTEMPTS_PER_HOUR = 4;

// ForceGarbageCollection() 和 PerformAutoGC() 逻辑完全相同
```

**问题**:
1. 手动GC会暂停所有线程(STW),影响用户体验
2. .NET GC已经很智能,手动干预往往适得其反
3. 自动GC策略与实际需求不符

#### 优化方案

**改为纯监控告警**:
```csharp
public class MemoryMonitoringService : IDisposable
{
    // 移除这些字段
    // private long _autoGCThreshold;
    // private long _lastGCTime;
    // private const int GC_COOLDOWN_SECONDS;
    
    // 只保留监控和告警
    public event EventHandler<MemoryWarningEventArgs> MemoryWarning;
    public event EventHandler<MemoryCriticalEventArgs> MemoryCritical;
    
    private void MonitorMemoryUsage(object state)
    {
        var memoryInfo = GetCurrentMemoryUsage();
        
        if (memoryInfo.WorkingSetMB >= CriticalThreshold)
        {
            _logger.LogWarning($"内存达到临界值: {memoryInfo.WorkingSetMB} MB");
            MemoryCritical?.Invoke(this, new MemoryCriticalEventArgs(memoryInfo));
            // 不再自动GC,由上层决定
        }
        else if (memoryInfo.WorkingSetMB >= WarningThreshold)
        {
            MemoryWarning?.Invoke(this, new MemoryWarningEventArgs(memoryInfo));
        }
    }
    
    // 保留手动GC方法,供管理员主动调用
    public void ForceGarbageCollection()
    {
        _logger.LogInformation("管理员触发强制GC");
        GC.Collect(GC.MaxGeneration, GCCollectionMode.Optimized, blocking: true, compacting: true);
        GC.WaitForPendingFinalizers();
        GC.Collect(GC.MaxGeneration, GCCollectionMode.Optimized, blocking: true, compacting: true);
    }
}
```

**上层订阅示例**:
```csharp
// frmMainNew.cs
_memoryMonitor.MemoryCritical += (sender, e) =>
{
    // 记录日志
    // 通知管理员
    // 可选: 调用 ForceGarbageCollection()
};
```

**预期收益**:
- 移除 ~50行自动GC逻辑
- 避免STW导致的卡顿
- 更灵活的内存管理策略
