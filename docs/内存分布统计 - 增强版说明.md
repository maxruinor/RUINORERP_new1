# 内存分布统计 - 增强版

## ✅ 新增功能（2026-04-23）

### 1. 新增内存统计模块

#### 工作流引擎 (WorkflowCore)
- **统计内容**: 运行中的工作流实例、运行数据、事件订阅
- **估算方法**: 通过反射获取 `IWorkflowHost.RunningWorkflows`
- **默认估算**: 20 MB（无法获取实际数据时）
- **子项**:
  - 运行中的工作流：每个实例约 1MB

#### 数据库连接池 (SqlSugar)
- **统计内容**: 数据库连接、命令缓存、结果集缓存
- **估算方法**: 基于连接池配置和活跃连接数
- **默认估算**: 150 MB（50 个连接 × 3MB）
- **子项**:
  - 连接池：50 个连接 × 3MB = 150MB
  - 命令缓存：10MB
  - 查询结果缓存：20MB

#### 事件处理器 (EventHandlers)
- **统计内容**: 事件订阅、委托引用、事件参数
- **估算方法**: 分类统计各主要事件源
- **默认估算**: 25 MB
- **子项**:
  - SessionService 事件：5MB
  - WorkflowCore 事件：10MB
  - Cache 事件：3MB
  - UI 事件：2MB
  - 其他事件：5MB

#### 委托缓存 (DelegateCache)
- **统计内容**: Lambda 表达式、闭包、动态生成的委托
- **估算方法**: 分类统计各类委托缓存
- **默认估算**: 30 MB
- **子项**:
  - LINQ 编译缓存：10MB
  - 动态代理（Castle）：10MB
  - 反射委托：5MB
  - 其他委托：5MB

#### 反射缓存 (ReflectionCache)
- **统计内容**: Type 信息、PropertyInfo、MethodInfo、Attribute 等
- **估算方法**: 分类统计各类反射缓存
- **默认估算**: 20 MB
- **子项**:
  - Type 信息缓存：8MB
  - MemberInfo 缓存：6MB
  - Attribute 缓存：3MB
  - 表达式树缓存：3MB

#### 其他托管内存 (Other)
- **统计内容**: 未分类的托管内存、临时对象、GC 堆碎片等
- **估算方法**: 总托管内存的 15%（至少 50MB）
- **默认估算**: 总托管内存 × 15%
- **子项**:
  - 临时对象：10MB
  - GC 堆碎片：20MB
  - 大对象堆 (LOH)：15MB
  - CLR 内部结构：10MB

## 🎯 内存占用热点识别

### 新增热点模块

#### 1. 数据库连接池 (150 MB)
**可能的问题**:
- 连接池过大（默认 100 个连接）
- 连接泄漏（未正确释放）
- 查询结果集过大

**优化建议**:
- 检查 `appsettings.json` 中的连接池配置
- 使用 `using` 语句确保正确释放连接
- 避免一次性查询大量数据

#### 2. 委托缓存 + 反射缓存 (50 MB)
**可能的问题**:
- 大量动态生成的代码
- 反射操作频繁
- 缓存未清理

**优化建议**:
- 使用缓存（如 `Expression.Lambda` 编译结果）
- 减少运行时反射
- 使用 `MethodInfo.CreateDelegate()` 代替 `Invoke()`

#### 3. 事件处理器 (25 MB)
**可能的问题**:
- 事件泄漏（订阅后未取消）
- 匿名方法捕获外部变量

**优化建议**:
- 及时取消事件订阅（`-=`）
- 使用弱事件模式
- 避免在事件处理中创建大对象

#### 4. 工作流引擎 (20 MB)
**可能的问题**:
- 工作流实例堆积
- 工作流数据过大

**优化建议**:
- 定期清理已完成的工作流
- 使用持久化存储工作流数据
- 优化工作流设计，减少内存占用

## 📈 内存估算方法说明

### 估算 vs 实际

**估算模块**:
- 数据库连接池：基于配置和连接数估算
- 事件处理器：基于事件源分类估算
- 委托缓存：基于缓存类型估算
- 反射缓存：基于反射使用程度估算
- 其他托管内存：基于总内存比例估算

**实际统计模块**:
- 实体缓存：实际统计每个表的缓存
- 会话管理：实际统计每个会话
- 智能提醒：实际统计规则数量
- 规则引擎：实际统计缓存对象

### 为什么需要估算？

.NET 运行时没有提供直接的 API 来获取：
- 委托缓存大小
- 反射缓存大小
- 事件处理器大小
- 数据库连接池大小

因此，我们基于：
1. **配置参数**（如连接池大小）
2. **对象数量**（如会话数、工作流数）
3. **经验值**（如每个对象平均大小）
4. **比例估算**（如总内存的 15%）

来进行合理估算。

## ⚠️ 注意事项

### 1. 估算值 vs 实际值
- 估算值仅供参考，可能与实际值有偏差
- 重点关注**趋势**而非绝对值
- 如果某模块占用异常高，需要深入分析

### 2. 会话详情限制
- 只显示前 10 个会话的详情
- 超过 10 个会话显示汇总
- 避免日志输出过长

### 3. 性能影响
- 详细统计会增加少量 CPU 开销
- 反射获取会话详情约 10-50ms
- 建议保持 1 分钟的分析间隔
