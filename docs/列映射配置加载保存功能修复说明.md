# 列映射配置加载和保存功能修复说明

## 修复日期
2026-04-26

## 问题概述
基础数据导入功能的列映射配置在加载和保存时存在多个问题,导致配置无法正确加载、保存后无法自动选中、字段列表恢复不完整等。

## 修复的问题列表

### 问题1: 保存配置后无法自动选中 ⚠️ → ✅ 已修复

**影响文件**: 
- `frmColumnMappingConfig.cs` (第999行)
- `UCBasicDataImport.cs` (第551行, 第1292行)

**问题描述**:
保存配置后,尝试通过 `comboBoxSavedMappings.Items.IndexOf(mappingName)` 选中新保存的配置,但下拉列表中的项格式是 `[配置] {mappingName}`,导致匹配失败,无法自动选中。

**修复方案**:
```csharp
// 修复前
comboBoxSavedMappings.SelectedIndex = comboBoxSavedMappings.Items.IndexOf(mappingName);

// 修复后
string newItemText = $"[配置] {mappingName}";
int newIndex = comboBoxSavedMappings.Items.IndexOf(newItemText);
if (newIndex >= 0)
{
    comboBoxSavedMappings.SelectedIndex = newIndex;
}
```

**同时修复 UCBasicDataImport 中的类似问题**:
```csharp
// 修复前
int index = kcmbDynamicMappingName.FindStringExact(frmMapping.SavedMappingName);

// 修复后
string configText = $"[配置] {frmMapping.SavedMappingName}";
int index = kcmbDynamicMappingName.FindStringExact(configText);
```

### 问题2: 加载配置时名称包含前缀导致文件路径错误 ⚠️ → ✅ 已修复

**影响文件**: 
- `frmColumnMappingConfig.cs` (第1056行)

**问题描述**:
从下拉列表选择配置时,获取的 `selectedItem` 包含 `[配置] ` 前缀,直接用作文件名会导致路径错误,例如:
- 期望路径: `C:\...\Configs\产品导入.xml`
- 实际路径: `C:\...\Configs\[配置] 产品导入.xml` ❌

**修复方案**:
```csharp
// 修复前
string mappingName = comboBoxSavedMappings.SelectedItem.ToString();
var config = _columnMappingManager.LoadConfiguration(mappingName, TargetEntityType);

// 修复后
string mappingName = selectedItem;
if (selectedItem.StartsWith("[配置] "))
{
    mappingName = selectedItem.Substring(5); // 移除 "[配置] " 前缀
}
var config = _columnMappingManager.LoadConfiguration(mappingName, TargetEntityType);
```

### 问题3: 加载配置后字段列表恢复不完整 ⚠️ → ✅ 已修复

**影响文件**: 
- `frmColumnMappingConfig.cs` (第1077-1081行)

**问题描述**:
加载配置后,需要将所有已映射的字段从可用列表中移除,但原代码没有区分真正的Excel列和特殊数据来源(如系统生成、默认值、外键关联等),导致:
1. 特殊数据来源的"虚拟列"被错误地添加到Excel列列表
2. 空值检查缺失可能导致NullReferenceException

**修复方案**:
```csharp
// 修复前
foreach (var mapping in ColumnMappings)
{
    RemoveFromExcelColumns(mapping.ExcelColumn);
    RemoveFromSystemFields(mapping.SystemField?.Key);
}

// 修复后
foreach (var mapping in ColumnMappings)
{
    // 只有真正的Excel列才需要从可用列表中移除
    if (!string.IsNullOrEmpty(mapping.ExcelColumn) && 
        !mapping.ExcelColumn.StartsWith("[") && 
        !mapping.ExcelColumn.Contains("]"))
    {
        RemoveFromExcelColumns(mapping.ExcelColumn);
    }
    
    if (mapping.SystemField != null && !string.IsNullOrEmpty(mapping.SystemField.Key))
    {
        RemoveFromSystemFields(mapping.SystemField.Key);
    }
}
```

**判断规则**:
- 真正的Excel列: `产品名称`、`规格`、`颜色` 等
- 特殊数据来源: `[系统生成] ID`、`[默认值:未知] 状态`、`[外键关联:供应商] 供应商ID` 等

### 问题4: 构造函数调用不规范 ⚠️ → ✅ 已修复

**影响文件**: 
- `UCBasicDataImport.cs` (第536行, 第1274行)

**问题描述**:
创建 `frmColumnMappingConfig` 实例时使用无参构造函数,然后手动设置属性,这违反了构造函数的设计意图,且可能导致初始化顺序问题。

**修复方案**:
```csharp
// 修复前
using (var frmMapping = new frmColumnMappingConfig())
{
    frmMapping.ExcelData = _rawExcelData;
    frmMapping.TargetEntityType = _selectedEntityType;
    frmMapping.ImportConfig = _currentConfig;
    // ...
}

// 修复后
using (var frmMapping = new frmColumnMappingConfig(_selectedEntityType, _rawExcelData))
{
    frmMapping.ImportConfig = _currentConfig;
    frmMapping.IsEditMode = true;
    frmMapping.OriginalMappingName = selectedConfig;
    // ...
}
```

**优势**:
1. 使用带参数的构造函数确保必要参数在初始化时就设置
2. 减少手动赋值,降低遗漏风险
3. 符合面向对象设计原则

### 问题5: 缺少空值检查 ⚠️ → ✅ 已修复

**影响文件**: 
- `frmColumnMappingConfig.cs` (第1068行)

**问题描述**:
加载配置后没有检查返回的配置对象是否为null,如果配置文件损坏或反序列化失败,后续操作会抛出NullReferenceException。

**修复方案**:
```csharp
// 修复前
var config = _columnMappingManager.LoadConfiguration(mappingName, TargetEntityType);
ImportConfig = config;

// 修复后
var config = _columnMappingManager.LoadConfiguration(mappingName, TargetEntityType);

if (config == null)
{
    MessageBox.Show($"加载配置失败: 配置数据为空", "错误", MessageBoxButtons.OK, MessageBoxIcon.Error);
    return;
}

ImportConfig = config;
```

## 技术细节

### 配置命名规范

为了区分普通配置和模板,采用以下命名约定:

| 类型 | 显示格式 | 存储格式 | 示例 |
|------|---------|---------|------|
| 新建映射 | `-- 新建映射 --` | N/A | 索引0 |
| 普通配置 | `[配置] {名称}` | `{名称}.xml` | `[配置] 产品导入` → `产品导入.xml` |
| 导入模板 | `[模板] {名称}` | JSON文件 | `[模板] 标准产品模板` |

### 字段恢复逻辑

加载配置后的字段恢复流程:

```
1. 重新加载所有系统字段到 listBoxSystemFields
2. 重新加载所有Excel列到 listBoxExcelColumns
3. 遍历 ColumnMappings:
   ├─ 如果是真正的Excel列 → 从 listBoxExcelColumns 移除
   ├─ 如果是特殊数据来源 → 不移除(保持可用)
   └─ 所有系统字段 → 从 listBoxSystemFields 移除
4. 更新映射列表显示 UpdateMappingsList()
```

### 特殊数据来源标识

| 数据类型 | 显示格式 | 是否从Excel列表移除 |
|---------|---------|-------------------|
| Excel数据源 | `产品名称` | ✅ 是 |
| 系统生成 | `[系统生成] ID` | ❌ 否 |
| 默认值 | `[默认值:未知] 状态` | ❌ 否 |
| 外键关联 | `[外键关联:供应商] 供应商ID` | ❌ 否 |
| 自身引用 | `[自身引用:父级ID] ParentID` | ❌ 否 |
| 字段复制 | `[字段复制:编码] Code` | ❌ 否 |
| 列拼接 | `[列拼接:姓+名] FullName` | ❌ 否 |
| Excel图片 | `[Excel图片:AutoIncrement] Photo` | ❌ 否 |

- [ ] 所有从外部加载的数据都要进行null检查
