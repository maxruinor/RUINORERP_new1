# 图片上传并发控制优化 - 修复总结

## 📋 修复概述

本次修复完善了图片上传功能的并发控制机制，实现了**以业务数据行主键ID为粒度的细粒度锁**，确保：
- ✅ 同一行数据的图片操作串行化（避免冲突）
- ✅ 不同行数据的图片操作并行化（提升性能）
- ✅ 批量上传场景下的并发限流（防止资源耗尽）
- ✅ 自动清理空闲锁（防止内存泄漏）

## 🔧 核心修复内容

### 1. FileManagementService.cs - 服务端锁机制完善

#### 1.1 添加锁清理定时器

```csharp
// 新增字段
private readonly System.Threading.Timer _lockCleanupTimer;

// 构造函数中启动定时器
public FileManagementService(...)
{
    // ...
    _lockCleanupTimer = new System.Threading.Timer(
        _ => CleanupIdleLocks(),
        null,
        TimeSpan.FromMinutes(5),  // 5分钟后开始
        TimeSpan.FromMinutes(5)   // 每5分钟执行一次
    );
}
```

**作用**: 定期清理不再使用的业务锁，防止 `_businessLocks` 字典无限增长导致内存泄漏。

#### 1.2 实现智能锁清理机制

```csharp
/// <summary>
/// 定期清理空闲的业务锁（防止内存泄漏）
/// </summary>
private void CleanupIdleLocks()
{
    try
    {
        // 找出所有空闲的锁（CurrentCount == 1 表示未被占用）
        var idleLocks = _businessLocks
            .Where(kvp => kvp.Value.CurrentCount == MaxConcurrentUploadsPerBusiness)
            .Select(kvp => kvp.Key)
            .ToList();
        
        foreach (var businessId in idleLocks)
        {
            // 双重检查，避免竞态条件
            if (_businessLocks.TryGetValue(businessId, out var lockObj) &&
                lockObj.CurrentCount == MaxConcurrentUploadsPerBusiness)
            {
                TryCleanupBusinessLock(businessId);
            }
        }
        
        if (idleLocks.Count > 0)
        {
            _log?.LogDebug("清理了{Count}个空闲业务锁，当前活跃锁数量: {ActiveCount}",
                idleLocks.Count, _businessLocks.Count);
        }
    }
    catch (Exception ex)
    {
        _log?.LogWarning(ex, "清理空闲锁时发生异常");
    }
}
```

**优势**:
- 只在锁空闲时才清理，不影响正在进行的上传
- 双重检查机制避免竞态条件
- 记录日志便于监控

#### 1.4 完善 Dispose 方法

```csharp
public void Dispose()
{
    if (_isDisposed) return;
    _isDisposed = true;

    try
    {
        // 1. 停止清理定时器
        _lockCleanupTimer?.Dispose();
        
        // 2. 清理所有业务锁
        foreach (var kvp in _businessLocks)
        {
            try
            {
                kvp.Value?.Dispose();
            }
            catch (Exception ex)
            {
                _log?.LogWarning(ex, "释放业务锁失败: BusinessId={BusinessId}", kvp.Key);
            }
        }
        _businessLocks.Clear();
        
        // 3. 清理其他资源
        _lockDictionaryLock?.Dispose();
        
        _log?.LogInformation("FileManagementService资源已释放");
    }
    catch (Exception ex)
    {
        _log?.LogError(ex, "释放FileManagementService资源时发生异常");
    }
}
```

### 2. UCProductList.cs - 批量上传并行化优化

#### 2.1 添加并发限流器

```csharp
/// <summary>
/// ✅ 新增: 批量上传并发限流器（最多同时3个上传任务）
/// </summary>
private readonly System.Threading.SemaphoreSlim _uploadThrottle = new System.Threading.SemaphoreSlim(3, 3);
```

**设计理由**:
- 限制并发数为3，平衡性能和服务器负载
- 避免大量SKU同时上传导致网络拥塞或服务器压力过大
- 可根据实际情况调整（建议2-5之间）

#### 2.2 重构 Save() 方法 - 从串行改为并行

**修复前（串行处理）**:
```csharp
foreach (var product in list)
{
    foreach (var detail in product.tb_ProdDetails)
    {
        await fileService.UploadImageAsync(...);  // 逐个等待
    }
}
```

**修复后（并行处理）**:
```csharp
// 第一步: 收集所有上传任务
var uploadTasks = new List<Task<UploadResult>>();

foreach (var product in list)
{
    if (product.HasUnsavedImageChanges)
        uploadTasks.Add(UploadProductMainImageAsync(product));
    
    foreach (var detail in product.tb_ProdDetails)
    {
        if (detail.HasUnsavedImageChanges)
            uploadTasks.Add(UploadSkuImageAsync(product, detail));
    }
}

// 第二步: 并行执行（最多3个并发）
var results = await Task.WhenAll(uploadTasks);

// 第三步: 统计结果
int totalSuccessCount = results.Count(r => r.IsSuccess);
int totalFailCount = results.Count(r => !r.IsSuccess);
```

**性能提升**:
- 假设15个图片上传，每个平均3秒
- **串行**: 15 × 3 = 45秒
- **并行(3个并发)**: 15 ÷ 3 × 3 = 15秒
- **提升**: 约 **67%** ⚡

#### 2.3 实现带限流的上传方法

```csharp
/// <summary>
/// ✅ 新增: 并行上传产品主图（带限流）
/// </summary>
private async Task<UploadResult> UploadProductMainImageAsync(tb_Prod product)
{
    await _uploadThrottle.WaitAsync();  // 获取许可
    try
    {
        return await ProcessProductMainImageInternalAsync(product);
    }
    finally
    {
        _uploadThrottle.Release();  // 释放许可
    }
}

/// <summary>
/// ✅ 新增: 并行上传SKU图片（带限流）
/// </summary>
private async Task<UploadResult> UploadSkuImageAsync(tb_Prod product, tb_ProdDetail detail)
{
    await _uploadThrottle.WaitAsync();
    try
    {
        return await ProcessSkuImageInternalAsync(product, detail);
    }
    finally
    {
        _uploadThrottle.Release();
    }
}
```

**工作原理**:
1. 任务开始时调用 `WaitAsync()` 获取许可
2. 如果已有3个任务在运行，新任务会等待
3. 任务完成后调用 `Release()` 释放许可，允许下一个任务开始
4. `finally` 块确保即使异常也会释放许可

## 📊 锁机制架构说明

### 锁的层次结构

```
┌─────────────────────────────────────────────────────────────┐
│                     UI层 (UCProductList)                      │
│                                                               │
│  _uploadThrottle (SemaphoreSlim, max=3)                      │
│  ├─ 控制全局并发上传数（最多3个）                              │
│  └─ 防止客户端资源耗尽                                        │
└──────────────────────┬──────────────────────────────────────┘
                       │ 网络请求
┌──────────────────────▼──────────────────────────────────────┐
│               服务层 (FileManagementService)                  │
│                                                               │
│  _businessLocks (ConcurrentDictionary<long, SemaphoreSlim>)  │
│  ├─ BusinessId=1001 → SemaphoreSlim(1,1)  [产品A]           │
│  ├─ BusinessId=1002 → SemaphoreSlim(1,1)  [产品B]           │
│  ├─ BusinessId=2001 → SemaphoreSlim(1,1)  [SKU X]           │
│  └─ BusinessId=2002 → SemaphoreSlim(1,1)  [SKU Y]           │
│                                                               │
│  特点:                                                         │
│  ✓ 不同BusinessId可并行                                       │
│  ✓ 相同BusinessId串行化                                       │
│  ✓ 自动清理空闲锁（每5分钟）                                   │
└──────────────────────┬──────────────────────────────────────┘
                       │ 网络请求
┌──────────────────────▼──────────────────────────────────────┐
│              服务器层 (FileCommandHandler)                    │
│                                                               │
│  事务控制 + 文件存储                                          │
└─────────────────────────────────────────────────────────────┘
```

### 锁的使用场景

| 场景 | 锁类型 | 示例 |
|------|--------|------|
| 产品A的主图上传 | BusinessId=ProdBaseID | 锁定该产品 |
| SKU X的图片上传 | BusinessId=ProdDetailID | 锁定该SKU |
| 产品A和产品B同时上传 | 不同BusinessId | **可并行** ✅ |
| SKU X和SKU Y同时上传 | 不同BusinessId | **可并行** ✅ |
| 同一SKU的多个图片 | 相同BusinessId | **串行化** ✅ |

## 🎯 关键设计决策

### 1. 为什么选择按BusinessId加锁？

**原因**:
- ✅ 符合业务语义：同一行数据的图片操作需要原子性
- ✅ 避免冲突：防止同一行的并发修改导致数据不一致
- ✅ 最大化并发：不同行的操作互不干扰

**对比全局锁**:
```
全局锁:     [上传1][上传2][上传3][上传4]...  (完全串行，慢)
BusinessId锁: [上传1][上传2]  (不同ID并行，快)
              [上传3][上传4]
```

### 2. 为什么UI层还要加限流器？

**原因**:
- ✅ 控制客户端资源：避免创建过多Task导致内存占用
- ✅ 控制网络并发：避免同时发起太多HTTP请求
- ✅ 保护服务器：避免瞬间大量请求冲击服务器

**两层控制的职责**:
- **UI层限流器**: 控制并发任务数（宏观控制）
- **服务层业务锁**: 保证数据一致性（微观控制）

### 3. 为什么需要定期清理锁？

**问题**:
```csharp
_businessLocks.GetOrAdd(businessId, ...)  // 每次都会创建新锁
// 如果没有清理，字典会无限增长
```

**解决方案**:
- 每5分钟扫描一次
- 清理 `CurrentCount == 1` 的空闲锁
- 双重检查避免竞态条件

**效果**:
- 正常使用时：保持活跃锁，清理空闲锁
- 长时间运行后：字典大小稳定在活跃业务数
