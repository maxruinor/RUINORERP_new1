## 1. 背景与问题

### 1.1 核心痛点

在之前的修复中,我们解决了**服务器端缓存不包含FileData二进制数据**的问题,但发现客户端UI组件在加载图片时,**未充分利用现有的客户端缓存机制**,导致:

1. **重复网络请求**: 同一张图片在短时间(1小时内)被多次下载
2. **用户体验差**: 每次打开单据都要等待图片加载(100-500ms)
3. **服务器压力大**: 大量重复的文件IO和网络传输

### 1.2 典型场景

```
用户A上午10:00查看费用报销单EXP-20260416-001
  → 下载结案凭证图片(FileId=12345, 500KB) → 耗时300ms

用户A上午10:30再次查看同一报销单
  → ❌ 又重新下载同一张图片 → 耗时300ms (浪费!)
  
用户B下午14:00审核该报销单
  → ❌ 又下载同一张图片 → 耗时350ms (跨用户无法共享)
```

## 3. 分层缓存策略

### 3.1 策略设计原则

根据**访问频率**、**数据量**、**时效性**三个维度,制定差异化缓存策略:

| 维度 | 产品图片 | 单据凭证 |
|------|---------|---------|
| **访问频率** | 🔥🔥🔥 高频 (每日多次查看) | 🔥🔥 中频 (审批流程中反复查看) |
| **数据量** | 小 (50-500KB/张) | 中 (100KB-2MB/张) |
| **更新频率** | 低 (产品确定后很少改) | 中 (可能补充凭证) |
| **共享范围** | 全公司共享 | 仅相关人员查看 |
| **时效要求** | 宽松 (允许几小时延迟) | 严格 (需要最新凭证) |

### 3.2 推荐策略

#### 策略1: 产品图片 - 双层缓存

```
┌─────────────────────────────────────────┐
│         客户端缓存 (IMemoryCache)        │
│  配置: 24h滑动过期 + 7天绝对过期         │
│  命中率: ~85% (同一用户频繁查看)         │
└──────────────┬──────────────────────────┘
               │ 未命中 ↓
┌──────────────▼──────────────────────────┐
│       服务器端缓存 (IMemoryCache)        │
│  配置: 8h绝对过期 + 4h滑动过期           │
│  命中率: ~60% (不同用户查看同一产品)     │
└──────────────┬──────────────────────────┘
               │ 未命中 ↓
┌──────────────▼──────────────────────────┐
│          文件系统读取 (SSD/HDD)          │
│  耗时: 5-20ms (本地磁盘IO)              │
└─────────────────────────────────────────┘
```

**优势**:
- ✅ 首次查看: 300-500ms (网络+文件IO)
- ✅ 同用户二次查看: 1-5ms (内存读取)
- ✅ 跨用户查看: 50-100ms (服务器缓存命中)
- ✅ 节省带宽: 减少80%的网络传输

**适用场景**:
- UCProdQuery 产品查询界面
- frmSKUImageEdit SKU图片编辑
- 任何展示产品主图/SKU图的界面

---

#### 策略2: 单据凭证 - 仅客户端缓存

```
┌─────────────────────────────────────────┐
│         客户端缓存 (IMemoryCache)        │
│  配置: 1h滑动过期 + 24h绝对过期          │
│  命中率: ~70% (同一用户在审批流程中)     │
└──────────────┬──────────────────────────┘
               │ 未命中 ↓
┌──────────────▼──────────────────────────┐
│      服务器端不缓存 (直接读文件系统)      │
│  原因: 命中率低,内存浪费,时效性强        │
└──────────────┬──────────────────────────┘
               │
┌──────────────▼──────────────────────────┐
│          文件系统读取 (SSD/HDD)          │
│  耗时: 5-20ms (本地磁盘IO)              │
└─────────────────────────────────────────┘
```

**为什么不放在服务器端缓存?**

1. **命中率低**: 不同用户查看不同单据,分散度高
   - 例: 用户A看EXP-001,用户B看EXP-002,用户C看EXP-003
   - 服务器缓存1000个单据图片,命中率可能只有5%

2. **内存浪费**: 服务器需缓存大量一次性图片
   - 假设100个并发用户,每人查看10个不同单据
   - 需要缓存1000张图片 × 500KB = 500MB内存
   - 但这些图片可能永远不会被第二次访问

3. **时效性强**: 凭证可能更新,需要最新数据
   - 费用报销可能被补充凭证
   - 销售订单可能修改订金凭证
   - 服务器缓存可能导致显示旧数据

4. **客户端足够**: 同一用户在短时间内会重复查看同一单据
   - 审批人可能在1小时内多次打开同一单据
   - 客户端缓存1小时滑动过期完全够用

**优势**:
- ✅ 首次查看: 100-500ms (网络+文件IO)
- ✅ 同用户二次查看(1小时内): 1-5ms (内存读取)
- ✅ 节省服务器内存: 不缓存低频数据
- ✅ 保证数据新鲜度: 直接从文件系统读取最新数据

**适用场景**:
- UCExpenseClaim 费用报销单 (结案凭证 + 明细凭证)
- UCSaleOrder 销售订单 (订金付款凭证)
- 其他业务单据的附件/凭证图片

```csharp
// ✅ 新代码: 封装DownloadImageWithCacheAsync方法
private async Task DownloadImageWithCacheAsync<T>(
    BaseEntity entity, 
    MagicPictureBox pictureBox, 
    Expression<Func<T, object>> exp) where T : BaseEntity
{
    var imageCacheService = Startup.GetFromFac<ImageCacheService>();
    
    // 解析FileId
    long fileId = ParseFileIdFromString(fieldValue.ToString());
    if (fileId <= 0) return;
    
    // ✅ 优先检查客户端缓存
    if (imageCacheService != null)
    {
        var cachedStorageInfo = imageCacheService.GetImageInfo(fileId);
        if (cachedStorageInfo != null && storageInfo.FileData != null)
        {
            MainForm.Instance.Invoke(() => LoadImageFromCache(pictureBox, storageInfo));
            MainForm.Instance.uclog.AddLog($"✅ 订金凭证缓存命中: FileId={fileId}");
            return;
        }
    }
    
    // 缓存未命中,调用原下载方法(会自动缓存)
    await DownloadImageAsync(entity, pictureBox, exp);
}
```

```csharp
// ✅ 实际预加载逻辑
private async Task DoPreloadBufferImagesAsync()
{
    // 1. 获取可见行范围
    int firstVisibleRow = newSumDataGridView产品.FirstDisplayedScrollingRowIndex;
    int visibleRowCount = newSumDataGridView产品.DisplayedRowCount(false);
    
    // 2. 计算预加载范围: 可见行 + 下方10行
    int preloadStartRow = firstVisibleRow + visibleRowCount;
    int preloadEndRow = Math.Min(preloadStartRow + PRELOAD_BUFFER_ROWS, totalRows - 1);
    
    // 3. 收集需要预加载的FileId(跳过已预加载的)
    var fileIdsToPreload = new List<long>();
    for (int i = preloadStartRow; i <= preloadEndRow; i++)
    {
        var fileId = GetFileIdFromViewProdDetail(row.DataBoundItem);
        if (fileId.HasValue && !_preloadedFileIds.Contains(fileId.Value))
        {
            fileIdsToPreload.Add(fileId.Value);
        }
    }
    
    // 4. 批量加载到缓存
    await imageCacheService.GetImageInfosBatchAsync(fileIdsToPreload);
    
    // 5. 标记为已预加载
    foreach (var fileId in fileIdsToPreload)
    {
        _preloadedFileIds.Add(fileId);
    }
}
```

### Phase 3: 分布式缓存 (可选,大规模部署)

**目标**: 多客户端共享缓存,进一步减少服务器压力

**方案**: 引入Redis作为分布式缓存层

```
┌──────────────┐     ┌──────────────┐
│  客户端A      │     │  客户端B      │
│  LocalCache  │     │  LocalCache  │
└──────┬───────┘     └──────┬───────┘
       │                    │
       └────────┬───────────┘
                │
       ┌────────▼───────────┐
       │   Redis Cluster    │
       │  (分布式缓存层)     │
       └────────┬───────────┘
                │
       ┌────────▼───────────┐
       │  服务器端缓存       │
       └────────┬───────────┘
                │
       ┌────────▼───────────┐
       │   文件系统          │
       └────────────────────┘
```

**优势**:
- ✅ 客户端A缓存的图片,客户端B可直接使用
- ✅ 服务器重启后缓存不丢失
- ✅ 支持水平扩展 (多服务器共享缓存)

**成本**:
- ⚠️ 需要部署Redis集群
- ⚠️ 增加系统复杂度
- ⚠️ 适合100+并发用户的大规模部署

### C. 常见问题

**Q1: 缓存会不会导致显示旧图片?**  
A: 不会。单据凭证设置了1小时滑动过期+24小时绝对过期,确保数据新鲜度。如果图片更新,FileId会变化,自动加载新图片。

**Q2: 内存占用会不会过大?**  
A: 不会。客户端缓存默认限制为200MB,超过后LRU自动清理。实际测试中,正常使用约50-80MB。

**Q3: 多用户如何共享缓存?**  
A: 当前架构下,每个客户端独立缓存。如需共享,可升级到Phase 3分布式缓存(Redis)。

**Q4: 缓存失效怎么办?**  
A: 缓存失效后会重新从服务器下载,并再次加入缓存。对用户透明,只是首次稍慢。
