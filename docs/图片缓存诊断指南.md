# 图片缓存诊断指南

## 📋 问题描述

查询产品时,每次查询都执行数据库查询,怀疑缓存未生效。

## 🔍 诊断步骤

### 1. 查看日志输出

运行程序后,在产品查询界面执行查询操作,观察日志窗口中的以下关键信息:

#### ✅ 正常情况(缓存生效)

**首次查询:**
```

📊 批量获取图片: 总请求50, 缓存命中0, 需DB查询50, 命中率0%
🔍 开始从数据库查询50个未缓存的图片
✅ 数据库返回48条记录
📦 已将48个图片加入缓存
✅ 图片预加载完成,耗时235ms
```

**第二次查询(缓存命中):**
```
📦 开始预加载50个SKU图片到缓存...
📊 批量获取图片: 总请求50, 缓存命中50, 需DB查询0, 命中率100%
✅ 图片预加载完成,耗时5ms
```

#### ❌ 异常情况(缓存未生效)

**每次都查询数据库:**
```
📦 开始预加载50个SKU图片到缓存...
📊 批量获取图片: 总请求50, 缓存命中0, 需DB查询50, 命中率0%
🔍 开始从数据库查询50个未缓存的图片
✅ 数据库返回48条记录
📦 已将48个图片加入缓存
✅ 图片预加载完成,耗时235ms

--- (滚动或切换Tab后再次查询) ---

📦 开始预加载50个SKU图片到缓存...
📊 批量获取图片: 总请求50, 缓存命中0, 需DB查询50, 命中率0%  ← 应该100%!
🔍 开始从数据库查询50个未缓存的图片  ← 不应该再查DB!
```

### 2. 检查缓存配置

确认`ImageCacheService`的构造函数参数:

```csharp
// ImageCacheServiceBase.cs 第31-44行
protected ImageCacheServiceBase(
    IMemoryCache memoryCache,
    int slidingExpirationMinutes = 240,  // 滑动过期4小时
    int absoluteExpirationHours = 8,      // 绝对过期8小时
    CacheItemPriority priority = CacheItemPriority.High)
```

**默认配置应该是正确的**,除非被显式修改。

### 3. 检查依赖注入

确认`Startup.cs`中正确注册了单例服务:

```csharp
// 应该是AddSingleton,而不是AddTransient或AddScoped
services.AddSingleton<ImageCacheService>();
```

如果注册为`AddTransient`,每次获取都会创建新实例,缓存无法共享!

## 🎯 可能的原因

### 原因1: 依赖注入生命周期错误 ⭐⭐⭐⭐⭐ ✅ 已修复

**症状:** 每次查询都100%未命中缓存

**根本原因:** 
- ImageCacheService通过Autofac自动批量注册,默认是`InstancePerDependency`(每次创建新实例)
- 每次调用`Startup.GetFromFac<ImageCacheService>()`都会创建**新的实例**
- 每个实例有**独立的IMemoryCache**,缓存无法共享!

**修复方法:** 在`Startup.cs`中显式注册为单例(已完成):
```csharp
// ✅ 正确: 单例,所有地方共享同一个缓存
builder.RegisterType<RUINORERP.UI.Network.Services.ImageCacheService>()
    .AsSelf()
    .SingleInstance()  // ⚠️ 关键: 必须是单例,否则缓存不共享!
    .PropertiesAutowired();
```

**验证方法:**
```csharp
// 在UCProdQuery中添加测试代码
var cache1 = Startup.GetFromFac<RUINORERP.UI.Network.Services.ImageCacheService>();
var cache2 = Startup.GetFromFac<RUINORERP.UI.Network.Services.ImageCacheService>();
MainForm.Instance.uclog.AddLog($"cache1 == cache2: {ReferenceEquals(cache1, cache2)}");
```

**期望结果:** `True` (同一个实例)  
**异常结果:** `False` (不同实例,缓存不共享)

---

### 原因2: 缓存被意外清除 ⭐⭐⭐

**症状:** 第一次查询有缓存,过一会儿又没了

**可能原因:**
- 调用了`ClearAllCache()`
- 内存压力导致缓存自动淘汰
- 应用重启

**检查方法:** 查看日志中是否有"已清空所有图片缓存"的记录

---

### 原因3: FileId解析失败 ⭐⭐

**症状:** 日志显示"没有需要预加载的图片"

**可能原因:**
- `View_ProdDetail.ImagesPath`为空
- FileId格式错误

**检查方法:**
```csharp
// 在GetFileIdFromViewProdDetail中添加日志
MainForm.Instance.uclog.AddLog($"View_ProdDetail.ProdDetailID={viewProdDetail.ProdDetailID}, ImagesPath={viewProdDetail.ImagesPath}");
```

---

### 原因4: CellPainting重复触发 ⭐

**症状:** 单个图片被多次查询

**可能原因:**
- DataGridView频繁重绘
- 异步绘制完成后再次触发CellPainting

**当前代码已有保护:**
```csharp
// DrawProductImageThumbnail开头已经有Invoke保护
if (newSumDataGridView产品.InvokeRequired)
{
    newSumDataGridView产品.Invoke(...);
}
```

## 🔧 优化建议

### 1. 减少CellPainting中的数据库查询

**当前流程:**
```
PreloadVisibleImagesAsync() → 批量预加载50个图片到缓存
↓
CellPainting触发(50次) → GetImageInfoByFileIdAsync() → 检查缓存(应全部命中)
```

**如果仍有DB查询,说明:**
- 预加载失败(网络超时/异常)
- 缓存实例不共享(依赖注入问题)
- FileId不匹配(预加载和绘制使用的FileId不同)

### 2. 添加缓存统计监控

在`UCProdQuery`中添加缓存统计按钮:

```csharp
private void btnShowCacheStats_Click(object sender, EventArgs e)
{
    var imageCacheService = Startup.GetFromFac<RUINORERP.UI.Network.Services.ImageCacheService>();
    if (imageCacheService != null)
    {
        var stats = imageCacheService.GetCacheStatistics();
        MainForm.Instance.uclog.AddLog($"📊 缓存统计: {stats}");
    }
}
```

### 3. 优化预加载策略

**当前:** 只预加载可见行的图片  
**改进:** 可以预加载前后各10行的图片,提升滚动体验

```csharp
// 扩展预加载范围
int currentIndex = GetCurrentVisibleRowIndex();
int start = Math.Max(0, currentIndex - 10);
int end = Math.Min(dataGridView.Rows.Count - 1, currentIndex + visibleRowCount + 10);
```

## 📝 测试用例

### 测试1: 基本缓存功能

1. 启动程序
2. 查询产品(首次,应看到DB查询日志)
3. 再次查询(应看到100%缓存命中)
4. 等待5分钟后再次查询(应仍命中,因为滑动过期4小时)

**预期结果:** 步骤3和4都应100%命中缓存

---

### 测试2: 缓存过期

1. 查询产品(建立缓存)
2. 手动调用`ClearAllCache()`或在代码中设置短过期时间测试
3. 再次查询(应重新查询DB)

**预期结果:** 步骤3应看到0%命中率

---

### 测试3: 多用户场景

1. 打开两个客户端实例
2. 客户端A查询产品(建立A的本地缓存)
3. 客户端B查询产品(B的缓存是空的,应查询DB)

**预期结果:** 
- 客户端A: 100%命中(自己的缓存)
- 客户端B: 0%命中(需要查询DB并建立自己的缓存)

**说明:** 这是正常的,因为是**客户端本地缓存**,不是服务器端共享缓存
