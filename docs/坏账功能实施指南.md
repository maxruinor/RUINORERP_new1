## 1. 实施概述

### 1.1 实施目标

1. 建立完整的坏账管理功能，支持应收/应付账款坏账的申请、审核、撤销
2. 实现坏账撤销机制，支持"客户重新愿意支付"的业务场景
3. 与现有应收/应付模块无缝集成，确保数据一致性
4. 符合财务会计处理规范，支持审计追溯

### 1.2 实施原则

1. **架构一致性**：遵循现有系统的设计模式和代码规范
2. **最小侵入性**：不修改现有核心表结构，通过新增表实现功能
3. **事务完整性**：关键操作使用事务保证数据一致性
4. **可逆性设计**：支持坏账撤销，恢复原始状态

#### 2.6.3 数据一致性验证

```sql
-- 验证1：坏账金额汇总 = 核销记录汇总
SELECT
    SUM(BadDebtAmount - ReversedAmount) as 坏账金额,
    (SELECT SUM(WriteOffLocalAmount) FROM tb_FM_BadDebtWriteOff
     WHERE IsReversedWriteOff = 0) as 核销金额
FROM tb_FM_BadDebt
WHERE BadDebtStatus NOT IN (4, 5) -- 排除已完结和已作废

-- 验证2：ARAP余额 + 已核销金额 = 总金额
SELECT
    ARAPId, ARAPNo,
    TotalLocalPayableAmount,
    LocalBalanceAmount,
    LocalPaidAmount,
    (LocalBalanceAmount + LocalPaidAmount) as 合计
FROM tb_FM_ReceivablePayable
WHERE ARAPId = @ARAPId
```

## 4. 迁移现有数据

### 4.1 迁移方案

对于现有已标记为坏账的应收/应付单：

```sql
-- 将现有ARAPStatus=坏账的单据生成坏账申请单
INSERT INTO tb_FM_BadDebt (
    BadDebtId, BadDebtNo, ARAPId, ARAPNo, ReceivePaymentType,
    CustomerVendor_ID, Currency_ID, ExchangeRate,
    OriginalAmount, UnpaidAmount, BadDebtAmount, BadDebtReasonType,
    BadDebtReason, Employee_ID, BadDebtStatus, DataStatus,
    IsWriteOff, WriteOffAt, Created_at, Created_by
)
SELECT
    NEXT VALUE FOR SEQ_tb_FM_BadDebt,
    'BD' + CONVERT(VARCHAR, GETDATE(), 112) + RIGHT('0000' + CAST(NEXT VALUE FOR SEQ_BadDebt as VARCHAR), 4),
    ARAPId, ARAPNo, ReceivePaymentType,
    CustomerVendor_ID, Currency_ID, ExchangeRate,
    TotalLocalPayableAmount, 0, TotalLocalPayableAmount, -- 原金额作为坏账金额
    9, -- 其他原因
    '系统迁移数据', Employee_ID, 4, 4, -- 已完结状态
    1, GETDATE(), GETDATE(), 1
FROM tb_FM_ReceivablePayable
WHERE ARAPStatus = 6 -- 坏账
AND NOT EXISTS (
    SELECT 1 FROM tb_FM_BadDebt
    WHERE tb_FM_BadDebt.ARAPId = tb_FM_ReceivablePayable.ARAPId
)
```

### 4.2 迁移验证

```sql
-- 验证迁移完整性
SELECT
    COUNT(*) as 现有坏账单数,
    (SELECT COUNT(*) FROM tb_FM_BadDebt WHERE IsWriteOff = 1) as 已迁移数
FROM tb_FM_ReceivablePayable
WHERE ARAPStatus = 6
```

## 5. 注意事项

### 5.1 金额校验

1. **坏账金额不能超过未核销金额**：申请和审核时都要校验
2. **并发控制**：使用乐观锁或事务隔离级别防止超付
3. **撤销金额校验**：撤销金额不能超过已核销金额

### 5.2 状态转换

1. **严格校验当前状态**：只有符合转换规则的才能转换
2. **撤销前校验ARAP状态**：确保ARAP状态是坏账才能撤销
3. **防止重复操作**：已撤销、已完结、已作废的状态不能再次操作

### 5.3 事务处理

1. **审核必须使用事务**：核销余额、创建核销记录、更新状态必须原子
2. **撤销必须使用事务**：恢复余额、创建红字核销、更新状态必须原子
3. **记录操作日志**：关键操作要记录审计日志

### 5.4 数据一致性

1. **定期校验任务**：每周执行数据一致性校验
2. **异常告警**：发现数据不一致立即告警
3. **修复机制**：提供数据修复工具

### 5.5 用户体验

1. **撤销操作二次确认**：防止误操作
2. **状态切换提示**：告知用户下一步操作
3. **异常信息友好**：返回给用户的错误信息要清晰易懂

## 6. 风险与对策

| 风险 | 影响 | 对策 |
|-----|------|-----|
| 并发超付 | 数据不一致 | 事务隔离+乐观锁 |
| 重复撤销 | 余额重复恢复 | 状态校验+幂等设计 |
| 历史数据迁移 | 迁移不完整 | 迁移脚本验证+人工复核 |
| 权限控制不当 | 安全风险 | 按钮级权限+审批流程 |
| 性能问题 | 响应慢 | 分页查询+索引优化 |
