## 1. 概述

### 1.1 业务背景

坏账是指企业无法收回或支付的应收/应付账款。在企业经营活动中，由于客户倒闭、信用风险、汇率波动等原因，可能导致部分应收款无法收回或应付款无需支付的情况。系统需要提供完整的坏账管理功能，记录坏账产生的全过程，并确保财务数据的准确性。

**关键业务场景：** 客户因特殊原因无法付款做坏账处理后，如果后续客户又愿意支付款项，需要能够撤销坏账，继续进行应收款的收款流程。

### 1.2 功能目标

- 提供坏账申请、审核、完结的完整业务流程
- 支持应收款坏账和应付款坏账两种场景
- **支持坏账撤销**：当客户重新愿意支付时，可撤销坏账恢复应收款
- 自动核销关联的应收/应付单余额
- 生成坏账核销记录，支持财务追溯
- 与现有应收/应付模块无缝集成

#### 2.2.3 tb_FM_ReceivablePayableControllerPartial.cs（应收应付处理）

**第1049-1070行：执行坏账处理方法 `WriteOffBadDebt`**

```csharp
/// <summary>
/// 执行坏账处理
/// </summary>
public async Task<ReturnResults<tb_FM_ReceivablePayable>> WriteOffBadDebt(
    tb_FM_ReceivablePayable receivablePayable, string reason)
{
    // 1. 验证未核销金额
    if (receivablePayable.ForeignBalanceAmount == 0 && receivablePayable.LocalBalanceAmount == 0)
        throw new Exception("无未核销金额，无法进行坏账处理");

    // 2. 更新状态信息
    receivablePayable.Remark = $"[坏账处理]原因：{reason};处理时间：{DateTime.Now}";
    receivablePayable.ARAPStatus = (int)ARAPStatus.坏账;

    // 3. 核销金额清零
    receivablePayable.ForeignBalanceAmount = 0;
    receivablePayable.LocalBalanceAmount = 0;
}
```

**问题分析：** 现有方案直接将ARAPStatus标记为坏账，余额清零，**这种方式无法支持坏账撤销**。

#### 2.2.4 GlobalStateRulesManager.cs（状态转换规则）

**第349-353行：ARAPStatus状态转换规则**

```csharp
[ARAPStatus.待支付] = new List<object> { ARAPStatus.待审核, ARAPStatus.部分支付, ARAPStatus.全部支付, ARAPStatus.坏账 },
[ARAPStatus.部分支付] = new List<object> { ARAPStatus.全部支付, ARAPStatus.坏账, ARAPStatus.已冲销 },
```

**问题分析：** 现有设计中 `坏账` 是终态，无法转换回其他状态，无法支持坏账撤销的业务需求。

### 2.3 现有方案的问题

| 问题 | 说明 | 影响 |
|-----|------|-----|
| 坏账状态不可逆 | ARAPStatus.坏账是终态 | 无法支持"客户重新愿意支付"的撤销场景 |
| 余额直接清零 | 直接将BalanceAmount清零 | 无法恢复余额进行重新收款 |
| 缺乏独立单据 | 没有独立的坏账申请表 | 缺少坏账申请、审核的完整流程 |
| 无坏账撤销机制 | 无法逆向恢复应收款 | 与实际业务需求不符 |

## 3. 新方案设计要点

### 3.1 核心改进

1. **独立的坏账申请表**：新增 `tb_FM_BadDebt` 表，记录坏账申请的完整信息

2. **支持撤销的状态流转**：坏账状态不是终态，可以通过"撤销坏账"操作恢复应收款

3. **余额恢复机制**：撤销坏账时，自动恢复应收/应付单的余额

4. **核销记录追溯**：完整的核销记录，支持审计和追溯

### 3.2 与现有系统集成

| 集成点 | 现有方式 | 新方案 |
|-------|---------|--------|
| 坏账标记 | 直接设置 ARAPStatus=坏账 | 通过独立的坏账申请单 |
| 核销记录 | tb_FM_PaymentSettlement | 新增 tb_FM_BadDebtWriteOff |
| 余额处理 | 直接清零 | 审核时清零，撤销时恢复 |
| 状态流转 | ARAPStatus.坏账不可逆 | 独立状态，支持撤销 |

## 4. 业务场景说明

### 4.1 标准坏账处理流程

```
1. 客户无法付款
   ↓
2. 财务人员在应收款界面执行"坏账处理"
   ↓
3. 系统生成坏账申请单（草稿状态）
   ↓
4. 提交审核
   ↓
5. 审核通过：
   - 自动核销应收款余额
   - 更新ARAPStatus为"坏账"
   - 生成核销记录
   ↓
6. 完结归档
```

### 4.2 坏账撤销流程（关键场景）

```
1. 客户又愿意支付款项
   ↓
2. 财务人员在坏账申请单执行"撤销坏账"
   ↓
3. 系统处理：
   - 恢复应收款的余额（将核销的金额加回）
   - 更新ARAPStatus为"待支付"或原状态
   - 生成撤销核销记录（红字核销）
   - 坏账申请单状态变为"已撤销"
   ↓
4. 继续进行正常的收款流程
   ↓
5. 客户付款
   ↓
6. 正常核销应收款
```

## 5. 设计原则

### 5.1 架构一致性

- 遵循现有系统的单据审核流程模式（草稿→新建→确认→完结）
- 使用现有的状态管理系统（GlobalStateRulesManager）
- 遵循现有的命名规范和代码结构

### 5.2 财务模块特殊性

- 坏账金额必须 ≤ 关联应收/应付单的未核销金额
- 所有坏账操作必须生成核销记录，支持财务追溯
- 撤销坏账时必须恢复原始余额，确保数据一致性
- 支持审计日志，记录所有操作

### 5.3 可撤销设计

- **关键改进**：坏账状态不再是终态
- 撤销操作需要审批流程（防止随意撤销）
- 撤销时自动恢复应收/应付单余额
- 生成红字核销记录，与原始核销对应
