# 基础数据缓存按需加载优化方案

## 1. 背景与目标

当前系统的基础数据缓存（如产品、客户、供应商等）在初始化时默认加载实体的**所有字段**（`SELECT *`）。对于字段较多或包含大文本/二进制数据的表，这会导致：
- **网络带宽浪费**：传输大量 UI 不需要的冗余字段。
- **内存占用高**：缓存中存储了大量“冷数据”。
- **CPU 开销大**：JSON 序列化/反序列化处理全量对象效率较低。

**优化目标**：实现“按需缓存”，仅查询和缓存业务真正需要的关键列（如 `ID`, `Code`, `Name`, `Spec` 等）。

---

## 2. 核心设计方案

### 2.1 注册方法重载设计 (`EntityCacheInitializationService.cs`)

为了支持更灵活的字段选择，我们在 `EntityCacheInitializationService.cs` 中对 `RegistInformation` 方法进行重构。

**设计思路：**
1.  **保留原方法**：确保向后兼容，默认 `cacheWholeRow = true`。
2.  **新增重载方法**：允许通过 `params` 传入多个显示字段表达式，并显式控制是否只缓存指定字段。

```csharp
/// <summary>
/// 注册表结构信息（增强版：支持多字段投影配置）
/// </summary>
/// <param name="primaryKeyExpression">主键字段</param>
/// <param name="isView">是否为视图</param>
/// <param name="isCacheable">是否参与缓存</param>
/// <param name="description">描述信息</param>
/// <param name="tableType">表类型</param>
/// <param name="cacheWholeRow">是否缓存整行（false 则仅缓存下方指定的字段）</param>
/// <param name="displayFieldExpressions">需要缓存的字段列表（ID, Name, Code 等）</param>
private void RegistInformation<T>(
    Expression<Func<T, object>> primaryKeyExpression,
    bool isView = false,
    bool isCacheable = true,
    string description = null,
    TableType tableType = TableType.Other,
    bool cacheWholeRow = true,
    params Expression<Func<T, object>>[] displayFieldExpressions) where T : class
{
    if (displayFieldExpressions == null || displayFieldExpressions.Length == 0)
    {
        throw new ArgumentException("至少需要指定一个显示字段（通常是 Name 或 Code）");
    }

    // 第一个参数作为主显示字段，其余作为辅助显示字段
    var mainDisplay = displayFieldExpressions[0];
    var otherDisplays = displayFieldExpressions.Skip(1).ToArray();

    _tableSchemaManager.RegisterTableSchema(
        primaryKeyExpression,
        mainDisplay,
        isView,
        isCacheable,
        description,
        cacheWholeRow, // 关键：控制是否开启按需缓存
        otherDisplays);

    // 设置表类型
    string tableName = typeof(T).Name;
    var schemaInfo = _tableSchemaManager.GetSchemaInfo(tableName);
    if (schemaInfo != null)
    {
        schemaInfo.Type = tableType;
    }
}
```

### 2.2 实体定义优化示例
根据 `RUINORERP.Model` 中的实体定义，我们对高频使用的基础表进行精细化配置：

| 表名 (Entity) | 建议缓存字段 (Expressions) | 理由 |
| :--- | :--- | :--- |
| **tb_Prod** | `k=>k.ProdBaseID`, `v=>v.CNName`, `v=>v.ProdCode`, `v=>v.Specification` | 产品表字段极多，UI 通常只需展示品号、名称和规格。 |
| **tb_CustomerVendor** | `k=>k.CustomerVendor_ID`, `v=>v.CVName`, `v=>v.CVCode` | 客商表包含大量地址、税务信息，缓存仅需识别身份即可。 |
| **tb_Employee** | `k=>k.Employee_ID`, `v=>v.Employee_Name`, `v=>v.Employee_Code` | 员工表可能包含身份证号、电话等敏感或冗余信息。 |
| **tb_Department** | `k=>k.DepartmentID`, `v=>v.DepartmentName` | 部门表结构简单，可保持全量缓存或仅缓存名称。 |

## 3. 实施步骤

### 第一步：修改 `EntityCacheInitializationService.cs`
在文件末尾或 `RegistInformation` 附近添加上述重载方法。并将现有的注册调用逐步迁移到新格式。

**迁移示例：**
```csharp
// 旧写法（全量缓存）
RegistInformation<tb_Prod>(k => k.ProdBaseID, v => v.CNName, tableType: TableType.Business);

// 新写法（按需缓存：仅缓存 ID, 中文名, 品号, 规格）
RegistInformation<tb_Prod>(
    k => k.ProdBaseID, 
    tableType: TableType.Business,
    cacheWholeRow: false, // 开启精简模式
    v => v.CNName, 
    v => v.ProdCode, 
    v => v.Specification
);
```

### 第二步：增强 `SqlSugarCacheDataProvider`
修改数据提供层，使其能识别 `TableSchemaInfo` 中的 `CacheWholeRow` 标志。

```csharp
// 在 GetEntityListFromSource 中增加逻辑
public List<T> GetEntityListFromSource<T>(string tableName) where T : class
{
    var schema = _tableSchemaManager.GetSchemaInfo(tableName);
    var db = _unitOfWorkManage.GetDbClient();
    
    // 如果配置了不按行缓存且有指定字段，则执行投影查询
    if (schema != null && !schema.CacheWholeRow && schema.DisplayFields.Any())
    {
        // 构造 Select 字符串，例如: "ProdBaseID, CNName, ProdCode"
        string selectCols = string.Join(",", schema.DisplayFields);
        return db.Queryable<T>().Select(selectCols).ToList();
    }

    // 默认全量查询
    return db.Queryable<T>().ToList();
}
```

### 第三步：缓存存储适配
由于投影查询返回的对象可能缺少某些属性，`EntityCacheManager` 在存储时应注意：
- 如果 `CacheWholeRow = false`，缓存中存储的是“部分字段对象”。
- UI 层在绑定这些缓存数据时，应确保只访问已缓存的字段，或通过 `GetEntity` 单独补全缺失字段。

---

## 4. 预期收益评估

1.  **SQL 性能提升**：对于 `tb_Prod` 这种拥有 50+ 字段的表，`SELECT` 子句缩小后，数据库扫描和返回的数据量可减少 **80%-90%**。
2.  **网络传输优化**：客户端启动时拉取的基础数据包体积将显著下降，加快登录和初始化速度。
3.  **内存利用率提高**：同样的内存空间可以缓存更多种类的基础表，减少因内存压力触发的 LRU 清理频率。

## 5. 结论
通过在 `RegistInformation` 中引入 `cacheWholeRow` 参数和多字段表达式支持，我们可以在不破坏现有架构的前提下，实现精细化的缓存控制。建议优先在**产品库**、**客商库**等大宽表上试点此方案。
