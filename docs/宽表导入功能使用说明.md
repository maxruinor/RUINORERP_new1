# 宽表拆分与智能外键映射导入功能使用说明

## 一、功能概述

本功能支持从单个Excel文件自动拆分并导入多个关联数据库表,实现"一张宽表搞定所有数据"的便捷导入体验。

### 核心特性

1. **单文件多表导入**: Excel中包含产品、供应商、类目等多个表的数据,系统自动拆分
2. **智能外键映射**: Excel中填写"供应商名称",系统自动查找或创建供应商并获取ID
3. **依赖顺序处理**: 先导入字典表(供应商/类目),再导入主表(产品),最后导入子表(SKU)
4. **自动去重**: 根据配置的业务键自动判断新增或更新

---

## 二、配置文件说明

### 2.1 配置文件位置

`RUINORERP.UI/SysConfig/DataMigration/Profiles/04_产品宽表导入.json`

### 2.2 配置结构

```json
{
  "ProfileName": "方案名称",
  "Description": "方案描述",
  
  // 主表配置
  "MasterTable": {
    "TargetTable": "tb_Prod",
    "BusinessKeys": ["ProdCode"],
    "ColumnMappings": [...]
  },
  
  // 依赖表配置(字典表,优先处理)
  "DependencyTables": [
    {
      "TargetTable": "tb_CustomerVendor",
      "BusinessKeys": ["VendorName"],
      "ColumnMappings": [...]
    }
  ],
  
  // 子表配置(一对多关系)
  "ChildTables": [
    {
      "TargetTable": "tb_ProdDetail",
      "ParentTableRefField": "ProdId",
      "ColumnMappings": [...]
    }
  ]
}
```

### 2.3 关键字段说明

#### TransformRule (转换规则)

格式: `REF:表名:业务键字段`

示例:
```json
{
  "ExcelHeader": "供应商名称",
  "DbColumn": "SupplierId",
  "TransformRule": "REF:tb_CustomerVendor:VendorName"
}
```

含义: Excel中的"供应商名称"列需要通过查询`tb_CustomerVendor`表的`VendorName`字段获取对应的ID,然后写入`tb_Prod.SupplierId`。

#### ForeignConfig.AutoCreateIfNotExists

- `true`: 如果数据库中不存在该供应商,自动创建新记录
- `false`: 如果数据库中不存在,抛出错误

## 四、使用流程

### 4.1 准备阶段

1. **创建JSON Profile配置文件**
   - 复制`04_产品宽表导入.json`作为模板
   - 根据实际需求修改表名、字段映射

2. **准备Excel文件**
   - 确保第一行为表头
   - 表头名称需与配置文件中的`ExcelHeader`一致

### 4.2 执行导入

#### 策略一: 宽表一键导入 (推荐新手)

```csharp
var engine = new SmartImportEngine(db);
var report = await engine.ExecuteWideTableImportAsync(filePath, "04_产品宽表导入");

if (report.IsSuccess)
{
    Console.WriteLine($"成功导入 {report.SuccessRows} 条记录");
}
else
{
    Console.WriteLine($"导入失败: {report.Message}");
}
```

**适用场景**:
- ✅ 数据量不大(<5000行)
- ✅ Excel结构清晰
- ✅ 希望一步到位

---

#### 策略二: 分步手动导入 (推荐高级用户)

**步骤1: 先导入基础表(供应商/类目)**

```csharp
// 仅导入依赖表
var report1 = await engine.ImportDependencyTablesOnlyAsync(filePath, "04_产品宽表导入");
Console.WriteLine($"基础表导入: {report1.Message}");
```

**步骤2: 验证基础数据**

```csharp
// 可以在数据库中查询确认
var suppliers = db.Queryable<tb_CustomerVendor>()
    .Where(s => s.VendorName == "华为")
    .ToList();
Console.WriteLine($"找到 {suppliers.Count} 个供应商");
```

**步骤3: 再导入主表(产品)**

```csharp
// 仅导入主表,系统会自动从数据库预加载基础数据
var report2 = await engine.ImportMasterTableOnlyAsync(filePath, "04_产品宽表导入");
Console.WriteLine($"主表导入: {report2.Message}");
```

**步骤4: 最后导入子表(SKU明细)**

```csharp
// 可以单独导入子表(需要额外实现)
// var report3 = await engine.ImportChildTablesOnlyAsync(...);
```

**适用场景**:
- ✅ 数据量大(>10000行)
- ✅ 需要逐步验证
- ✅ 基础数据已存在,只需导入主表
- ✅ 调试和排查问题

## 五、工作原理

### 5.1 三阶段处理流程

```
┌─────────────────────────────────────────┐
│ 阶段1: 预加载依赖表                      │
│ - 查询tb_CustomerVendor已有供应商        │
│ - 查询tb_ProdCategories已有类目          │
│ - 建立"名称→ID"缓存                      │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│ 阶段2: 拆分宽表                          │
│ - 提取供应商数据 → 写入tb_CustomerVendor │
│ - 提取类目数据 → 写入tb_ProdCategories   │
│ - 提取产品数据 → 注入外键ID              │
│ - 提取SKU数据 → 注入产品ID               │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│ 阶段3: 按依赖顺序写入                    │
│ 1. 依赖表(供应商/类目)                   │
│ 2. 主表(产品)                            │
│ 3. 子表(SKU明细)                         │
└─────────────────────────────────────────┘
```

### 5.2 外键ID注入示例

假设Excel中某行数据:
- 产品编码: P001
- 供应商名称: 华为

处理流程:
1. 检查缓存: `tb_CustomerVendor["华为"]` 是否存在?
2. 如果存在: 直接返回ID(如123)
3. 如果不存在: 
   - 查询数据库: `SELECT ID FROM tb_CustomerVendor WHERE VendorName='华为'`
   - 如果查到: 缓存并返回ID
   - 如果查不到且`AutoCreateIfNotExists=true`: 
     - 插入新记录: `INSERT INTO tb_CustomerVendor (VendorName) VALUES ('华为')`
     - 返回新生成的ID(如456)
4. 将产品记录的`SupplierId`设置为返回的ID

## 六、常见问题

### Q1: 如何处理一对多关系?

**答**: 在`ChildTables`中配置子表,系统会自动为每条SKU记录注入父表(产品)的ID。

### Q2: 供应商已存在时会重复创建吗?

**答**: 不会。系统会先查询数据库,只有不存在时才会创建(且需`AutoCreateIfNotExists=true`)。

### Q3: 如何调试导入过程?

**答**: 
1. 检查JSON配置文件格式是否正确
2. 确认Excel表头与`ExcelHeader`完全匹配
3. 查看`ImportReport.Message`获取详细错误信息

### Q4: 性能如何?

**答**: 
- 1000行数据约需5-10秒(含网络延迟)
- 瓶颈主要在数据库写入,可通过批量插入优化
- 预加载机制避免了N+1查询问题

## 七、注意事项

1. **事务一致性**: 任一环节失败会全部回滚,确保数据完整性
2. **唯一约束**: 建议在数据库层面为业务键添加唯一索引,防止并发冲突
3. **AI映射**: 当前版本未启用AI自动映射,需手动配置JSON
4. **扩展性**: 可根据需要添加更多依赖表或子表

## 八、单表导入兼容性

### 8.1 宽表与单表的关系

**重要**: 宽表导入架构完全兼容单表导入场景。

- **宽表导入**: MasterTable + DependencyTables + ChildTables
- **单表导入**: 仅配置 MasterTable,DependencyTables 和 ChildTables 为空数组

### 8.2 单表导入示例

配置文件: `05_供应商单表导入.json`

```json
{
  "MasterTable": {
    "TargetTable": "tb_CustomerVendor",
    "BusinessKeys": ["VendorName"],
    "ColumnMappings": [...]
  },
  "DependencyTables": [],  // 空数组
  "ChildTables": []        // 空数组
}
```

调用方式:
```csharp
// 方式1: 使用宽表导入方法(推荐,统一接口)
var report = await engine.ExecuteWideTableImportAsync(filePath, "05_供应商单表导入");

// 方式2: 使用原有单表导入方法(向后兼容)
var profile = LoadProfile("05_供应商单表导入");
var report = await engine.ExecuteWithProfileAsync(filePath, profile);
```

### 8.3 优势

1. **统一架构**: 无需维护两套代码
2. **平滑迁移**: 现有单表导入可逐步升级为宽表导入
3. **灵活扩展**: 单表可随时添加依赖表变为宽表
