## 三、关键设计亮点

### 3.1 统一架构设计

**核心理念**: 宽表是单表的超集,一套代码支持两种场景

```csharp
// 宽表配置
{
  "MasterTable": {...},
  "DependencyTables": [...],  // 可以有0个或多个
  "ChildTables": [...]        // 可以有0个或多个
}

// 单表配置 (宽表的特例)
{
  "MasterTable": {...},
  "DependencyTables": [],     // 空数组
  "ChildTables": []           // 空数组
}
```

### 3.2 混合来源解析

**问题**: Excel中填写"华为",数据库需要ID

**解决方案**:
```
1. 查缓存 → 命中则返回
2. 查数据库 → 找到则缓存并返回
3. 自动创建 → 插入新记录并返回新ID
```

### 3.3 依赖驱动执行

```
执行顺序:
1. DependencyTables (字典表) → 先建立基础数据
2. MasterTable (主表) → 引用字典表ID
3. ChildTables (子表) → 引用主表ID
```

## 四、技术实现细节

### 4.1 IdRemappingEngine扩展

```csharp
// 预加载依赖表到缓存
public async Task PreloadDependencyTablesAsync(List<ImportProfile> profiles)
{
    foreach (var profile in profiles)
    {
        await LoadExistingLogicalKeysByProfileAsync(profile);
    }
}

// 获取或创建外键ID (核心方法)
public async Task<long> GetOrCreateForeignKeyIdAsync(
    string tableName,
    string businessKeyName,
    string businessKeyValue,
    DataRow sourceRow,
    ImportProfile tableProfile)
{
    // 1. 查缓存
    if (_globalLogicalKeyCache.ContainsKey(tableName) &&
        _globalLogicalKeyCache[tableName].TryGetValue(businessKeyValue, out long cachedId))
    {
        return cachedId;
    }

    // 2. 查数据库
    var sql = $"SELECT TOP 1 [ID] FROM [{tableName}] WHERE [{businessKeyName}] = @val";
    var result = await _db.Ado.GetScalarAsync(sql, new { val = businessKeyValue });
    
    if (result != null && long.TryParse(result.ToString(), out long dbId))
    {
        CacheLogicalKey(tableName, businessKeyValue, dbId);
        return dbId;
    }

    // 3. 自动创建
    if (tableProfile != null && ShouldAutoCreateForProfile(tableProfile))
    {
        return await CreateNewRecordAndReturnIdAsync(...);
    }

    throw new Exception($"外键值不存在且不允许自动创建");
}
```

### 4.2 DataSplitterService核心逻辑

```csharp
public async Task<Dictionary<string, DataTable>> SplitWideTableWithDependenciesAsync(
    DataTable sourceData,
    WideTableImportProfile wideProfile)
{
    var result = new Dictionary<string, DataTable>();

    // 阶段1: 预加载依赖表
    await _remapper.PreloadDependencyTablesAsync(wideProfile.DependencyTables);

    // 阶段2: 处理依赖表
    foreach (var depProfile in wideProfile.DependencyTables)
    {
        var depData = ExtractTableData(sourceData, depProfile);
        await _dbWriter.BatchUpsertAsync(depData, depProfile, _remapper);
        result[depProfile.TargetTable] = depData;
    }

    // 阶段3: 处理主表(注入外键ID)
    var masterData = ExtractTableData(sourceData, wideProfile.MasterTable);
    await InjectForeignKeysIntoMasterTableAsync(masterData, wideProfile.MasterTable, ...);
    result[wideProfile.MasterTable.TargetTable] = masterData;

    // 阶段4: 处理子表
    foreach (var childProfile in wideProfile.ChildTables)
    {
        var childData = ExtractChildTableData(sourceData, childProfile, ...);
        InjectParentForeignKey(childData, childProfile, ...);
        result[childProfile.TargetTable] = childData;
    }

    return result;
}
```

## 七、性能考虑

### 7.1 优化措施

1. **批量预加载**: 一次性查询所有依赖表数据到内存缓存
2. **避免N+1查询**: 通过缓存减少数据库访问次数
3. **事务控制**: 每个表独立事务,失败时回滚

### 7.2 预期性能

- **100行数据**: ~1-2秒
- **1000行数据**: ~5-10秒
- **瓶颈**: 主要在数据库写入和网络延迟

### 7.3 后续优化方向

1. 使用Bulk Insert提升大数据量写入速度
2. 并行处理多个依赖表
3. 异步流式处理超大文件
