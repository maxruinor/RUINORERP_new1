#### **手动触发页**

#### **帮助说明页**
- ✅ 详细的功能使用说明
- ✅ 最佳实践建议

### 2. **性能优化**

#### **工作流步骤增强**
- ⏱️ 添加执行耗时统计（Stopwatch）
- 📝 完善的日志记录（ILogger + SafeLogInfo）
- ❌ 正确的错误处理（失败时终止工作流）
- ✅ 异步方法正确await

- 仅对有变化的库存生成快照

- 减少数据库写入压力

### 3. **时间配置灵活性**

#### **配置文件方式**（生产环境推荐）

#### **自定义时间**（UI界面）
- 点击"自定义时间"按钮

## 🚀 使用指南

### **快速开始**

1. **启动服务器**
   ```
   运行 RUINORERP.Server.exe
   ```

2. **打开库存快照管理**
   - 点击左侧导航栏的"库存快照管理"按钮
   - 或在主菜单中选择相应选项

3. **查看当前配置**
   - 切换到"配置管理"标签页
   - 查看任务状态、执行时间、下次执行时间

4. **修改执行时间**
   - **方式1**：点击快捷按钮（凌晨1-4点）
   - **方式2**：点击"自定义时间"按钮
   - **方式3**：直接编辑 `ScheduledTasks.json` 文件

5. **测试功能**
   - 切换到"手动触发"标签页
   - 点击"立即执行库存快照"按钮
   - 查看日志输出确认执行结果

### **推荐配置**

#### **生产环境**
```
执行时间：凌晨 02:00:00
调试模式：关闭
保留月数：12个月
```

**理由**：
- 凌晨2点是业务低峰期
- 避免与文件清理（03:00）、临时图片清理（04:00）冲突
- 12个月保留期平衡存储成本和历史数据需求

#### **测试环境**
```
执行时间：任意
调试模式：开启
执行间隔：5分钟
```

**理由**：
- 快速验证功能
- 实时观察执行情况

## 📊 性能指标

### **执行耗时参考**

| 场景 | 预计耗时 | 说明 |
|------|---------|------|
| 首次执行（全量） | 5-30秒 | 取决于库存数据量 |
| 日常执行（增量） | 1-5秒 | 仅有变化的库存 |
| 清理过期数据 | <1秒 | 删除少量记录 |

### **资源占用**

- **CPU**：执行期间短暂升高（<10%）
- **内存**：无明显增长
- **磁盘IO**：中等（写入快照数据）
- **网络**：无影响（服务端本地执行）

## 🔧 高级配置

### **修改保留月数**

编辑 `InventorySnapshotWorkflow.cs`：

```csharp
// CleanupOldSnapshotsStep.RunAsync 方法中
int keepMonths = 12;  // 修改为你需要的月数
var count = _snapshotService.CleanupExpiredSnapshots(keepMonths);
```

### **调整工作时间判断**

编辑 `appsettings.json`：

```json
{
  "InventoryMonitoring": {
    "WorkingHoursStart": 9,   // 工作开始时间
    "WorkingHoursEnd": 18     // 工作结束时间
  }
}
```

### **启用/禁用任务**

**方式1：UI界面**
- 点击"启用/禁用"按钮

**方式2：配置文件**
```json
{
  "tasks": [
    {
      "id": "InventorySnapshot",
      "enabled": false  // 改为false禁用
    }
  ]
}
```

## 🐛 故障排查

### **问题1：工作流不执行**

**检查项**：
1. 任务是否启用？（`enabled: true`）
2. 调试模式是否关闭？（生产环境应关闭）
3. 服务是否重启？（修改配置后需重启）

**解决方法**：
- 在UI中查看任务状态
- 检查 `ScheduledTasks.json` 文件
- 重启服务器

### **问题2：执行失败**

**检查项**：
1. 查看日志输出（richTextBoxLog）
2. 检查数据库连接
3. 确认库存表有数据

**解决方法**：
- 查看详细错误信息
- 检查数据库是否正常
- 联系技术支持

### **问题3：执行时间不准确**

**原因**：
- 首次执行有延迟计算
- 系统时间与预期不符

**解决方法**：
- 查看"下次执行时间"显示
- 确认服务器系统时间正确
- 重启服务重新调度

## 💡 最佳实践

### **1. 执行时间选择**

✅ **推荐**：凌晨1-4点
- 业务低峰期
- 客户端未使用
- 不影响正常业务

❌ **避免**：
- 工作时间（9:00-18:00）
- 业务高峰期
- 与其他定时任务冲突

### **2. 调试模式使用**

✅ **测试环境**：
- 开启调试模式
- 设置较短间隔（5分钟）
- 快速验证功能

❌ **生产环境**：
- 必须关闭调试模式
- 使用配置文件控制时间
- 避免频繁执行

### **3. 监控和维护**

- 定期检查日志，确认执行成功
- 监控数据库快照表大小
- 根据业务需求调整保留月数
- 首次执行后验证数据准确性
