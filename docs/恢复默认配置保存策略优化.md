# 恢复默认配置保存策略优化

## 🎯 问题识别

你的观察非常准确！**`frmColumnsSets.btnRestoreDefaultConfig_Click` 不应该直接保存和应用配置**，而应该由 `CustomizeGrid.SetColumns` 统一处理。

---

## ❌ 之前的设计问题

### 1. 职责混乱

```csharp
// frmColumnsSets.btnRestoreDefaultConfig_Click
private async void btnRestoreDefaultConfig_Click(object sender, EventArgs e)
{
    // ... 更新 UI ...
    
    // ❌ 问题1: 直接操作 DataGridView（越权）
    DataGridViewSetTarget.ColumnDisplays.Clear();
    DataGridViewSetTarget.BindColumnStyle();
    
    // ❌ 问题2: 直接保存到数据库（越权）
    await ColumnConfigManager.Instance.ResetToDefaultAsync(...);
}
```

**违反原则**：
- ❌ **单一职责原则**：`frmColumnsSets` 应该只负责 UI 交互
- ❌ **依赖倒置原则**：不应该直接依赖 `ColumnConfigManager`
- ❌ **开闭原则**：新增功能需要修改多处代码

### 2. 保存策略不合理

| 操作类型 | 之前 | 应该 |
|---------|------|------|
| **手动修改列** | 延时批量保存 ✅ | 延时批量保存 ✅ |
| **恢复默认值** | 延时批量保存 ❌ | **立即保存** ✅ |

**为什么恢复默认要立即保存？**
- 用户明确表达了"重置"意图
- 不应该与其他修改合并
- 避免用户误操作后无法撤销

### 代码实现

#### 1. frmColumnsSets 简化为纯 UI

```csharp
// frmColumnsSets.cs
private void btnRestoreDefaultConfig_Click(object sender, EventArgs e)
{
    try
    {
        // ✅ 只负责更新 UI
        listView1.Items.Clear();
        foreach (ColDisplayController keyValue in InitColumnDisplays)
        {
            if (!keyValue.Disable)
            {
                ListViewItem lvi = new ListViewItem();
                lvi.Checked = keyValue.Visible;
                lvi.Name = keyValue.ColName;
                lvi.Tag = keyValue;
                lvi.Text = keyValue.ColDisplayText;
                lvi.ImageKey = keyValue.ColDisplayIndex.ToString();
                listView1.Items.Add(lvi);
            }
        }

        // 同步更新 ColumnDisplays 集合
        ColumnDisplays.Clear();
        ColumnDisplays.AddRange(InitColumnDisplays);
        
        // ✅ 不再直接保存和应用，由 CustomizeGrid.SetColumns 统一处理
        // 只需要设置 DialogResult = OK，让调用方处理
    }
    catch (Exception ex)
    {
        MessageBox.Show($"恢复默认列配置失败：{ex.Message}", "错误", 
            MessageBoxButtons.OK, MessageBoxIcon.Error);
    }
}
```

**删除了**：
- ❌ `SaveDefaultConfigToDatabase()` 方法
- ❌ 直接操作 `DataGridViewSetTarget`
- ❌ 直接调用 `ColumnConfigManager`

#### 2. CustomizeGrid 统一处理

```csharp
// CustomizeGrid.cs
public async void SetColumns(List<ColDisplayController> InitColumnDisplays)
{
    ForCustomizeGrid.frmColumnsSets set = new ForCustomizeGrid.frmColumnsSets();
    // ... 初始化 ...
    
    if (set.ShowDialog() == DialogResult.OK)
    {
        // ✅ 从 frmColumnsSets 获取修改后的配置
        ColumnDisplays.Clear();
        ColumnDisplays.AddRange(set.ColumnDisplays);
        
        // 同步到 DataGridView
        if (targetDataGridView != null)
        {
            targetDataGridView.ColumnDisplays.Clear();
            targetDataGridView.ColumnDisplays.AddRange(ColumnDisplays);
        }
        
        // 应用配置到 UI
        targetDataGridView.BindColumnStyle();

        if (NeedSaveColumnsXml)
        {
            // ✅ 判断是否是恢复默认操作
            bool isRestoredDefault = ColumnDisplays.Count == InitColumnDisplays.Count &&
                ColumnDisplays.All(c => InitColumnDisplays.Any(ic => 
                    ic.ColName == c.ColName && 
                    ic.Visible == c.Visible && 
                    ic.ColDisplayIndex == c.ColDisplayIndex));
            
            if (isRestoredDefault)
            {
                // ✅ 恢复默认：立即保存到数据库（不延时）
                if (!string.IsNullOrEmpty(this.GridKeyName) && this.MenuId.HasValue)
                {
                    await ColumnConfigManager.Instance.ResetToDefaultAsync(
                        this.GridKeyName,
                        this.MenuId.Value,
                        ColumnDisplays
                    );
                }
            }
            else
            {
                // ✅ 手动修改：延时批量保存
                SaveColumnsList(ColumnDisplays);
            }
        }
    }
}
```

## 🎯 关键技术点

### 1. 如何判断是否是"恢复默认"

```csharp
bool isRestoredDefault = ColumnDisplays.Count == InitColumnDisplays.Count &&
    ColumnDisplays.All(c => InitColumnDisplays.Any(ic => 
        ic.ColName == c.ColName && 
        ic.Visible == c.Visible && 
        ic.ColDisplayIndex == c.ColDisplayIndex));
```

**判断逻辑**：
- 列数量相同
- 每个列的名称、可见性、显示索引都相同

**优点**：
- 不需要额外的事件或标志位
- 自动检测，无需用户显式声明

**缺点**：
- 如果用户手动调整为与默认相同，也会被视为"恢复默认"
- 但这种情况很少见，且行为合理（立即保存也无妨）

### 2. 为什么恢复默认要立即保存

| 场景 | 延时保存的问题 | 立即保存的优势 |
|------|--------------|--------------|
| **用户误操作** | 可能与其他修改合并，难以撤销 | 独立保存，易于追溯 |
| **程序崩溃** | 配置丢失 | 已保存，不会丢失 |
| **用户体验** | 不确定是否生效 | 立即生效，反馈明确 |
| **数据一致性** | 可能与后续修改冲突 | 作为明确的检查点 |
