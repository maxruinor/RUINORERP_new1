# 核心业务类代码优化方案

## 📋 概述

本文档针对四个核心业务类进行性能和代码质量优化:
- `tb_PurOrderControllerPartial.cs` - 采购订单
- `tb_PurEntryControllerPartial.cs` - 采购入库
- `tb_SaleOrderControllerPartial.cs` - 销售订单(已部分优化)
- `tb_SaleOutControllerPartial.cs` - 销售出库

**优化目标**:
1. ✅ 事务范围最小化
2. ✅ 验证判断前置化
3. ✅ 消除重复冗余代码
4. ✅ 保持业务正确性
5. ✅ 避免过度设计
6. 保持业务逻辑不变
---

## 🔍 问题分析

### 问题1: 事务范围过大 ⚠️ P0

#### 现状分析

**采购入库审核** (`tb_PurEntryControllerPartial.cs` ApprovalAsync):
```csharp
// 第365行: 过早开启事务
_unitOfWorkManage.BeginTran();

// 第367-383行: 更新订单状态(应该在事务内)
await _unitOfWorkManage.GetDbClient().Updateable(entity.tb_purorder)...

// 第390-489行: 库存计算和更新(应该在事务内)
foreach (var group in inventoryGroups) { ... }

// 第567-589行: ❌ 财务处理在事务内,持有时间过长
if (authorizeController.EnableFinancialModule())
{
    var ctrpayable = _appContext.GetRequiredService<...>();
    tb_FM_ReceivablePayable Payable = await ctrpayable.BuildReceivablePayable(...);
    ReturnMainSubResults<...> rmr = await ctrpayable.BaseSaveOrUpdateWithChild(...);
}

// 第593行: 提交事务
_unitOfWorkManage.CommitTran();
```

**问题影响**:
- 财务模块处理可能涉及复杂查询和多次数据库操作
- 长时间持有数据库锁,增加死锁风险
- 并发性能下降

**对比优秀实践** (`tb_SaleOrderControllerPartial.cs`):
```csharp
// 第260行: 开启主事务
_unitOfWorkManage.BeginTran();

// 第263-286行: 只处理核心业务(库存+订单状态)
await dbHelper.BaseDefaultAddElseUpdateAsync(invList);
await _unitOfWorkManage.GetDbClient().Updateable(entity)...

// 第287行: 快速提交主事务
_unitOfWorkManage.CommitTran();

// 第291-301行: ✅ 财务独立事务处理
if (needProcessFinance)
{
    var financeResult = await ProcessFinanceOrderApprovalAsync(...);
    // 即使失败也不回滚主事务
}
```

#### 优化方案

**方案A: 财务独立事务模式**(推荐,参考销售订单实现)

优点:
- ✅ 主事务持有时间短(<100ms)
- ✅ 财务失败不影响核心业务
- ✅ 支持补偿机制

缺点:
- 需要实现补偿逻辑
- 数据最终一致性(非强一致)

**方案B: 事务拆分模式**

将大事务拆分为多个小事务:
1. 事务1: 更新订单状态
2. 事务2: 更新库存
3. 事务3: 创建财务单据

优点:
- ✅ 每个事务职责单一
- ✅ 锁粒度更小

缺点:
- ❌ 需要分布式事务保证一致性
- ❌ 实现复杂度高

**推荐**: 采用**方案A**,与销售订单保持一致。

**采购入库审核验证分散**:
```csharp

    // 第121-130行: 验证4 - 订单状态校验

    // 第137-146行: 验证5 - 产品归属校验
    foreach (var child in entity.tb_PurEntryDetails) { ... }
    
    // 第158-229行: 验证6 - 数量超限校验(复杂嵌套循环)
    for (int i = 0; i < entity.tb_purorder.tb_PurOrderDetails.Count; i++) { ... }
    
    // 第365行: ❌ 此时才开启事务,但前面已有6处验证

```

**问题**:
1. 验证和业务逻辑混杂
2. 部分验证在事务内执行(浪费资源)
3. 验证顺序不合理(应先验证简单规则,再验证复杂规则)

#### 优化方案

**验证前置化原则**:
```
1. 基础数据验证(非DB) → 2. 业务规则验证(DB查询) → 3. 开启事务 → 4. 执行业务逻辑
```

**重构后的结构**:
```csharp
public async override Task<ReturnResults<T>> ApprovalAsync(T ObjectEntity)
{
    // ========== 第一阶段: 预处理验证(无事务) ==========
    
    // 1.1 基础验证
    var validationResult = ValidateBasicRules(entity);
    if (!validationResult.Succeeded) return validationResult;
    
    // 1.2 加载依赖数据
    await LoadRelatedDataAsync(entity);
    
    // 1.3 业务规则验证
    var businessValidation = await ValidateBusinessRulesAsync(entity);
    if (!businessValidation.Succeeded) return businessValidation;
    
    // 1.4 预加载库存和价格记录(性能优化)
    var preloadedData = await PreloadInventoryAndPricesAsync(entity);
    
    // ========== 第二阶段: 事务内执行 ==========
    _unitOfWorkManage.BeginTran();
    try
    {
        // 2.1 更新订单状态
        await UpdateOrderStatusAsync(entity);
        
        // 2.2 更新库存
        await UpdateInventoryAsync(entity, preloadedData);
        
        // 2.3 更新其他业务数据
        await UpdateRelatedDataAsync(entity);
        
        // 2.4 更新单据状态
        await UpdateDocumentStatusAsync(entity);
        
        _unitOfWorkManage.CommitTran();
        
        // ========== 第三阶段: 后置处理(独立事务) ==========
        if (needProcessFinance)
        {
            await ProcessFinanceAsync(entity);
        }
        
        return ReturnResults<T>.Success(entity);
    }
    catch (Exception ex)
    {
        _unitOfWorkManage.RollbackTran();
        return ReturnResults<T>.Fail(ex.Message);
    }
}
```

#### 优化方案

**提取公共服务类**:

创建 `InventoryAggregationService.cs`:
```csharp
namespace RUINORERP.Business.CommService
{
    /// <summary>
    /// 库存聚合服务 - 消除重复的库存分组和预加载逻辑
    /// </summary>
    public class InventoryAggregationService
    {
        private readonly IUnitOfWorkManage _unitOfWorkManage;
        
        public InventoryAggregationService(IUnitOfWorkManage unitOfWorkManage)
        {
            _unitOfWorkManage = unitOfWorkManage;
        }
        
        /// <summary>
        /// 预加载库存数据(死锁优化)
        /// </summary>
        public async Task<Dictionary<(long ProdDetailID, long LocationID), tb_Inventory>> 
            PreloadInventoryAsync(IEnumerable<long> prodDetailIds, 
                                  IEnumerable<(long ProdDetailID, long LocationID)> requiredPairs)
        {
            var inventoryList = await _unitOfWorkManage.GetDbClient()
                .Queryable<tb_Inventory>()
                .Where(i => prodDetailIds.Contains(i.ProdDetailID))
                .ToListAsync();
            
            var pairSet = requiredPairs.ToHashSet();
            inventoryList = inventoryList
                .Where(i => pairSet.Contains((i.ProdDetailID, i.Location_ID)))
                .ToList();
            
            return inventoryList.ToDictionary(i => (i.ProdDetailID, i.Location_ID));
        }
        
        /// <summary>
        /// 按(ProdDetailID, LocationID)分组聚合明细数据
        /// </summary>
        public Dictionary<(long ProdDetailID, long LocationID), TAggregate> 
            AggregateDetails<TDetail, TAggregate>(
                IEnumerable<TDetail> details,
                Func<TDetail, (long ProdDetailID, long LocationID)> keySelector,
                Func<TDetail, TAggregate> initialFactory,
                Action<TAggregate, TDetail> accumulator)
            where TAggregate : class
        {
            var groups = new Dictionary<(long, long), TAggregate>();
            
            foreach (var detail in details)
            {
                var key = keySelector(detail);
                
                if (!groups.TryGetValue(key, out var aggregate))
                {
                    aggregate = initialFactory(detail);
                    groups[key] = aggregate;
                }
                else
                {
                    accumulator(aggregate, detail);
                }
            }
            
            return groups;
        }
        
        /// <summary>
        /// 对库存列表按(ProdDetailID, LocationID)排序(死锁预防)
        /// </summary>
        public List<tb_Inventory> SortInventoryForDeadlockPrevention(
            IEnumerable<tb_Inventory> inventories)
        {
            return inventories
                .OrderBy(i => i.ProdDetailID)
                .ThenBy(i => i.Location_ID)
                .ToList();
        }
    }
}
```

**使用示例**:
```csharp

public tb_PurEntryController(..., InventoryAggregationService invAggService)
{
    _invAggService = invAggService;
}

// 简化后的代码

    var invDict = await _invAggService.PreloadInventoryAsync(prodDetailIds, requiredPairs);

    var inventoryGroups = _invAggService.AggregateDetails(
        entity.tb_PurEntryDetails,

        },

```

#### 优化方案

**统一结案逻辑**:

```csharp
public async override Task<ReturnResults<bool>> BatchCloseCaseAsync(List<T> NeedCloseCaseList)
{
    var entitys = NeedCloseCaseList as List<tb_SaleOut>;
    var rs = new ReturnResults<bool>();
    
    try
    {
        // ========== 第一阶段: 预处理(无事务) ==========
        
        // 1.1 验证所有单据状态
        foreach (var entity in entitys)
        {
            if (entity.DataStatus != (int)DataStatus.确认 || 
                entity.ApprovalStatus != (int)ApprovalStatus.审核通过 ||
                entity.ApprovalResults != true)
            {
                rs.ErrorMsg = $"单据{entity.SaleOutNo}状态不符合结案要求";
                rs.Succeeded = false;
                return rs;
            }
        }
        
        // 1.2 收集需要更新的订单ID
        var orderIds = entitys
            .Where(e => e.tb_saleorder != null)
            .Select(e => e.tb_saleorder.SOrder_ID)
            .Distinct()
            .ToList();
        
        // 1.3 批量加载订单(避免N+1查询)
        var orders = await _unitOfWorkManage.GetDbClient()
            .Queryable<tb_SaleOrder>()
            .Where(o => orderIds.Contains(o.SOrder_ID))
            .ToListAsync();
        
        var orderDict = orders.ToDictionary(o => o.SOrder_ID);
        
        // ========== 第二阶段: 事务内执行 ==========
        _unitOfWorkManage.BeginTran();
        
        try
        {
            // 2.1 更新订单付款状态
            foreach (var entity in entitys)
            {
                if (entity.tb_saleorder != null && 
                    orderDict.TryGetValue(entity.tb_saleorder.SOrder_ID, out var order))
                {
                    // 只有完全出库时才更新付款状态
                    if (order.TotalQty == entity.tb_SaleOutDetails.Sum(c => c.Quantity))
                    {
                        order.PayStatus = entity.PayStatus.Value;
                        BusinessHelper.Instance.EditEntity(order);
                    }
                }
                
                // 2.2 更新出库单状态
                entity.DataStatus = (int)DataStatus.完结;
                BusinessHelper.Instance.EditEntity(entity);
            }
            
            // 2.3 批量更新(减少数据库往返)
            if (orders.Any())
            {
                await _unitOfWorkManage.GetDbClient()
                    .Updateable(orders)
                    .UpdateColumns(o => new { o.PayStatus, o.Modified_by, o.Modified_at })
                    .ExecuteCommandAsync();
            }
            
            await _unitOfWorkManage.GetDbClient()
                .Updateable(entitys)
                .UpdateColumns(e => new { e.DataStatus, e.Modified_by, e.Modified_at })
                .ExecuteCommandAsync();
            
            _unitOfWorkManage.CommitTran();
            rs.Succeeded = true;
            return rs;
        }
        catch (Exception ex)
        {
            _unitOfWorkManage.RollbackTran();
            throw;
        }
    }
    catch (Exception ex)
    {
        _logger.Error(ex, "批量结案失败");
        rs.ErrorMsg = ex.Message;
        rs.Succeeded = false;
        return rs;
    }
}
```

**问题**:

#### 优化方案

**创建业务异常类**:

```csharp
namespace RUINORERP.Common.Exceptions
{
    /// <summary>
    /// 业务异常基类
    /// </summary>
    public class BusinessException : Exception
    {
        public string ErrorCode { get; }
        public object ErrorData { get; }
        
        public BusinessException(string message, string errorCode = "BUSINESS_ERROR", object errorData = null) 
            : base(message)
        {
            ErrorCode = errorCode;
            ErrorData = errorData;
        }
    }
    
    /// <summary>
    /// 库存相关异常
    /// </summary>
    public class InventoryException : BusinessException
    {
        public InventoryException(string message, string errorCode = "INVENTORY_ERROR", object errorData = null) 
            : base(message, errorCode, errorData) { }
    }
    
    /// <summary>
    /// 订单相关异常
    /// </summary>
    public class OrderException : BusinessException
    {
        public OrderException(string message, string errorCode = "ORDER_ERROR", object errorData = null) 
            : base(message, errorCode, errorData) { }
    }
    
    /// <summary>
    /// 财务相关异常
    /// </summary>
    public class FinanceException : BusinessException
    {
        public FinanceException(string message, string errorCode = "FINANCE_ERROR", object errorData = null) 
            : base(message, errorCode, errorData) { }
    }
}
```

## ⚠️ 风险评估

### 风险1: 财务独立事务导致数据不一致

**场景**: 主事务提交成功,财务事务失败

**缓解措施**:
1. 实现补偿机制(删除已创建的财务单据)
2. 添加定时任务检测并修复不一致数据
3. 记录详细日志,便于人工介入

**影响**: 低 - 财务数据可通过补偿机制恢复

### 风险2: 公共服务类引入新的bug

**缓解措施**:
1. 编写完整的单元测试
2. 灰度发布,先在小范围测试
3. 保留原有代码作为备份

**影响**: 中 - 需充分测试

### 风险3: 重构期间影响正常业务

**缓解措施**:
1. 分阶段实施,每阶段独立测试
2. 提供回滚方案
3. 选择业务低峰期部署

**影响**: 低 - 可控
