## 🎯 核心功能

### 1. 自动超时回滚

```csharp
// 如果忘记提交/回滚,60秒后会自动回滚
await _unitOfWork.BeginTranAsync(timeoutSeconds: 60);
try
{
    await DoWork();
    // 忘记调用 CommitTranAsync...
}
catch
{
    await _unitOfWork.RollbackTranAsync();
    throw;
}

// 60秒后触发:
// 1. 记录错误日志
// 2. 执行强制回滚
// 3. 重置事务状态
```

### 2. 长事务监控

```
[Transaction-xxx] ⚠️ 长事务警告: 已运行 75秒 (>60秒), 调用方=tb_PurInController.ApproveAsync
[Transaction-yyy] 🚨 超长事务警告! 已运行 350秒 (>300秒), 调用方=tb_ProdOrderController.SubmitAsync
```

### 3. 资源泄漏检测

```
[Transaction-zzz] ⚠️ Dispose时发现长事务! 已运行 120秒, 深度=1, 调用方=tb_StockController.AdjustAsync
[Transaction-zzz] 检测到未完成的事务，执行强制回滚
```


## 🛡️ 安全保障

### 多层防护

1. **第一层: 超时自动回滚**
   - 超时后立即尝试回滚
   - 防止数据库锁长期持有

2. **第二层: Dispose 检测**
   - 对象销毁时检测未完成事务
   - 记录警告并强制回滚

3. **第三层: 长事务告警**
   - 60秒警告,300秒错误
   - 便于及时发现性能问题

### 线程安全

- ✅ 每个 TransactionContext 独立的 CancellationTokenSource
- ✅ AsyncLocal 确保不同异步流隔离
- ✅ 无跨上下文竞争

### 性能影响

- ✅ 开销极小(仅一个 Timer)
- ✅ 正常事务无性能损失
- ✅ 超时后才执行额外操作

## ⚠️ 注意事项

### 1. 超时是"尽力而为"的保护

- ✅ 主要目的: **检测和告警**
- ✅ 次要目的: **尽可能自动回滚**
- ❌ 不是: **替代正确的异常处理**

### 2. 超时回调的局限性

由于在后台线程执行:
- 可能无法获取 HttpContext
- 数据库连接可能已关闭
- 强制回滚可能失败

**解决方案**: 即使超时回滚失败,也会记录日志,Dispose 会再次尝试。

### 3. 不要依赖超时作为正常流程

```csharp
// ❌ 错误做法: 故意让事务超时
await _unitOfWork.BeginTranAsync(timeoutSeconds: 5);
await LongRunningWork();  // 期望6秒后自动回滚

// ✅ 正确做法: 主动管理事务
await _unitOfWork.BeginTranAsync(timeoutSeconds: 60);
try
{
    await LongRunningWork();
    await _unitOfWork.CommitTranAsync();  // 明确提交
}
catch
{
    await _unitOfWork.RollbackTranAsync();  // 明确回滚
    throw;
}
```
