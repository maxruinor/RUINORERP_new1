## 📊 核心功能展示

### 功能1: 自动超时回滚

```csharp
// 如果忘记提交/回滚,60秒后会自动回滚
await _unitOfWork.BeginTranAsync(timeoutSeconds: 60);
try
{
    await DoWork();
    // 忘记调用 CommitTranAsync...
}
catch
{
    await _unitOfWork.RollbackTranAsync();
    throw;
}

// 60秒后触发:
// [Transaction-xxx] ⚠️ 事务超时! 配置超时=60秒, 实际运行=60.5秒
// [Transaction-xxx] 执行强制回滚
// [Transaction-xxx] 超时强制回滚成功
```

```
[Transaction-zzz] 🚨 超长事务警告! 已运行 350秒 (>300秒), 调用方=tb_ProdOrderController.SubmitAsync
```

### 功能3: 资源泄漏检测

## 🛡️ 安全保障

### 多层防护机制

1. **第一层: 超时自动回滚**
   - 超时后立即尝试回滚
   - 防止数据库锁长期持有

2. **第二层: Dispose 检测**
   - 对象销毁时检测未完成事务
   - 记录警告并强制回滚

3. **第三层: 长事务告警**

## ⚠️ 注意事项

### 1. 超时是"尽力而为"的保护

- ✅ **主要目的**: 检测和告警问题事务
- ✅ **次要目的**: 在可能的情况下自动回滚
- ❌ **不是**: 替代正确的异常处理和事务管理

### 2. 不要依赖超时作为正常流程

```csharp
// ❌ 错误做法
await _unitOfWork.BeginTranAsync(timeoutSeconds: 5);
await LongRunningWork();  // 期望6秒后自动回滚

// ✅ 正确做法
await _unitOfWork.BeginTranAsync(timeoutSeconds: 60);
try
{
    await LongRunningWork();
    await _unitOfWork.CommitTranAsync();  // 明确提交
}
catch
{
    await _unitOfWork.RollbackTranAsync();  // 明确回滚
    throw;
}
```

### 3. 超时回调的局限性

由于在后台线程执行:
- 可能无法获取 HttpContext
- 数据库连接可能已关闭
- 强制回滚可能失败

**解决方案**: 即使超时回滚失败,也会记录日志,Dispose 会再次尝试。

## 💬 常见问题

### Q1: 超时时间设置多少合适?

**A**: 
- 短事务: 30秒
- 普通事务: 60秒(默认)
- 复杂事务: 120-300秒
- 批量处理: 根据数据量调整

### Q2: 可以禁用自动超时吗?

**A**: 可以,设置 `EnableAutoTransactionTimeout = false`,但不推荐。

### Q3: 超时后一定会回滚吗?

**A**: 如果 `ForceRollbackOnTimeout = true` 且连接有效,会尝试回滚。但如果连接已关闭或事务已完成,回滚可能失败,此时会记录错误日志。

### Q4: 会影响现有代码吗?

**A**: 不会。所有新参数都是可选的,默认行为与之前一致。现有代码无需修改即可享受超时保护。

### Q5: 如何监控超时事件?

**A**: 查看日志中的 Error 级别消息,搜索关键词 "事务超时"。未来可以集成 Prometheus + Grafana 实现可视化监控。
