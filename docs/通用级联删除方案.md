# 通用级联删除方案 - 基于实体导航属性自动处理

## 📋 概述

本方案通过在 `BaseController` 中添加通用辅助方法 `DeleteDeepLevelRelationsByNavigationAsync`,**自动扫描实体的导航属性并删除深层级关联数据**,避免外键约束冲突。

### ✨ 核心优势

1. **无需手动维护**: 不需要在每个 Controller 中硬编码删除逻辑
2. **自动化处理**: 基于实体类中的 `[Navigate]` 特性自动识别关联关系
3. **智能检测**: 自动检测哪些关联表有子表,优先删除子表数据
4. **递归支持**: 支持多层级级联删除(可配置深度)
5. **容错性强**: 单个导航属性处理失败不影响其他属性

## 🔧 使用方法

### 1. 在 Controller 的 BaseDeleteByNavAsync 中调用

```csharp
public async override Task<bool> BaseDeleteByNavAsync(T model) 
{
    var entity = model as YourEntity;
    
    // ✅ 只需一行代码,自动处理所有深层级关联
    await DeleteDeepLevelRelationsByNavigationAsync(entity);
    
    // 然后执行正常的级联删除
    bool rs = await _unitOfWorkManage.GetDbClient()
        .DeleteNav<YourEntity>(m => m.Id == entity.Id)
        .Include(m => m.RelatedTable1)
        .Include(m => m.RelatedTable2)
        .ExecuteCommandAsync();
    
    return rs;
}
```

### 2. 工作原理

```
CustomerVendor (主表)
├── tb_FM_ReceivablePayable (一对多)
│   └── tb_FM_ReceivablePayableDetail (子表) ← 自动检测并先删除
├── tb_FM_Invoice (一对多)
│   └── tb_FM_InvoiceDetail (子表) ← 自动检测并先删除
└── tb_FM_Statement (一对多)
    └── tb_FM_StatementDetail (子表) ← 自动检测并先删除
```

**执行流程**:
1. 扫描 `tb_CustomerVendor` 的所有 `[Navigate(NavigateType.OneToMany)]` 属性
2. 对每个导航属性,检查目标表是否有自己的 OneToMany 导航(即是否有子表)
3. 如果有子表:
   - 查询所有关联的主表记录ID
   - 批量删除子表数据
4. 继续执行 SqlSugar 的 `DeleteNav().Include()` 删除主表和直接关联表

## 📝 技术细节

### 关键方法说明

#### `DeleteDeepLevelRelationsByNavigationAsync<TMain>(TMain mainEntity, int maxDepth = 2)`

**参数**:
- `mainEntity`: 主实体实例(必须已设置主键值)
- `maxDepth`: 最大递归深度,默认2层(防止无限递归)

**功能**:
- 自动扫描实体的所有 OneToMany 导航属性
- 检测目标表是否有子表
- 递归删除深层级关联数据

#### 内部辅助方法

- `ProcessOneToManyRelationAsync`: 处理单个一对多关系
- `DeleteChildTableDataAsync`: 删除子表数据
- `GetNavigationTargetType`: 获取导航属性的目标类型
- `GetPrimaryKeyName`: 获取实体的主键名称

## ⚠️ 注意事项

### 1. 实体类必须有正确的 Navigate 特性

```csharp
// ✅ 正确示例
[Navigate(NavigateType.OneToMany, nameof(tb_FM_ReceivablePayableDetail.ARAPId))]
public virtual List<tb_FM_ReceivablePayableDetail> tb_FM_ReceivablePayableDetails { get; set; }

// ❌ 错误:缺少 Navigate 特性或 ForeignKeys 参数
public virtual List<tb_FM_ReceivablePayableDetail> tb_FM_ReceivablePayableDetails { get; set; }
```

### 2. 主键必须正确设置

```csharp
var entity = new tb_CustomerVendor();
entity.CustomerVendor_ID = 123; // ✅ 必须设置主键
await controller.BaseDeleteByNavAsync(entity);
```

### 3. 递归深度控制

默认 `maxDepth = 2`,可根据实际情况调整:

```csharp
// 如果需要更深的递归(不推荐超过3层,性能考虑)
await DeleteDeepLevelRelationsByNavigationAsync(entity, maxDepth: 3);
```

### 4. 日志输出

方法会自动记录删除操作:

```
已清理 tb_FM_ReceivablePayable 的 5 条记录的子表数据
已删除 tb_FM_ReceivablePayableDetail 的 12 条记录
```

## 🎯 适用场景

### ✅ 适合使用

1. **有大量一对多关系的实体**: 如客户、供应商、产品等基础资料
2. **关联表有明细子表**: 如订单→订单明细、发票→发票明细
3. **需要统一处理多个 Controller**: 避免重复编写删除逻辑

### ❌ 不适合使用

1. **简单的单表删除**: 没有关联关系或只有一层关联
2. **特殊的业务逻辑**: 需要在删除前执行特殊校验或操作
3. **性能敏感场景**: 反射和动态SQL有一定开销

## 🚀 迁移指南

### 从手动删除迁移到自动删除

**之前的代码**:
```csharp
// ❌ 需要手动维护,容易遗漏
private async Task DeleteDeepLevelRelationsAsync(long customerVendorId)
{
    var arapIds = await db.Queryable<tb_FM_ReceivablePayable>()
        .Where(x => x.CustomerVendor_ID == customerVendorId)
        .Select(x => x.ARAPId)
        .ToListAsync();
    
    if (arapIds.Any())
    {
        await db.Deleteable<tb_FM_ReceivablePayableDetail>()
            .Where(x => arapIds.Contains(x.ARAPId))
            .ExecuteCommandAsync();
    }
    // ... 更多手动代码
}
```

**现在的代码**:
```csharp
// ✅ 一行搞定,自动处理
await DeleteDeepLevelRelationsByNavigationAsync(entity);
```

## 🐛 常见问题

### Q1: 为什么有些关联没有被删除?

**A**: 检查以下几点:
1. 实体类中是否有 `[Navigate(NavigateType.OneToMany)]` 特性
2. `ForeignKeys` 参数是否正确(格式: "TableName.FieldName")
3. 目标表的主键是否能正确识别

### Q2: 性能如何?

**A**: 
- 对于普通实体(10-20个导航属性),额外开销 < 100ms
- 主要开销在于反射和动态SQL生成
- 建议设置合理的 `maxDepth`(默认2层)

### Q3: 能否禁用某个导航属性的自动处理?

**A**: 可以,在该导航属性上添加自定义标记,然后在 `DeleteDeepLevelRelationsByNavigationAsync` 中过滤掉。
