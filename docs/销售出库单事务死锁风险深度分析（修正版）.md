| 前期错误分析 | 实际代码实现 | 真相 |

| "事务上下文在 async/await 后丢失" | `AsyncLocal<TransactionContext>` (Line 32-33) | ✅ **事务上下文不会丢失**，AsyncLocal 保证异步流传递 |

### 🎯 真正的死锁风险点

基于代码实证，真正的风险在于:

1. **长事务持有大量锁资源** (Line 52-223, RefundProcessAsync 事务跨度约 170 行代码)
2. **保存点回滚的复杂性** (Line 313-324, 多层嵌套可能导致回滚混乱)
3. **缺少 TransactionScope 保护** (虽然 AsyncLocal 有效，但 .NET Framework 有已知问题)
4. **高并发场景下的锁竞争** (平台退款 + 人工审核 + 自动转单并发执行)

**关键技术发现**:

| 嵌套层级 | 实际行为 | SQL Server 对应命令 |
|---------|---------|-------------------|
| Depth=1 | 开启物理事务 | `BEGIN TRANSACTION` |
| Depth=2 | 创建保存点 SP_2 | `SAVE TRANSACTION SP_..._2` |
| Depth=3 | 创建保存点 SP_3 | `SAVE TRANSACTION SP_..._3` |
| Commit (Depth>1) | 仅减少计数 | 无 SQL 命令 |
| Commit (Depth=1) | 提交物理事务 | `COMMIT TRANSACTION` |
| Rollback (Depth>1) | 回滚到保存点 | `ROLLBACK TRANSACTION SP_...` |
| Rollback (Depth=1) | 回滚全部 | `ROLLBACK TRANSACTION` |

#### 1.1.3 提交与回滚逻辑

```csharp

private void CommitTranInternal()
{
    var context = CurrentTransactionContext;
    var dbClient = GetDbClient();
    
    _rwLock.EnterWriteLock();
    try
    {
        if (context.Depth <= 0) return;
        
        // Line 214-220: 检查回滚标记
        if (context.ShouldRollback)
        {
            _logger.LogWarning($"事务已标记为回滚，跳过提交");
            context.Depth--;
            _tranDepth.Value--;
            return;  // ← 直接返回，不提交
        }
        
        // Line 222-231: 减少深度
        context.Depth--;
        _tranDepth.Value--;
        
        // 内层提交：仅更新深度，不真正提交
        if (context.Depth > 0)
        {
            return;  // ← 关键！内层 Commit 不执行任何数据库操作
        }
        
        // Line 233-254: 最外层提交
        try
        {
            dbClient.Ado.CommitTran();  // ← 只有这里真正提交
            context.Status = TransactionStatus.Committed;
            
            // 长事务警告
            var duration = context.GetDuration().TotalSeconds;
            if (duration > 10)
            {
                _logger.LogWarning($"[Transaction-{context.TransactionId}] 长事务提交耗时：{duration:F2}秒");
            }
        }
        catch (Exception commitEx)
        {
            context.Status = TransactionStatus.RolledBack;
            throw;
        }
        finally
        {
            ResetTransactionState();  // 清理上下文
        }
    }
    finally
    {
        _rwLock.ExitWriteLock();
    }
}
```

**关键发现**:
- ✅ **深度计数保护**: 内层 Commit 只是减少计数，不会真正提交
- ✅ **回滚标记检查**: 如果标记了 ShouldRollback，Commit 会被跳过
- ⚠️ **长事务监控**: Line 240-243 有长事务日志，但只是记录，没有自动终止

```csharp
public void RollbackTran()
{
    var context = CurrentTransactionContext;
    var dbClient = GetDbClient();
    
    _rwLock.EnterWriteLock();
    try
    {
        if (context.Depth <= 0) return;
        
        // Line 291-297: 设置回滚标记并减少深度
        context.ShouldRollback = true;
        context.Depth--;
        _tranDepth.Value--;
        
        // Line 299-311: 最外层回滚
        if (context.Depth == 0)
        {
            dbClient.Ado.RollbackTran();  // ← 回滚整个事务
            context.Status = TransactionStatus.RolledBack;
        }
        // Line 313-324: 嵌套回滚 - 回滚到保存点
        else if (context.SavePointStack.TryPop(out var savePoint))
        {
            dbClient.Ado.ExecuteCommand($"ROLLBACK TRANSACTION {savePoint}");  // ← 只回滚到保存点
        }
    }
    finally
    {
        if (context.Depth <= 0)
        {
            ResetTransactionState();
        }
        _rwLock.ExitWriteLock();
    }
}
```

## 2. 真正的死锁风险点（基于代码实证）

### 2.1 风险点 1: 长事务持有大量锁资源

**代码位置**: `RefundProcessAsync` Line 52-223

**事务跨度分析**:
```csharp
BeginTran()  // Line 52
├─ 对象映射 (Line 53-83)
├─ 循环处理明细 (Line 95-166)
│  └─ 缓存查询 (Line 108-115, 142-150)
├─ 运费计算 (Line 175-187)
├─ 业务编号生成 (Line 198-199)
├─ BaseSaveOrUpdateWithChild (Line 210-211)
│  └─ 插入 SaleOutRe (Depth=2, 保存点内)
└─ 更新 SaleOut (Line 216-219)
CommitTran()  // Line 223
```

**执行时间估算**:
| 操作 | 预估耗时 | 持有锁类型 |
|------|---------|-----------|
| 对象映射 | 5-10ms | 无锁 |
| 缓存查询 (多次) | 50-200ms | 无锁 |
| 业务编号生成 | 10-50ms | 无锁 |
| **BaseSaveOrUpdateWithChild** | **100-500ms** | **X 锁 (SaleOutRe 表)** |
| **更新 SaleOut** | **50-200ms** | **X 锁 (SaleOut 表)** |
| **总计** | **215-960ms** | **X 锁持有 ~700ms** |

**死锁场景**:

```
时间轴:
T0:   事务 A (RefundProcessAsync) 开启
T1:   事务 B (审核流程) 开启
T2:   事务 A 插入 SaleOutRe (持有 SaleOutRe 表 X 锁)
T3:   事务 B 更新 SaleOut (持有 SaleOut 表 X 锁)
T4:   事务 A 尝试更新 SaleOut → ⛔ 阻塞，等待事务 B 释放锁
T5:   事务 B 尝试查询 SaleOutRe → ⛔ 阻塞，等待事务 A 释放锁
T6:   SQL Server 检测到死锁，选择牺牲者
```

**根本原因**:
- ❌ **事务跨度过大**: 170 行代码持有事务锁
- ❌ **锁粒度粗**: 更新操作使用排他锁 (X 锁)
- ❌ **缺少索引**: 可能导致锁升级

### 2.2 风险点 2: 保存点回滚的复杂性

**代码位置**: `UnitOfWorkManage.cs` Line 313-324

**危险场景**:


**实际问题**:

- ⚠️ **部分提交风险**: 操作 1 和操作 3 被提交，操作 2 被回滚

```csharp
public async Task<ReturnResults<tb_SaleOutRe>> RefundProcessAsync(tb_SaleOut saleout)
{
    int retryCount = 0;
    const int MAX_RETRY = 3;
    
    while (true)
    {
        try
        {
            _unitOfWorkManage.BeginTran();
            
            // ... 业务逻辑 ...
            
            _unitOfWorkManage.CommitTran();
            rs.Succeeded = true;
            return rs;
        }
        catch (SqlException sqlEx) when (sqlEx.Number == 1205 && retryCount < MAX_RETRY)
        {
            retryCount++;
            var delayMs = (int)(100 * Math.Pow(2, retryCount)); // 指数退避
            
            _logger.LogWarning(sqlEx, $"检测到死锁，第 {retryCount}/{MAX_RETRY} 次重试，延迟 {delayMs}ms");
            _unitOfWorkManage.RollbackTran();
            
            await Task.Delay(delayMs);
        }
        catch (Exception ex)
        {
            _unitOfWorkManage.RollbackTran();
            _logger.Error(ex);
            rs.ErrorMsg = ex.Message;
            rs.Succeeded = false;
            return rs;
        }
    }
}
```
