## 二、改善方案（稳定可行）

### 2.1 方案概述

**核心思路**：为整个审核事务添加重试保护，确保发生死锁时自动重试。

```
┌─────────────────────────────────────────────────────────────────┐
│                    ExecuteWithRetryAsync 包装                    │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │ BeginTran() → [所有审核操作] → CommitTran()              │  │
│  └───────────────────────────────────────────────────────────┘  │
│                              ↓                                   │
│                     死锁时自动重试 (3次)                        │
└─────────────────────────────────────────────────────────────────┘
```

#### 修改2：处理财务模块的特殊性

**问题**: 财务独立事务在主事务提交后执行，如果也用同一重试包装，会导致财务失败时整个事务回滚。

**解决方案**: 将财务处理放在重试范围外，单独处理。

**修改后 ApprovalAsyncCore 结构**:
```csharp
private async Task<ReturnResults<T>> ApprovalAsyncCore(tb_SaleOut entity, FMAuditLogHelper fMAuditLog)
{
    ReturnResults<T> rrs = new ReturnResults<T>();
    
    // === 1. 预加载阶段（无事务）===
    // 批量查询库存和价格记录
    var inventoryDict = await LoadInventoryDictAsync(entity);
    var priceRecordDict = await LoadPriceRecordDictAsync(entity);
    
    // === 2. 主事务阶段（有重试保护）===
    _unitOfWorkManage.BeginTran();
    try
    {
        // 库存更新
        await ProcessInventoryUpdatesAsync(entity, inventoryDict);
        
        // 订单回写
        await ProcessOrderUpdatesAsync(entity);
        
        // CRM更新
        await ProcessCrmUpdatesAsync(entity);
        
        // 出库单状态更新
        await UpdateSaleOutStatusAsync(entity);
        
        // 提交前保存财务数据
        bool needProcessFinance = CaptureFinanceData(entity, out var financeData);
        
        _unitOfWorkManage.CommitTran();
        
        // === 3. 财务独立处理（在事务外）===
        if (needProcessFinance)
        {
            await ProcessFinanceIndependentlyAsync(financeData, fMAuditLog);
        }
        
        rrs.Succeeded = true;
        rrs.ReturnObject = entity as T;
        return rrs;
    }
    catch (Exception ex)
    {
        _unitOfWorkManage.RollbackTran();
        throw; // 抛出让ExecuteWithRetry捕获重试
    }
}
```

## 四、重试机制详解

### 4.1 ExecuteWithRetryAsync 执行流程

```
┌──────────────────────────────────────────────────────────────┐
│                  ExecuteWithRetryAsync                       │
├──────────────────────────────────────────────────────────────┤
│  第1次尝试:                                                   │
│  ┌─────────────────────────────────────────────────────┐     │
│  │ BeginTran() → [业务逻辑] → CommitTran()             │     │
│  └─────────────────────────────────────────────────────┘     │
│                           ↓                                   │
│                    成功? → 返回结果 ✅                          │
│                           ↓ 失败                               │
│                    是死锁错误?                                 │
│                    (SqlException.Number == 1205)              │
│                           ↓                                   │
│                      是 → 记录日志 → 回滚                      │
│                           ↓                                   │
│                   延迟 100ms × 2^(重试次数-1)                  │
│                    (第1次: 100ms, 第2次: 200ms)               │
│                           ↓                                   │
│  第2次尝试: (重复上述流程)                                     │
│                           ↓                                   │
│  第3次尝试: (重复上述流程)                                     │
│                           ↓                                   │
│                   超过最大重试次数?                            │
│                      是 → 抛出异常 ❌                          │
└──────────────────────────────────────────────────────────────┘
```

### 4.2 重试参数

| 参数 | 当前值 | 说明 |
|------|--------|------|
| maxRetryCount | 3 | 最大重试3次 |
| retryDelayMs | 100 | 初始延迟100ms |
| 指数退避 | 2^(n-1) | 第1次100ms, 第2次200ms, 第3次400ms |
| 总最大延迟 | ~700ms | 100+200+400 |
