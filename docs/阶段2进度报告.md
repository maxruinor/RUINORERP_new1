```csharp
public void PreloadForeignKeyData(IEnumerable<ColumnMapping> mappings)

public object GetForeignKeyValue(string tableName, string fieldValue, out string errorMessage)

public object GetForeignKeyId(string tableName, string fieldName, string fieldValue, out string errorMessage)
```

#### 性能优势

- **预加载模式**: 导入前一次性查询所有关联表，避免逐行查询
- **缓存命中率**: 预期 > 90%（相同外键值多次出现）
- **查询优化**: 参数化SQL防止注入，支持模糊匹配

```csharp
public DeduplicationResult Deduplicate(
    DataTable dataTable, 
    List<string> deduplicationFields, 
    string strategy = "FirstOccurrence")

// 2. 查找重复记录（用于UI提示）
public List<List<int>> FindDuplicates(
    DataTable dataTable, 
    List<string> deduplicateFields, 
    bool ignoreEmptyValues = true)
```

#### 去重策略

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| **FirstOccurrence** | 保留第一条记录 | 默认策略，适合大多数场景 |
| **LastOccurrence** | 保留最后一条记录 | 后续数据覆盖前面数据 |

#### 算法特点

- **HashSet去重**: O(n)时间复杂度，高效
- **多字段组合**: 支持多个字段组合作为去重键
- **空值处理**: 可配置是否忽略空值
- **顺序保持**: 去重后保持原始数据顺序

## 💡 关键技术决策

### 为什么简化DataDeduplicationService的方法签名？

**原因**:
1. **解耦配置类** - ImportConfiguration是UI层的配置类，Business层不应依赖
2. **灵活性** - 直接传入参数，调用方可以自由组合
3. **测试友好** - 单元测试不需要构造复杂的ImportConfiguration对象

### 为什么ForeignKeyCacheService保留两个Get方法？

**原因**:
1. **GetForeignKeyValue** - 简单场景，只需要表名和值
2. **GetForeignKeyId** - 完整场景，需要指定字段名，支持缓存未命中时查询数据库
3. **向后兼容** - 保留原有调用方式，同时提供更强大的新版本

### 为什么去除ConvertToDisplayNames方法？

**原因**:
1. **职责分离** - 字段名映射是UI层的职责
2. **Business层纯粹** - Business层只关心数据库字段名
3. **避免循环依赖** - ImportConfiguration在UI层，Business层不应引用

## ⚠️ 注意事项

### 1. 命名空间统一
所有新创建的服务类必须使用：

### 2. 依赖注入
所有服务通过构造函数注入ISqlSugarClient：
```csharp
public ForeignKeyCacheService(ISqlSugarClient db)
{
    _db = db ?? throw new ArgumentNullException(nameof(db));
}
```

使用Debug.WriteLine记录关键操作：
```csharp
Debug.WriteLine($"成功预加载外键数据: {tableName}.{fieldName}, 缓存记录数: {count}");
```

### 4. 异常处理
- 业务异常：抛出具体异常类型
- 系统异常：捕获并记录日志，返回null或错误信息
- 不直接在Business层显示MessageBox
