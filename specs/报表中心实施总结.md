## 架构亮点

### 1. 高内聚低耦合的设计

通过三层架构实现关注点分离:

```
UI 层 (报表容器/查询面板)
    ↓
适配层 (BaseChartAdapter)
    ↓
服务层 (IStatisticsService + AggregationService)
    ↓
数据层 (SqlSugar + Database)
```

**优势**:
- 每层职责明确，易于测试和维护
- 支持灵活替换实现 (如 MemoryCache → Redis)
- 新业务场景只需扩展 Adapter，无需修改核心逻辑

### 2. 模板方法模式的应用

`BaseChartAdapter` 定义了标准流程:

```csharp
public async Task<ChartData> GetDataAsync(DataRequest request)
{
    ValidateRequest(request);          // 验证
    var query = BuildBaseQuery(request); // 构建查询
    ApplyTimeFilter(ref query, request); // 时间过滤
    ApplyCustomFilters(ref query, request); // 自定义过滤
    ApplyGroupBy(ref query, request);     // 分组
    var rawData = await ExecuteQueryAsync(query); // 执行
    return TransformToChartData(rawData, request); // 转换
}
```

子类只需关注:
- `PrimaryTableName`: 指定主表
- `GetDimensions()`: 定义维度
- `GetMetrics()`: 定义指标
- `TransformToChartData()`: 数据转换

### 3. 动态 SQL 构建

`DynamicSqlBuilder` 支持运行时构建复杂查询:

```csharp
var sqlBuilder = new DynamicSqlBuilder(_db);
var query = sqlBuilder
    .From("tb_SaleOrder")
    .SelectDimensions(new[] { "Employee_ID", "Customer_ID" })
    .SelectMetrics(new[] { "TotalAmount", "OrderID" })
    .WhereTimeRange("Created_at", startTime, endTime)
    .ApplyFilters(filters)
    .GroupByDimensions(dimensions)
    .Page(1, 100)
    .Build();
```

### 4. 智能缓存策略

`AggregationService` 自动生成缓存键:

```csharp
// 缓存键示例
Stats:tb_SaleOrder:D=Created_at,Employee_ID:M=OrderID,TotalAmount:ST=20240101120000:ET=20240401120000
```

相同参数的查询会直接返回缓存结果 (TTL=30 分钟)。

### 5. 插件化报表容器

`ReportContainer` 支持动态注册报表类型:

```csharp
reportContainer.RegisterReportType<OrderAnalysisReport>();
reportContainer.RegisterReportType<InventoryAnalysisReport>();
```

每个报表独立管理自己的:
- 查询参数
- 数据源
- 渲染逻辑

## 关键技术决策

### 决策 1: 为什么不引入 OLAP 引擎？

**背景**: 设计方案中明确不引入专门数据仓库。

**理由**:
1. 系统规模尚未达到 PB 级
2. 实时性要求高于离线分析
3. 增加运维复杂度
4. 现有 SqlSugar+ 视图方案足够

**替代方案**:
- 预聚合视图 (参考 `Proc_SaleOutStatisticsByEmployee`)
- Redis 缓存热点数据
- 分页查询 + 懒加载

### 决策 2: 继续使用 LiveChartsCore vs 商业控件

**选择**: LiveChartsCore + SkiaSharp

**理由**:
- ✅ 团队已有技术积累
- ✅ GPU 加速渲染 (GpuAcceleratedRenderer.cs 已实现)
- ✅ 开源免费，无商业授权风险
- ✅ 支持 WinForms/WPF 双平台

**风险缓解**:
- 已在 `UCCRMChart.cs` 中验证中文字体支持
- 提供 `ChartHelper`工具类简化颜色处理

### 决策 3: Service vs Adapter 的选择

**指导原则**:

| 场景 | 推荐 | 原因 |
|------|------|------|
| 简单聚合 (SUM/COUNT) | IStatisticsService | 通用、自带缓存 |
| 多表关联 | BaseChartAdapter | 灵活编写 SQL |
| 跨实体统计 | IStatisticsService | 动态构建 |
| 复杂业务逻辑 | Adapter | 可继承重写 |

## 与现有代码的集成

### 1. 复用 ChartFramework 现有组件

新架构完全兼容现有的:
- `ChartData` / `DataRequest` / `MetricConfig`等模型
- `ChartBuilderFactory` 图表构建工厂
- `ChartTheme`主题系统
- `ChartControl`包装控件

### 2. 迁移现有 Adapter

原有适配器 (如 `CustomerDataAdapter`, `CRMDataAdapter`) 可以逐步迁移到新架构:

**迁移步骤**:
1. 将类名改为 `XxxAnalysisAdapter`
2. 继承自 `BaseChartAdapter`
3. 实现抽象方法
4. 删除硬编码逻辑

**示例**:
```csharp
// 旧代码
public class CustomerDataAdapter : IDataProvider
{
    public async Task<ChartData> GetDataAsync(DataRequest request)
    {
        // 大量硬编码...
    }
}

// 新代码
public class CustomerAnalysisAdapter : BaseChartAdapter
{
    protected override string PrimaryTableName => "tb_Customer";
    
    // 更清晰的职责划分
}
```

### 3. 保留存储过程方案

对于性能敏感的统计，仍可沿用存储过程:

```csharp
public class ProcSaleAnalysisAdapter : BaseChartAdapter
{
    protected override async Task<List<dynamic>> ExecuteQueryAsync(ISugarQueryable<dynamic> query)
    {
        // 调用存储过程
        return await _db.Ado.ExecuteCommandAsync("Proc_SaleOutStatisticsByEmployee", 
            new { Start = request.StartTime, End = request.EndTime });
    }
}
```
