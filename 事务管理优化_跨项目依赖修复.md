# 跨项目依赖问题修复说明

## 🐛 问题描述

**错误信息：**
- 当前上下文中不存在名称"SqlsugarSetup"
- 命名空间"RUINORERP"中不存在类型或命名空间名"Extensions"

**根本原因：**
`UnitOfWorkManage` (位于 RUINORERP.Repository 项目) 试图调用 `SqlsugarSetup` (位于 RUINORERP.Extensions 项目)，但两个项目之间没有引用关系。

---

## ✅ 解决方案

### 移除跨项目调用

**修改内容：**
1. 删除 `using RUINORERP.Extensions;` 引用
2. 移除所有 `SqlsugarSetup.ApplyAopConfiguration()` 调用
3. 改用 `ConnectionConfig.AopEvents` 自动继承 AOP 配置

### 技术原理

SqlSugar 的 `ConnectionConfig` 包含 `AopEvents` 属性，该属性存储了所有 AOP 事件处理器。当创建新的 `SqlSugarScope` 实例并复制此配置时，新客户端会**自动继承**所有 AOP 事件，无需手动重新应用。

```csharp
// ✅ 正确的做法
var originalConfig = (_sqlSugarClient as SqlSugarScope)?.CurrentConnectionConfig;

if (originalConfig != null)
{
    var newConfig = new ConnectionConfig
    {
        // ... 其他配置
        ConfigureExternalServices = originalConfig.ConfigureExternalServices,
        AopEvents = originalConfig.AopEvents  // 关键：复制 AOP 事件
    };
    
    _asyncLocalClient.Value = new SqlSugarScope(newConfig);
    // 新客户端已自动拥有相同的 AOP 事件，无需额外配置
}
```

## 📝 架构优势

### 1. 保持项目独立性
- **Repository 项目**：专注于数据访问和事务管理
- **Extensions 项目**：提供 DI 注册和全局配置
- 两者解耦，符合单一职责原则

### 2. 利用 SqlSugar 的配置继承机制
SqlSugar 在设计时考虑了配置复用场景：
- `ConnectionConfig.AopEvents` 存储所有事件处理器
- 复制配置后，新实例自动使用相同的事件处理
- 避免了外部依赖和重复配置

### 3. 简化维护
- 减少项目间耦合
- AOP 配置集中管理（在 Extensions 项目中）
- Repository 项目只需复制配置即可

## 💡 最佳实践总结

### 1. 项目依赖原则
```
❌ 错误：底层模块依赖上层模块
   Repository → Extensions
   
✅ 正确：上层模块配置底层模块
   Extensions (配置) → Repository (使用)
```

### 2. 配置复用策略
```csharp
// 方案 A：复制配置（推荐）
var newConfig = new ConnectionConfig
{
    AopEvents = originalConfig.AopEvents
};

// 方案 B：共享配置对象（谨慎使用）
// 注意：多个客户端共享同一配置可能导致事件重复触发
```

### 3. AOP 配置时机
```
✅ 推荐：在 DI 注册时统一配置
   services.AddSqlsugarSetup(...)

✅ 推荐：复制配置到新客户端
   newConfig.AopEvents = originalConfig.AopEvents

❌ 避免：在业务代码中手动配置 AOP
   unitOfWork.GetDbClient().Aop.OnLogExecuting = ...
```
