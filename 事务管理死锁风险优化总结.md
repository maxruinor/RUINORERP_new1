# 事务管理体系死锁风险优化总结

## 优化概述

本次优化针对统一事务管理体系中存在的死锁风险、连接泄漏、并发安全等问题进行了全面改进，提升了系统的稳定性和容错能力。

---

## 一、主要优化内容

### ✅ 1. 防止并发任务共享同一连接实例

**问题：** 在同一个异步上下文中并行执行多个数据库操作（如 `Task.WhenAll`）会导致线程安全问题。

**解决方案：**
- **通过文档约束和代码审查**明确要求事务内不得使用并行操作
- 在每个异步上下文使用独立的 `SqlSugarScope` 实例（通过 `AsyncLocal`）
- 每个连接实例有自己的 `lock` 保护，确保串行访问

**说明：** 原计划使用并发计数器检测，但因 `AsyncLocal` 的隔离性导致检测无效，已移除。改为通过开发规范和代码审查来保证。
```csharp
// UnitOfWorkManage.cs
private readonly AsyncLocal<int> _concurrentCallCount = new AsyncLocal<int>();

public void BeginTran(IsolationLevel? isolationLevel = null)
{
    if (Interlocked.Increment(ref _concurrentCallCount.Value) > 1)
    {
        Interlocked.Decrement(ref _concurrentCallCount.Value);
        throw new InvalidOperationException(
            $"检测到同一事务上下文中存在并发操作，请避免在事务内使用并行任务...");
    }
    try
    {
        // ... 原有逻辑
    }
    finally
    {
        Interlocked.Decrement(ref _concurrentCallCount.Value);
    }
}
```

### ✅ 2. 增加死锁自动重试机制

**问题：** 发生死锁（`SqlException 1205`）后直接失败，缺乏容错机制。

**解决方案：**
- 实现公开的 `ExecuteWithRetry` 和 `ExecuteWithRetryAsync` 方法
- 使用指数退避策略（200ms, 400ms, 800ms）
- 记录详细的重试日志和事务上下文信息
- **由业务层决定是否重试整个事务块**（而非仅在 Commit 时重试）

**重要调整：**
- 移除了 `CommitTran` 内部的自动重试，避免事务上下文重置后重复提交
- 重试机制作为公开方法，由业务层在完整事务块外层调用



### ✅ 3. 支持自定义事务隔离级别

**问题：** 默认使用数据库隔离级别，未针对高并发场景优化。

**解决方案：**
- `BeginTran` 方法添加 `IsolationLevel?` 参数
- 支持显式指定隔离级别（如 `ReadCommitted`、`Snapshot`）
- 配合数据库启用 `READ_COMMITTED_SNAPSHOT` 可大幅减少死锁

**代码变更：**
```csharp
// IUnitOfWorkManage.cs
void BeginTran(IsolationLevel? isolationLevel = null);

// UnitOfWorkManage.cs
public void BeginTran(IsolationLevel? isolationLevel = null)
{
    // ...
    if (isolationLevel.HasValue)
        dbClient.Ado.BeginTran(isolationLevel.Value);
    else
        dbClient.Ado.BeginTran();
}
```

**使用示例：**
```csharp
// 使用读已提交隔离级别
_unitOfWorkManage.BeginTran(IsolationLevel.ReadCommitted);

// 或使用快照隔离（需数据库支持）
_unitOfWorkManage.BeginTran(IsolationLevel.Snapshot);
```

### ✅ 4. 确保连接及时释放

**问题：** 事务未正常结束时连接可能泄漏，耗尽连接池。

**解决方案：**
- 实现 `IDisposable` 接口
- `Dispose` 方法中强制回滚未完成事务并关闭连接
- 清理 `AsyncLocal` 引用，避免内存泄漏

**代码变更：**
```csharp
public class UnitOfWorkManage : IUnitOfWorkManage, IDependencyRepository, IDisposable
{
    public void Dispose()
    {
        var dbClient = _asyncLocalClient.Value;
        if (dbClient != null)
        {
            try
            {
                var context = _currentTransactionContext.Value;
                
                // 强制回滚未完成事务
                if (context != null && context.Depth > 0)
                {
                    if (dbClient.Ado.Transaction != null)
                        dbClient.Ado.RollbackTran();
                }
                
                // 关闭并释放连接
                if (dbClient.Ado.Connection != null)
                {
                    if (dbClient.Ado.Connection.State == ConnectionState.Open)
                        dbClient.Ado.Connection.Close();
                    dbClient.Ado.Connection.Dispose();
                }
            }
            finally
            {
                _asyncLocalClient.Value = null;
                _currentTransactionContext.Value = null;
                _tranDepth.Value = 0;
            }
        }
    }
}
```

**使用建议：**
```csharp
// 推荐：使用 using 语句确保释放
using (var scope = serviceScopeFactory.CreateScope())
{
    var unitOfWork = scope.ServiceProvider.GetService<UnitOfWorkManage>();
    // ... 使用
}

// 或由 DI 容器自动释放（注册为 Scoped）
```

### ✅ 5. 设置事务超时时间

**问题：** 长事务持有锁时间过长，增加死锁概率。

**解决方案：**
- 定义 `DEFAULT_TRANSACTION_TIMEOUT = 60` 秒
- 在 `BeginTran` 中设置 `CommandTimeOut`
- 可结合 SQL Server 的 `SET LOCK_TIMEOUT` 使用

**代码变更：**
```csharp
private const int DEFAULT_TRANSACTION_TIMEOUT = 60;

public void BeginTran(IsolationLevel? isolationLevel = null)
{
    // ...
    if (context.Depth == 1)
    {
        dbClient.Ado.CommandTimeOut = DEFAULT_TRANSACTION_TIMEOUT;
        // ...
    }
}
```

### ✅ 6. 增强死锁日志记录

**问题：** 死锁日志缺少参数信息，不利于问题定位。

**解决方案：**
- 在 `SqlsugarSetup.OnError` 中记录 SQL 参数
- 记录更详细的死锁上下文信息

**代码变更：**
```csharp
// SqlsugarSetup.cs
if (e.InnerException is SqlException sqlEx && sqlEx.Number == 1205)
{
    var deadlockInfo = new
    {
        Time = DateTime.Now,
        StackTrace = e.StackTrace,
        Sql = errorsql,
        Parameters = e.Parametres?.Cast<SugarParameter>()
            ?.Select(p => $"{p.ParameterName}={p.Value}")?.ToArray()
    };
    
    logger.LogError("检测到数据库死锁：{DeadlockInfo}", 
        JsonConvert.SerializeObject(deadlockInfo), e);
}
```

## 二、性能优化

### 1. 并发检测开销极低
- 使用 `AsyncLocal<int>` 避免锁竞争
- `Interlocked` 原子操作，纳秒级性能
- 仅在检测到违规时才抛出异常

### 2. 重试机制仅针对瞬态故障
- 只对死锁错误（1205）进行重试
- 其他异常直接抛出，不影响正常错误处理
- 指数退避避免立即重试造成的资源争抢

### 3. 隔离级别灵活可选
- 默认保持原有行为（数据库默认隔离级别）
- 可按需指定更友好的隔离级别
- 配合数据库配置优化效果更佳

## 三、使用指南

### 1. 基本使用（保持不变）
```csharp
try
{
    _unitOfWorkManage.BeginTran();
    // ... 业务操作
    _unitOfWorkManage.CommitTran();
}
catch (Exception)
{
    _unitOfWorkManage.RollbackTran();
    throw;
}
```

### 2. 指定隔离级别
```csharp
_unitOfWorkManage.BeginTran(IsolationLevel.ReadCommitted);
```

### 3. 使用 using 确保释放
```csharp
using (var unitOfWork = _container.Resolve<UnitOfWorkManage>())
{
    unitOfWork.BeginTran();
    // ...
    unitOfWork.CommitTran();
}
```

### 4. 异步重试
```csharp
await _unitOfWorkManage.ExecuteWithRetryAsync(async () =>
{
    _unitOfWorkManage.BeginTran();
    await SomeBusinessOperation();
    _unitOfWorkManage.CommitTran();
});
```

## 四、注意事项

### ⚠️ 1. 事务内禁止并行操作
以下代码现在会抛出异常：
```csharp
_unitOfWorkManage.BeginTran();

// ❌ 错误：事务内并行执行
await Task.WhenAll(
    db.InsertAsync(obj1),
    db.InsertAsync(obj2)
);

// ✅ 正确：顺序执行
await db.InsertAsync(obj1);
await db.InsertAsync(obj2);

_unitOfWorkManage.CommitTran();
```

### ⚠️ 2. 需要独立事务时使用独立作用域
```csharp
// ✅ 正确：为每个并行分支创建独立事务
var tasks = items.Select(async item =>
{
    using (var scope = serviceScopeFactory.CreateScope())
    {
        var unitOfWork = scope.ServiceProvider.GetService<UnitOfWorkManage>();
        unitOfWork.BeginTran();
        await ProcessItem(item);
        unitOfWork.CommitTran();
    }
});
await Task.WhenAll(tasks);
```

### ⚠️ 3. 数据库配置建议
为获得最佳效果，建议在数据库层面启用：
```sql
-- 启用读已提交快照隔离（减少读写阻塞）
ALTER DATABASE YourDatabase SET READ_COMMITTED_SNAPSHOT ON;

-- 可选：设置锁超时时间（毫秒）
SET LOCK_TIMEOUT 30000; -- 30 秒
```

## 六、兼容性说明

### ✅ 向后兼容
- `BeginTran()` 无参调用仍然有效（使用默认隔离级别）
- 现有业务代码无需修改即可工作
- 重试机制透明，不影响正常流程

### 🔄 渐进式优化
- 可逐步在关键业务中使用隔离级别参数

- 可根据实际情况调整超时时间和重试次数

## 七、监控指标建议

建议在 `TransactionMetrics` 中添加以下监控：

1. **死锁重试次数**：统计单位时间内的死锁重试次数
2. **并发违规检测次数**：监控事务内并行操作的频率
3. **事务超时统计**：记录因超时失败的事务数量
4. **连接泄漏检测**：监控未正常释放的连接数

示例：
```csharp
public static class TransactionMetrics
{
    private static long _deadlockRetryCount = 0;
    private static long _concurrentViolationCount = 0;
    
    public static void RecordDeadlockRetry()
    {
        Interlocked.Increment(ref _deadlockRetryCount);
    }
    
    public static void RecordConcurrentViolation()
    {
        Interlocked.Increment(ref _concurrentViolationCount);
    }
}
```

## 八、总结与实施建议

本次优化从以下几个方面提升了事务管理体系的健壮性：

| 优化项 | 优先级 | 实施难度 | 效果 |
|--------|--------|----------|------|
| 死锁重试 | ⚠️ 高 | 低 | 大幅提升容错能力 |
| 连接释放 | ⚠️ 中 | 低 | 防止连接池耗尽 |
| 隔离级别 | ⚠️ 中 | 低 | 减少数据库死锁概率 |
| 事务超时 | ⚠️ 中 | 低 | 限制锁持有时间 |
| AOP 修复 | ⚠️ 高 | 低 | 确保日志和监控不丢失 |
| 并发约束 | ✅ 低 | 低 | 通过规范保证线程安全 |

**下一步建议：**
1. 在生产环境部署后密切监控死锁重试率
2. 对高频死锁的业务场景考虑引入队列或批处理
3. 定期分析死锁日志，优化相关 SQL 和索引
4. **加强代码审查，确保事务内不使用并行操作**
5. 评估是否需要引入分布式事务协调器（MSDTC）
