# 借出单反审核与反执行拆分实现说明

## 修改概述

以 `tb_ProdBorrowing`（借出单）为实例，将原来的反审核操作拆分为两个独立的方法：
- **AntiApprovalAsync**：仅状态回退（确认→新建）
- **AntiConfirmExecutionAsync**：库存回滚 + 状态回退（完结→确认）

## 文件修改清单

### 1. BaseControllerGeneric.cs
**位置**：`RUINORERP.Business\BaseControllerGeneric.cs`

**修改内容**：
- 更新 `AntiApprovalAsync` 方法注释，明确说明仅处理状态回退
- 新增 `AntiConfirmExecutionAsync` 虚方法定义，作为反执行的基类接口

```csharp
/// <summary>
/// 反审核（仅状态回退）：将单据状态从确认回退到新建
/// 不包含库存操作，库存操作由 AntiConfirmExecutionAsync 处理
/// </summary>
public async virtual Task<ReturnResults<T>> AntiApprovalAsync(T entity)

/// <summary>
/// 反执行（库存回滚）：回滚执行时的库存操作
/// 将库存操作反向变动，并回退状态到确认
/// </summary>
public async virtual Task<ReturnResults<T>> AntiConfirmExecutionAsync(T entity)
```

### 2. tb_ProdBorrowingControllerPartial.cs
**位置**：`RUINORERP.Business\tb_ProdBorrowingControllerPartial.cs`

**修改内容**：
- 重写 `AntiApprovalAsync`：实现纯状态回退逻辑
- 重写 `AntiConfirmExecutionAsync`：实现库存回滚逻辑

## 详细实现

### 一、反审核方法（AntiApprovalAsync）

#### 职责
仅负责将单据状态从**确认**回退到**新建**，不涉及任何库存操作。

#### 业务逻辑
```csharp
public async override Task<ReturnResults<T>> AntiApprovalAsync(T ObjectEntity)
{
    // 1. 验证单据合法性
    if (entity == null) return Error("单据实体为空");
    
    // 2. 检查状态（必须是确认状态）
    if (entity.DataStatus != (int)DataStatus.确认) 
        return Error("只有已审核的单据才能反审核");
    
    // 3. 检查是否已执行（完结状态不能直接反审核）
    if (entity.DataStatus == (int)DataStatus.完结)
        return Error("此单据已执行，请先反执行再反审核");
    
    // 4. 开启事务
    _unitOfWorkManage.BeginTran();
    
    // 5. 更新状态（仅状态字段）
    entity.DataStatus = (int)DataStatus.新建;
    entity.ApprovalResults = false;
    entity.ApprovalStatus = (int)ApprovalStatus.未审核;
    
    // 6. 提交事务
    _unitOfWorkManage.CommitTran();
    
    return Success(entity);
}
```

#### 关键点
- ✅ 只更新状态相关字段
- ✅ 不触碰库存数据
- ✅ 检查是否已执行（完结状态）
- ✅ 事务保证数据一致性

### 二、反执行方法（AntiConfirmExecutionAsync）

#### 职责
负责回滚执行时的所有库存操作，并将状态从**完结**回退到**确认**。

#### 业务逻辑
```csharp
public async override Task<ReturnResults<T>> AntiConfirmExecutionAsync(T ObjectEntity)
{
    // 1. 验证单据合法性
    if (entity == null) return Error("单据实体为空");
    
    // 2. 检查状态（必须是完结状态）
    if (entity.DataStatus != (int)DataStatus.完结)
        return Error("只有已执行的单据才能反执行");
    
    // 3. 权限检查
    if (!HasPermission("ReverseExecute"))
        return Error("没有反执行权限");
    
    // 4. 【死锁优化】事务外批量预加载库存
    var invDict2 = PreloadInventory(entity.tb_ProdBorrowingDetails);
    
    // 5. 开启事务
    _unitOfWorkManage.BeginTran();
    
    // 6. 遍历明细，反向操作库存
    foreach (var detail in entity.tb_ProdBorrowingDetails)
    {
        // 6.1 获取预加载的库存
        var inv = invDict2[(detail.ProdDetailID, detail.Location_ID)];
        
        // 6.2 反向更新库存（增加）
        inv.Quantity = inv.Quantity + detail.Qty;
        
        // 6.3 创建反向库存流水
        CreateReverseTransaction(detail, inv);
    }
    
    // 7. 批量更新库存
    await BatchUpdateInventory(invUpdateList);
    
    // 8. 记录库存流水
    await BatchRecordTransactions(transactionList);
    
    // 9. 回退状态到确认
    entity.DataStatus = (int)DataStatus.确认;
    entity.CloseCaseOpinions = "反执行";
    await UpdateEntityStatus(entity);
    
    // 10. 提交事务
    _unitOfWorkManage.CommitTran();
    
    return Success(entity);
}
```

#### 关键点
- ✅ 需要特殊权限控制
- ✅ 死锁优化：事务外预加载库存
- ✅ 反向库存操作（增加库存）
- ✅ 创建反向库存流水记录
- ✅ 回退状态到确认
- ✅ 事务保证原子性

## 操作流程对比

### 原流程（混合模式）
```
反审核 = 状态回退 + 库存冲减
```

### 新流程（分离模式）
```
场景 1：未执行的单据
  反审核 → 直接调用 AntiApprovalAsync
  └─ 确认 → 新建（仅状态变更）

场景 2：已执行的单据
  Step 1: 反执行 → 调用 AntiConfirmExecutionAsync
  └─ 完结 → 确认（库存回滚 + 状态回退）
  
  Step 2: 反审核 → 调用 AntiApprovalAsync
  └─ 确认 → 新建（仅状态变更）
```

## 状态流转图

```
草稿 → 新建 → 确认 → 完结
         ↑    │     │  │
         │    │反审  │  │反结
         │    │     ↓  ↓案
         │    └── 反执行 ←┘
         │          │
         └──────────┘
```

## 使用示例

### UI 层调用代码

#### 反审核按钮
```csharp
private async void btnReverseReview_Click(object sender, EventArgs e)
{
    var entity = GetCurrentEntity();
    var currentStatus = entity.GetCurrentStatus();
    
    // 检查是否已执行
    if (currentStatus == DataStatus.完结)
    {
        MessageBox.Show(
            "⚠️ 此单据已执行，无法直接反审核！\n\n" +
            "请先执行【反执行】操作回滚库存，然后再反审核。");
        return;
    }
    
    // 确认状态可以直接反审核
    if (currentStatus != DataStatus.确认)
    {
        MessageBox.Show("只有已审核的单据才能反审核！");
        return;
    }
    
    // 二次确认
    if (MessageBox.Show("确定要反审核吗？", "确认", 
        MessageBoxButtons.OKCancel) != DialogResult.OK)
        return;
    
    try
    {
        // 调用反审核方法（仅状态回退）
        var result = await controller.AntiApprovalAsync(entity);
        
        if (result.Succeeded)
        {
            MessageBox.Show("反审核成功！");
            RefreshUI();
        }
        else
        {
            MessageBox.Show($"反审核失败：{result.ErrorMsg}");
        }
    }
    catch (Exception ex)
    {
        MessageBox.Show($"反审核失败：{ex.Message}");
    }
}
```

#### 反执行按钮
```csharp
private async void btnReverseExecute_Click(object sender, EventArgs e)
{
    var entity = GetCurrentEntity();
    var currentStatus = entity.GetCurrentStatus();
    
    // 只有完结状态可以反执行
    if (currentStatus != DataStatus.完结)
    {
        MessageBox.Show("只有已执行的单据才能反执行！");
        return;
    }
    
    // 权限检查
    if (!HasPermission("ReverseExecute"))
    {
        MessageBox.Show("您没有反执行的权限！");
        return;
    }
    
    // 警告确认
    if (MessageBox.Show(
        "⚠️ 警告：反执行将回滚所有库存操作！\n\n确定要继续吗？",
        "确认",
        MessageBoxButtons.OKCancel,
        MessageBoxIcon.Warning) != DialogResult.OK)
        return;
    
    try
    {
        // 调用反执行方法（库存回滚 + 状态回退）
        var result = await controller.AntiConfirmExecutionAsync(entity);
        
        if (result.Succeeded)
        {
            MessageBox.Show("反执行成功！\n单据状态已回退到【已审核】。");
            RefreshUI();
        }
        else
        {
            MessageBox.Show($"反执行失败：{result.ErrorMsg}");
        }
    }
    catch (Exception ex)
    {
        MessageBox.Show($"反执行失败：{ex.Message}");
    }
}
```

## 总结

通过将反审核操作拆分为两个独立方法，实现了：

1. ✅ **职责清晰**：状态管理 vs 业务操作
2. ✅ **灵活性高**：可以根据实际情况选择调用
3. ✅ **易于维护**：每个方法只做一件事
4. ✅ **数据一致**：事务保证原子性
5. ✅ **性能优化**：死锁预防措施

这种设计模式可以推广到其他类似单据（如销售订单、采购订单等）。
