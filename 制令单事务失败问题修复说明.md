# 制令单事务失败问题修复说明

## 问题描述

制令单在**保存、提交、审核**操作时频繁出现事务失败,错误信息如下:

```
无法执行该事务操作,因为有挂起请求正在此事务上运行。
类型: System.Data.SqlClient.SqlException
堆栈: 在 SqlSugar.AdoProvider.CommitTran()
     在 RUINORERP.Repository.UnitOfWorks.UnitOfWorkManage.CommitTranInternal()
```

同时伴随以下警告日志:
- `[Transaction-Unknown] 强制回滚:事务连接已关闭或无效`
- `[CommitTran] 提交请求但无事务上下文`
- `[Transaction-xxx] 事务连接已关闭或无效,无法提交`

## 根本原因分析

### 1. 核心问题:在事务内创建了独立的DbHelper实例

在 `tb_ManufacturingOrderControllerPartial.cs` 的三个关键方法中:
- `BatchCloseCaseAsync` (批量结案)
- `AntiApprovalAsync` (反审核)
- `ApprovalAsync` (审核)

都存在以下**致命模式**:

```csharp
// ❌ 错误做法:在事务内创建新的DbHelper
DbHelper<tb_Inventory> dbHelper = _appContext.GetRequiredService<DbHelper<tb_Inventory>>();
var InvMainCounter = await dbHelper.BaseDefaultAddElseUpdateAsync(invMain);
```

**问题分析**:
1. `DbHelper<T>` 是通过DI容器获取的**新实例**
2. 这个新实例会创建**独立的数据库连接**,不在当前事务上下文中
3. 当异步操作还未完成时就尝试 `CommitTran()`,导致"挂起请求"错误
4. 多个独立连接之间产生**资源竞争和死锁**

### 2. 次要问题:不必要的外部Controller引用

```csharp
// ❌ 不必要的依赖注入
tb_InventoryController<tb_Inventory> ctrinv = _appContext.GetRequiredService<tb_InventoryController<tb_Inventory>>();
```

虽然这行代码没有直接使用,但它可能导致:
- 额外的对象创建开销
- 潜在的循环依赖风险
- 混淆代码意图

### 3. 逻辑错误:变量判断错误

在 `AntiApprovalAsync` 第319行:
```csharp
// ❌ 错误:应该判断 inv 而不是 invMain
if (invMain != null)
{
    inv.NotOutQty -= item.ShouldSendQty.ToInt();
}
```

这会导致即使 `inv` 为 null 也会执行更新,引发空引用异常。

## 修复方案

### 修复原则

**所有数据库操作必须使用当前事务的 `_unitOfWorkManage.GetDbClient()`**,确保:
1. 共享同一个数据库连接
2. 在同一个事务上下文中执行
3. 避免连接泄漏和事务冲突

### 具体修改

#### 1. BatchCloseCaseAsync 方法

**修改前**:
```csharp
DbHelper<tb_Inventory> dbHelper = _appContext.GetRequiredService<DbHelper<tb_Inventory>>();
var InvMainCounter = await dbHelper.BaseDefaultAddElseUpdateAsync(invUpdateList);
```

**修改后**:
```csharp
// ✅ 使用当前事务的DbClient批量更新
int InvMainCounter = 0;
if (invUpdateList.Count > 0)
{
    InvMainCounter = await _unitOfWorkManage.GetDbClient()
        .Updateable(invUpdateList)
        .ExecuteCommandAsync();
}

if (InvMainCounter == 0 && invUpdateList.Count > 0)
{
    _logger.Debug($"{entitys[m].MONO}更新库存结果为0行,请检查数据!");
}
```

#### 2. AntiApprovalAsync 方法

**修改点1**: 移除不必要的Controller引用
```csharp
// ❌ 删除
// tb_InventoryController<tb_Inventory> ctrinv = _appContext.GetRequiredService<tb_InventoryController<tb_Inventory>>();
```

**修改点2**: 主表库存更新改用Storageable
```csharp
// ✅ 使用当前事务的DbClient直接操作
var InvMainCounter = await _unitOfWorkManage.GetDbClient()
    .Storageable(invMain)
    .ToStorage()
    .AsInsertable.ExecuteReturnSnowflakeIdAsync(); // 不存在插入
    
if (InvMainCounter == 0)
{
    // 如果插入返回0,尝试更新
    InvMainCounter = await _unitOfWorkManage.GetDbClient()
        .Updateable(invMain)
        .ExecuteCommandAsync();
}
```

**修改点3**: 明细库存更新改用Updateable
```csharp
// ✅ 使用当前事务的DbClient批量更新
int InvUpdateCounter = 0;
if (invUpdateList.Count > 0)
{
    InvUpdateCounter = await _unitOfWorkManage.GetDbClient()
        .Updateable(invUpdateList)
        .ExecuteCommandAsync();
}
```

**修改点4**: 修复变量判断错误
```csharp
// ✅ 正确判断 inv 而不是 invMain
if (inv != null)
{
    inv.NotOutQty -= item.ShouldSendQty.ToInt();
    BusinessHelper.Instance.EditEntity(inv);
    invUpdateList.Add(inv);
}
else
{
    _unitOfWorkManage.RollbackTran();
    throw new Exception($"ProdDetailID:{item.ProdDetailID}, LocationID:{item.Location_ID} 不存在库存信息");
}
```

#### 3. ApprovalAsync 方法

采用与 `AntiApprovalAsync` 相同的修复策略:
1. 移除不必要的Controller引用
2. 主表库存使用 Storageable + Updateable 组合
3. 明细库存使用 Updateable 批量更新
4. 增加重复检测失败时的回滚

## 技术要点

### 1. SqlSugar 的正确用法

**在事务中操作数据的标准模式**:

```csharp
// ✅ 正确:使用当前事务的DbClient
_unitOfWorkManage.BeginTran();

// 查询
var data = await _unitOfWorkManage.GetDbClient()
    .Queryable<T>()
    .Where(...)
    .ToListAsync();

// 插入/更新(自动识别)
var count = await _unitOfWorkManage.GetDbClient()
    .Storageable(entity)
    .ToStorage()
    .AsInsertable.ExecuteReturnSnowflakeIdAsync();

// 批量更新
var count = await _unitOfWorkManage.GetDbClient()
    .Updateable(list)
    .ExecuteCommandAsync();

_unitOfWorkManage.CommitTran();
```

**❌ 绝对禁止**:
```csharp
// 不要在事务内创建新的DbHelper或Controller
var dbHelper = _appContext.GetRequiredService<DbHelper<T>>();
await dbHelper.SomeMethodAsync(); // 这会创建新连接!
```

### 2. Storageable vs Updateable

| 方法 | 适用场景 | 特点 |
|------|---------|------|
| `Storageable` | 不确定记录是否存在,需要"有则更新,无则插入" | 自动判断,适合Upsert场景 |
| `Updateable` | 确定记录已存在,只需更新 | 性能更好,只生成UPDATE语句 |
| `Insertable` | 确定是新记录,只需插入 | 性能最好,只生成INSERT语句 |

本次修复中:
- **主表库存**: 使用 `Storageable` (可能新增也可能更新)
- **明细库存**: 使用 `Updateable` (预加载时已确保存在)

### 3. 事务嵌套的最佳实践

```csharp
// ✅ 推荐:扁平化事务,减少嵌套深度
public async Task<bool> ProcessOrder(Order order)
{
    _unitOfWorkManage.BeginTran();
    try
    {
        // 所有数据库操作都用 _unitOfWorkManage.GetDbClient()
        await UpdateInventory(order);
        await UpdateOrderStatus(order);
        
        _unitOfWorkManage.CommitTran();
        return true;
    }
    catch
    {
        _unitOfWorkManage.RollbackTran();
        throw;
    }
}

// ❌ 避免:深层嵌套的事务调用
public async Task<bool> ProcessOrder(Order order)
{
    _unitOfWorkManage.BeginTran();
    await InventoryService.Update(order);  // 内部又BeginTran
    await OrderService.Update(order);      // 内部又BeginTran
    _unitOfWorkManage.CommitTran();
}
```

## 后续优化建议

### 1. 代码规范
制定团队开发规范:
- **禁止**在事务方法内通过DI获取其他Controller/Service
- **必须**使用 `_unitOfWorkManage.GetDbClient()` 进行所有数据库操作
- 复杂业务逻辑抽取到私有方法,但仍使用同一DbClient

### 2. 架构改进
考虑引入**领域服务层**:

```csharp
public class ManufacturingOrderDomainService

    {
        // 所有操作都使用 _unitOfWork.GetDbClient()
        var db = _unitOfWork.GetDbClient();
        await db.Updateable(...).ExecuteCommandAsync();
    }
}
```

## 总结

本次修复的核心是**统一事务上下文**,确保所有数据库操作都在同一个连接和事务中执行。通过移除独立的DbHelper实例,从根本上解决了"挂起请求"导致的提交失败问题。

**关键教训**:
1. 事务内的所有数据库操作必须共享同一个连接
2. 避免在事务方法中通过DI获取新的数据访问组件
3. 使用SqlSugar的Storageable/Updateable代替自定义DbHelper
4. 保持事务扁平化,减少嵌套深度
