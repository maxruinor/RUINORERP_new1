# 性能数据上报日志优化说明

## 问题描述

性能数据上报功能在Token未就绪或网络重连时会产生大量ERROR级别的日志，包括：
- `发送数据包时发生错误: CommandId=0x0012:PerformanceDataUpload, 发送请求失败: 没有合法授权令牌`
- `发送带响应命令失败: CommandId=0x0012:PerformanceDataUpload, RequestId=..., Error=...`
- `未处理的异常:带响应命令发送失败: CommandId=PerformanceDataUpload, ...`
- `性能数据上报失败:`

这些日志对于一个辅助性功能来说过于繁琐，影响了日志的可读性。

## 根本原因分析

### 1. Token生命周期问题
- 性能监控服务在登录成功后启动（`FrmLogin.cs:1148`）
- 在网络重连时，虽然会重置并重启性能监控（`MainForm.cs:446`），但此时可能Token尚未完全恢复
- 导致性能数据上报时Token检查失败

### 2. 错误处理链过长
- `ClientPerformanceMonitorService.OnPerformanceDataUpload` (async void) 
  → `SendPerformanceDataAsync` 
  → `ClientCommunicationService.SendCommandWithResponseAsync` 
  → `SendRequestAsync` 
  → `SendPacketCoreAsync`
  → 抛出异常
  → `SendCommandWithResponseAsync`捕获并调用`_clientEventManager.OnErrorOccurred`
  → `ClientEventManager.LogException`记录ERROR日志

### 3. 日志级别不合理
- 对于性能数据上报这种辅助功能，所有失败都记录为ERROR级别
- 每次失败都会触发多次ERROR日志输出

## 优化方案

### 1. 提前检查Token有效性（ClientPerformanceMonitorService.cs）

在`OnPerformanceDataUpload`方法中增加前置检查：
```csharp
// ✅ 提前检查：网络连接状态
if (_communicationService == null || _communicationService.ConnectionManager?.IsConnected != true)
{
    _logger?.LogDebug("网络连接不可用，跳过性能数据上报");
    return;
}

// ✅ 提前检查：Token有效性（避免进入发送流程后才失败）
var appContext = MainForm.Instance?.AppContext;
if (appContext == null || string.IsNullOrEmpty(appContext.SessionId))
{
    _logger?.LogDebug("会话未初始化或Token无效，跳过性能数据上报");
    return;
}
```

**优势**：在进入发送流程前就拦截无效请求，避免后续复杂的错误处理。

### 2. 降低日志级别（ClientPerformanceMonitorService.cs）

#### OnPerformanceDataUpload方法
```csharp
catch (Exception ex)
{
    // ✅ async void方法的最后一道防线
    if (ex.Message.Contains("授权令牌") || ex.Message.Contains("Token"))
    {
        _logger?.LogDebug("Token未就绪，跳过本次性能数据上报: {Message}", ex.Message);
        return;
    }
    
    // ✅ 其他错误也降低为Warning级别，因为这是辅助功能
    _logger?.LogWarning(ex, "性能数据上报异常（非关键错误，已静默处理）");
}
```

#### SendPerformanceDataAsync方法
- 重试次数从2次降低到1次
- 超时时间从20秒降低到15秒
- 所有失败日志从Error/Warning降低到Debug级别
- 最终失败不抛出异常，静默返回

```csharp
const int maxRetries = 1; // ✅ 辅助功能，只重试1次

// 失败时只记录Debug日志
_logger?.LogDebug($"性能数据上报失败: {response?.ErrorMessage}");

// 超时也只记录Debug日志
_logger?.LogDebug(ex, $"性能数据上报超时 (尝试 {retryCount}/{maxRetries + 1})");

// ✅ 不抛出异常，静默失败
return;
```

### 3. 添加非关键命令标记（ClientCommunicationService.cs）

在`SendCommandWithResponseAsync`方法中添加`isNonCriticalCommand`参数：

```csharp
public async Task<TResponse> SendCommandWithResponseAsync<TResponse>(
    CommandId commandId,
    IRequest request,
    CancellationToken ct = default,
    int timeoutMs = 30000,
    bool isNonCriticalCommand = false)  // ✅ 新增参数
{
    // ✅ 对于非关键命令，减少重试次数
    const int maxRetriesNormal = 2;
    const int maxRetriesNonCritical = 0; // 非关键命令不重试
    int maxRetries = isNonCriticalCommand ? maxRetriesNonCritical : maxRetriesNormal;
    
    // ...
    
    catch (Exception ex)
    {
        // ✅ 对于非关键命令，降低日志级别且不触发错误事件
        if (isNonCriticalCommand)
        {
            _logger?.LogDebug(ex, "非关键命令发送失败: CommandId={CommandId}, ...", ...);
        }
        else
        {
            // 关键命令：记录ERROR日志并触发错误事件
            _logger?.LogError(ex, "发送带响应命令失败: ...", ...);
            _clientEventManager.OnErrorOccurred(new Exception(...));
        }
    }
}
```

**关键改进**：
- 非关键命令不触发`_clientEventManager.OnErrorOccurred`，避免产生ERROR日志
- 非关键命令不重试（maxRetries=0），快速失败
- 日志级别从Error降低到Debug

### 4. 调用时标记为非关键命令（ClientPerformanceMonitorService.cs）

```csharp
var response = await _communicationService.SendCommandWithResponseAsync<PerformanceDataUploadResponse>(
    SystemCommands.PerformanceDataUpload,
    request,
    CancellationToken.None,
    15000,
    true); // ✅ 标记为非关键命令
```

## 设计原则

### 1. 分层防御
- **第一层**：在`OnPerformanceDataUpload`中提前检查网络和Token状态
- **第二层**：在`SendPerformanceDataAsync`中捕获所有异常并静默处理
- **第三层**：在`SendCommandWithResponseAsync`中针对非关键命令降低日志级别

### 2. 优雅降级
- Token未就绪 → 跳过本次上报，等待下次
- 网络断开 → 暂停上报，重连后恢复
- 发送失败 → 静默失败，不影响主业务流程

### 3. 日志分级
- **ERROR**：关键业务失败（如登录、下单、支付）
- **WARN**：重要但可恢复的问题（如配置同步失败）
- **DEBUG**：辅助功能的正常行为（如性能数据上报失败）

## 兼容性说明

- 添加了默认参数`isNonCriticalCommand = false`，现有调用无需修改
- 只有显式传递`true`时才会启用非关键命令模式
- 不影响其他关键业务的错误处理和日志输出
