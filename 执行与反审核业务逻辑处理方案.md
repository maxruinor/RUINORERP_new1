# 执行与反审核的业务逻辑处理方案

## 问题背景

原流程中：
- **审核** = 状态变化 + 库存操作
- **反审核** = 状态回退 + 库存冲减

新流程拆分后：
- **审核** = 仅状态变化（确认）
- **执行** = 实际库存操作
- **反审核** = 状态回退 + ？？？

## 核心设计原则

### 1. 状态与业务分离原则

| 操作 | 状态变化 | 业务操作 | 是否可逆 |
|------|---------|---------|---------|
| 审核 | 新建→确认 | 无 | 是（反审核） |
| 执行 | 确认→完结 | 库存操作 | 需要支持反向 |
| 结案 | 确认→完结 | 无（强制终止） | 是（反结案） |

### 2. 反审核的处理策略

根据单据是否已执行，采用不同策略：

#### 场景 A：已审核但未执行
```
状态：确认
库存：未变动
反审核：直接回退状态到新建
```

#### 场景 B：已审核且已执行
```
状态：完结
库存：已变动
反审核：
  方案 1: 不允许反审核，必须先"反执行"
  方案 2: 自动触发"反执行"+反审核
```

## 推荐方案：分级控制

### 方案说明

1. **确认状态（未执行）**：允许直接反审核
2. **完结状态（已执行）**：不允许直接反审核，需要先反执行

### 状态流转图

```
                    ┌─────────────────────────────────────┐
                    │                                     │
                    ▼                                     │
草稿 → 新建 → 确认 → 完结                              │
         ▲        │    │                                  │
         │        │    │                                  │
         │ 反审   │反审│ 反结案                            │
         │        │    │                                  │
         │        ▼    ▼                                  │
         │      不允许反审核，需先反执行                   │
         │             │                                  │
         │             ▼                                  │
         │         反执行（库存回滚）                      │
         │             │                                  │
         └─────────────┘（回到确认状态）
```

## 详细实现方案

### 1. 反审核逻辑

```csharp
private async void toolStripBtnReverseReview_Click(object sender, EventArgs e)
{
    var entity = GetCurrentEntity();
    if (entity == null) return;
    
    var currentStatus = entity.GetCurrentStatus();
    
    // 检查是否已执行
    if (currentStatus == DataStatus.完结)
    {
        // 已执行的单据，提示需要先反执行
        MessageBox.Show(
            "此单据已执行，无法直接反审核！\n\n" +
            "请先执行【反执行】操作回滚库存，然后再反审核。",
            "操作提示",
            MessageBoxButtons.OK,
            MessageBoxIcon.Warning);
        return;
    }
    
    // 确认状态（未执行），可以直接反审核
    if (currentStatus != DataStatus.确认)
    {
        MessageBox.Show("只有已审核的单据才能反审核！");
        return;
    }
    
    // 二次确认
    if (MessageBox.Show(
        "确定要反审核此单据吗？\n\n" +
        "反审核后单据将回到【已提交】状态。",
        "反审核确认",
        MessageBoxButtons.OKCancel,
        MessageBoxIcon.Question) != DialogResult.OK)
    {
        return;
    }
    
    try
    {
        // 执行反审核（只改变状态）
        var result = await stateManager.SetStatusByActionAsync(entity, MenuItemEnums.反审);
        
        if (result.IsSuccess)
        {
            await SaveEntityAsync(entity);
            MessageBox.Show("反审核成功！");
            RefreshUI();
        }
        else
        {
            MessageBox.Show($"反审核失败：{result.ErrorMessage}");
        }
    }
    catch (Exception ex)
    {
        MessageBox.Show($"反审核失败：{ex.Message}");
    }
}
```

### 2. 反执行逻辑（新增）

```csharp
private async void toolStripBtnReverseExecute_Click(object sender, EventArgs e)
{
    var entity = GetCurrentEntity();
    if (entity == null) return;
    
    var currentStatus = entity.GetCurrentStatus();
    
    // 只有完结状态的单据才能反执行
    if (currentStatus != DataStatus.完结)
    {
        MessageBox.Show("只有已执行的单据才能反执行！");
        return;
    }
    
    // 权限检查（反执行是敏感操作）
    if (!HasPermission("ReverseExecute"))
    {
        MessageBox.Show("您没有反执行的权限！");
        return;
    }
    
    // 二次确认
    if (MessageBox.Show(
        "⚠️ 警告：反执行将回滚所有库存操作！\n\n" +
        "确定要继续吗？",
        "反执行确认",
        MessageBoxButtons.OKCancel,
        MessageBoxIcon.Warning) != DialogResult.OK)
    {
        return;
    }
    
    try
    {
        // 1. 先执行库存回滚操作
        await ReverseInventoryOperationAsync(entity);
        
        // 2. 库存回滚成功后，回退状态到确认
        var result = await stateManager.SetBusinessStatusAsync(entity, DataStatus.确认);
        
        if (result.IsSuccess)
        {
            await SaveEntityAsync(entity);
            MessageBox.Show("反执行成功！\n\n单据状态已回退到【已审核】。");
            RefreshUI();
        }
        else
        {
            MessageBox.Show($"状态回退失败：{result.ErrorMessage}");
        }
    }
    catch (Exception ex)
    {
        // 库存回滚失败，状态不会改变
        MessageBox.Show($"反执行失败：{ex.Message}\n\n" +
                       $"单据状态仍为【完结】。");
        
        _logger.Error(ex, $"反执行单据 {entity.PrimaryKeyID} 失败");
    }
}
```

### 3. 状态转换规则更新

```csharp
private void InitializeDataStatusTransitionRules()
{
    var statusType = typeof(DataStatus);
    _stateTransitionRules[statusType] = new Dictionary<object, List<object>>
    {
        // 草稿态
        [DataStatus.草稿] = new List<object> { DataStatus.新建，DataStatus.作废 },
        
        // 新建态
        [DataStatus.新建] = new List<object> { DataStatus.草稿，DataStatus.确认，DataStatus.作废 },
        
        // 确认态（已审核）
        // 可以反审核（到新建）、可以执行（到完结）、可以结案（到完结）
        [DataStatus.确认] = new List<object> { 
            DataStatus.新建，    // 反审核
            DataStatus.完结，    // 执行或结案
            DataStatus.作废      
        },
        
        // 完结态（已执行或已结案）
        // 可以反结案（到确认）、可以反执行（到确认）
        [DataStatus.完结] = new List<object> { 
            DataStatus.确认      // 反结案或反执行
        },
        
        // 作废态
        [DataStatus.作废] = new List<object> { }
    };
}
```

        // ... 其他
    }
    
    return currentStatus;

## 权限控制建议

### 反执行权限

```csharp
/// <summary>
/// 检查是否有反执行权限
/// </summary>
private bool HasPermission(string permissionCode)
{
    // 从用户角色或权限系统检查
    var user = CurrentUser;
    return user.Roles.Any(r => r.Permissions.Contains(permissionCode));
}

// 使用示例
if (!HasPermission("ReverseExecute"))
{
    MessageBox.Show("您没有反执行的权限！");
    return;
}
```

## 总结

### 反审核处理策略

| 场景 | 状态 | 库存 | 反审核处理 |
|------|------|------|-----------|
| 未执行 | 确认 | 未变动 | 直接反审核 |
| 已执行 | 完结 | 已变动 | 先反执行，再反审核 |

### 关键要点

1. ✅ **状态与业务分离**：审核只变状态，执行才动库存
2. ✅ **分级控制**：根据是否已执行采用不同策略
3. ✅ **反执行权限**：需要严格控制
4. ✅ **审计日志**：记录所有关键操作
5. ✅ **事务保证**：确保数据一致性

这样的设计既保持了清晰的职责分离，又确保了业务的可追溯性和数据的一致性。
