# 执行按钮状态管理优化说明

## 业务需求分析

根据实际业务需求，"执行"按钮用于审核单据后的最后一步实际操作（一般是修改库存）。

### 核心业务逻辑

1. **执行操作**：
   - 执行成功 → 状态变为**完结**
   - 执行失败 → 状态保持**确认**（回滚到执行前）
   
2. **结案操作**：
   - 对采购/销售订单的强制终止
   - 状态也变为**完结**

3. **不需要"已执行"中间状态**

## 优化内容

### 1. DataStatus 枚举保持不变

```csharp
草稿 = 1
新建 = 2
确认 = 4
完结 = 8      // 执行成功或结案后状态
作废 = 16
```

### 2. 状态流转图

```
┌─────────┐   提交   ┌─────────┐   审核   ┌─────────┐   执行/结案   ┌─────────┐
│  草稿   │ ───────> │  新建   │ ───────> │  确认   │ ────────────> │  完结   │
│ (Draft) │          │ (New)   │          │(Confirmed)│              │ (Close) │
└─────────┘          └─────────┘          └─────────┘              └─────────┘
                         │                    │                          │
                         │ 反审               │ 反审                      │ 反结案
                         │                   ↓                          ↓
                         └──────────────>  ┌─────────┐              ┌─────────┐
                                           │  新建   │              │  确认   │
                                           └─────────┘              └─────────┘
```

### 3. 执行和结案的区别

| 操作 | 业务含义 | 源状态 | 目标状态 | 说明 |
|------|---------|--------|---------|------|
| 执行 | 完成实际业务操作（如修改库存） | 确认 | 完结 | 执行业务逻辑，成功后变完结，失败保持确认 |
| 结案 | 强制终止单据 | 确认 | 完结 | 无需执行业务操作，直接变更状态 |

### 4. 关键实现策略

#### 执行操作的最佳实践

```csharp
private async void toolStripButtonExecute_Click(object sender, EventArgs e)
{
    var entity = GetCurrentEntity();
    if (entity == null) return;
    
    // 验证当前状态
    var currentStatus = entity.GetCurrentStatus();
    if (currentStatus != DataStatus.确认)
    {
        MessageBox.Show("只有已审核的单据才能执行！");
        return;
    }
    
    try
    {
        // 【重要】1. 先执行业务操作（例如：修改库存）
        // 如果失败会抛出异常，不会执行后面的状态更新
        await ExecuteInventoryOperationAsync(entity);
        
        // 2. 业务操作成功后，更新状态为"完结"
        var result = await stateManager.SetStatusByActionAsync(entity, MenuItemEnums.执行);
        
        if (result.IsSuccess)
        {
            // 3. 保存实体
            await SaveEntityAsync(entity);
            
            MessageBox.Show("执行成功，单据已完结！");
            RefreshUI();
        }
        else
        {
            MessageBox.Show($"状态更新失败：{result.ErrorMessage}");
        }
    }
    catch (Exception ex)
    {
        // 【重要】业务操作失败时的处理
        // 状态不会改变，仍为"确认"
        MessageBox.Show($"执行失败：{ex.Message}\n\n单据状态已回滚到执行前（已审核状态）");
        
        // 可选：记录日志
        _logger.Error(ex, $"执行单据 {entity.PrimaryKeyID} 失败");
    }
}
```

#### 结案操作的实现

```csharp
private async void toolStripButtonCaseClosed_Click(object sender, EventArgs e)
{
    var entity = GetCurrentEntity();
    if (entity == null) return;
    
    var currentStatus = entity.GetCurrentStatus();
    if (currentStatus != DataStatus.确认)
    {
        MessageBox.Show("只有已审核的单据才能结案！");
        return;
    }
    
    // 二次确认（关键操作）
    if (MessageBox.Show(
        "确定要结案此单据吗？\n\n结案后将无法再修改或执行此单据。",
        "结案确认",
        MessageBoxButtons.OKCancel,
        MessageBoxIcon.Warning) != DialogResult.OK)
    {
        return;
    }
    
    try
    {
        // 执行结案操作
        var result = await stateManager.SetStatusByActionAsync(entity, MenuItemEnums.结案);
        
        if (result.IsSuccess)
        {
            await SaveEntityAsync(entity);
            MessageBox.Show("结案成功！");
            RefreshUI();
        }
        else
        {
            MessageBox.Show($"结案失败：{result.ErrorMessage}");
        }
    }
    catch (Exception ex)
    {
        MessageBox.Show($"结案失败：{ex.Message}");
    }
}
```

## 事务控制建议

### 使用 TransactionScope

```csharp
using (var scope = new TransactionScope(TransactionScopeAsyncFlowOption.Enabled))
{
    try
    {
        // 1. 执行业务操作
        await ExecuteInventoryOperationAsync(entity);
        
        // 2. 更新状态
        var result = await stateManager.SetStatusByActionAsync(entity, MenuItemEnums.执行);
        
        // 3. 保存数据
        await SaveEntityAsync(entity);
        
        // 4. 提交事务
        scope.Complete();
        
        MessageBox.Show("执行成功！");
    }
    catch (Exception ex)
    {
        // 事务会自动回滚
        MessageBox.Show($"执行失败：{ex.Message}");
    }
}
```

## 注意事项

1. **业务操作优先**
   - 必须先执行业务操作
   - 业务操作成功后才能更新状态

2. **异常处理**
   - 业务操作失败时抛出异常
   - 状态不会改变，保持为"确认"

3. **事务控制**
   - 建议使用 TransactionScope
   - 确保业务操作和状态变更的原子性

4. **权限控制**
   - 执行和结案都是关键操作
   - 需要适当的权限验证

5. **二次确认**
   - 结案操作建议二次确认
   - 执行操作可根据业务需要决定是否确认

## 测试建议

### 单元测试

```csharp
[Test]
public void Execute_WhenBusinessOperationSucceed_ShouldChangeStatusToClose()
{
    // Arrange
    var entity = new PurchaseOrder { DataStatus = DataStatus.确认 };
    MockInventoryService.Setup(x => x.Execute(It.IsAny<PurchaseOrder>())).Returns(true);
    
    // Act
    await stateManager.SetStatusByActionAsync(entity, MenuItemEnums.执行);
    
    // Assert
    Assert.That(entity.DataStatus, Is.EqualTo(DataStatus.完结));
}

[Test]
public void Execute_WhenBusinessOperationFails_ShouldKeepStatusAsConfirmed()
{
    // Arrange
    var entity = new PurchaseOrder { DataStatus = DataStatus.确认 };
    MockInventoryService.Setup(x => x.Execute(It.IsAny<PurchaseOrder>())).Throws<Exception>();
    
    // Act & Assert
    Assert.ThrowsAsync<Exception>(async () => 
        await stateManager.SetStatusByActionAsync(entity, MenuItemEnums.执行)
    );
    Assert.That(entity.DataStatus, Is.EqualTo(DataStatus.确认));
}
```

## 总结

通过正确的状态管理设计，实现了以下优化：

1. ✅ **业务流程清晰**：执行和结案都导致完结，但业务含义不同
2. ✅ **状态流转合理**：确认 → 完结（执行成功或结案）
3. ✅ **数据一致性好**：业务操作失败时自动回滚
4. ✅ **实现简单**：不需要额外的中间状态

关键点：
- **先执行业务操作，后更新状态**
- **业务操作失败时抛出异常，状态不变**
- **使用事务确保原子性**
