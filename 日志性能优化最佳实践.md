# 日志性能优化最佳实践

## 核心原则

**日志是为了排查问题，不是为了记录所有事情。**

频繁的日志I/O会严重影响系统性能，特别是在高并发场景下。

## 常见误区

### ❌ 误区1：在循环中记录日志

```csharp
// 错误示例
foreach (var item in largeCollection)
{
    if (item.IsValid)
    {
        _logger.LogDebug("处理项目: {ItemId}", item.Id);  // ❌ 可能调用10000次
    }
}
```

**问题**：
- 每次循环都调用日志方法（方法调用开销）
- 即使日志被过滤，仍有字符串格式化开销
- 如果日志级别开启，会产生大量I/O

### ✅ 正确做法：汇总统计

```csharp
// 正确示例
int processedCount = 0;
int skippedCount = 0;

foreach (var item in largeCollection)
{
    if (item.IsValid)
    {
        processedCount++;
    }
    else
    {
        skippedCount++;
    }
}

// 只在循环结束后记录一条日志
_logger.LogDebug("批量处理完成: 成功{Processed}个, 跳过{Skipped}个", 
    processedCount, skippedCount);
```

**优势**：
- 只调用一次日志方法
- 只进行一次I/O操作
- 信息更清晰（汇总 vs 分散）

---

### ❌ 误区2：使用过高的日志级别

```csharp
// 错误示例
if (config == null)
{
    _logger.LogWarning("配置实例为空");  // ❌ Warning会写入文件
}

if (user.NotFound)
{
    _logger.LogError("用户不存在: {UserId}", userId);  // ❌ Error会触发告警
}
```

**问题**：
- `Warning` 和 `Error` 级别通常会写入文件
- `Error` 可能触发监控告警
- 对于预期的、非异常情况，不应该使用高级别

### ✅ 正确做法：选择合适的日志级别

```csharp
// 正确示例
if (config == null)
{
    // Debug: 调试信息，生产环境默认不输出
    _logger.LogDebug("配置实例为空，已跳过");
}

if (user.NotFound)
{
    // Information: 正常业务流程中的信息
    _logger.LogInformation("用户不存在: {UserId}", userId);
}

if (database.ConnectionFailed)
{
    // Warning: 可恢复的异常，需要关注
    _logger.LogWarning("数据库连接失败，尝试重连...");
}

if (system.CriticalFailure)
{
    // Error: 严重错误，需要立即处理
    _logger.LogError(ex, "系统关键组件失败");
}
```

**日志级别选择指南**：

| 级别 | 使用场景 | 是否写文件 | 是否告警 |
|------|---------|-----------|---------|
| Trace | 最详细的调试信息 | 否 | 否 |
| Debug | 调试信息 | 否（默认） | 否 |
| Information | 正常业务流程 | 可选 | 否 |
| Warning | 可恢复的异常 | 是 | 可选 |
| Error | 严重错误 | 是 | 是 |
| Critical | 系统崩溃 | 是 | 是 |

### ❌ 误区3：在高频路径中记录日志

```csharp
// 错误示例 - 每秒可能调用数千次
public bool IsUserLoggedIn(string userId)
{
    _logger.LogDebug("检查用户登录状态: {UserId}", userId);  // ❌ 太频繁
    return _cache.ContainsKey(userId);
}
```

**问题**：
- 高频调用导致大量日志
- 即使使用Debug级别，方法调用本身也有开销
- 可能成为性能瓶颈

### ✅ 正确做法：移除或采样

```csharp
// 方案1：完全移除（推荐）
public bool IsUserLoggedIn(string userId)
{
    return _cache.ContainsKey(userId);
}

// 方案2：采样记录（每100次记录1次）
private static int _checkCount = 0;

public bool IsUserLoggedIn(string userId)
{
    int count = Interlocked.Increment(ref _checkCount);
    if (count % 100 == 0)
    {
        _logger.LogDebug("用户登录状态检查次数: {Count}", count);
    }
    return _cache.ContainsKey(userId);
}

// 方案3：条件记录（只在特定条件下）
public bool IsUserLoggedIn(string userId)
{
    if (_logger.IsEnabled(LogLevel.Debug))
    {
        // 先检查日志级别，避免不必要的字符串格式化
        _logger.LogDebug("检查VIP用户登录状态: {UserId}", userId);
    }
    return _cache.ContainsKey(userId);
}
```

### ❌ 误区4：记录无意义的日志

```csharp
// 错误示例
_logger.LogDebug("开始执行方法");  // ❌ 无意义
_logger.LogDebug("进入if分支");     // ❌ 无意义
_logger.LogDebug("变量值为: " + value);  // ❌ 应该用结构化日志
```

**问题**：
- 增加日志量但不提供有用信息
- 浪费I/O资源
- 干扰真正重要的日志

### ✅ 正确做法：记录有价值的信息

```csharp
// 正确示例
_logger.LogInformation("订单处理开始: OrderId={OrderId}, Amount={Amount}", 
    orderId, amount);

_logger.LogWarning("库存不足: ProductId={ProductId}, Requested={Requested}, Available={Available}", 
    productId, requestedQty, availableQty);

_logger.LogError(ex, "订单处理失败: OrderId={OrderId}, UserId={UserId}", 
    orderId, userId);
```

**好的日志特征**：
- 包含关键业务标识（OrderId, UserId等）
- 包含上下文信息（预期值 vs 实际值）
- 结构化日志（使用占位符，而非字符串拼接）
- 能够单独通过日志定位问题

## 性能优化技巧

### 1. 先检查日志级别

```csharp
// 优化前
_logger.LogDebug("详细数据: {Data}", expensiveToStringOperation());  // 即使Debug关闭也会执行

// 优化后
if (_logger.IsEnabled(LogLevel.Debug))
{
    _logger.LogDebug("详细数据: {Data}", expensiveToStringOperation());
}
```

**注意**：Microsoft.Extensions.Logging的占位符语法已经做了优化，只有在日志级别启用时才会格式化。但复杂表达式仍建议手动检查。

### 2. 避免在日志中序列化大对象

```csharp
// 错误示例
_logger.LogDebug("完整订单数据: {Order}", JsonConvert.SerializeObject(order));  // ❌ 可能很大

// 正确示例
_logger.LogDebug("订单摘要: OrderId={OrderId}, Amount={Amount}, Status={Status}", 
    order.Id, order.Amount, order.Status);
```

### 3. 使用异步日志（如果支持）

```csharp
// Serilog等支持异步日志
Log.Logger = new LoggerConfiguration()
    .WriteTo.Async(a => a.File("logs/app.log"))
    .CreateLogger();
```

### 4. 限制日志文件大小和数量

```xml
<!-- log4net配置示例 -->
<appender name="RollingFile" type="log4net.Appender.RollingFileAppender">
    <maximumFileSize value="10MB" />
    <maxSizeRollBackups value="10" />
</appender>
```

### 5. 生产环境调整日志级别

```json
// appsettings.Production.json
{
  "Logging": {
    "LogLevel": {
      "Default": "Warning",
      "Microsoft": "Warning",
      "System": "Warning",
      "YourApp.Namespace": "Information"  // 只对关键模块开启Info
    }
  }
}
```

## 实际案例对比

### 案例1：批量数据处理

#### ❌ 错误做法
```csharp
public async Task ProcessBatch(List<Order> orders)
{
    foreach (var order in orders)
    {
        _logger.LogDebug("开始处理订单: {OrderId}", order.Id);
        
        try
        {
            await ProcessOrder(order);
            _logger.LogDebug("订单处理成功: {OrderId}", order.Id);
        }
        catch (Exception ex)
        {
            _logger.LogError(ex, "订单处理失败: {OrderId}", order.Id);
        }
    }
}
```

**问题**：
- 1000个订单产生2000+条日志
- 每条日志都有I/O开销
- 日志文件迅速膨胀

#### ✅ 正确做法
```csharp
public async Task ProcessBatch(List<Order> orders)
{
    var stopwatch = Stopwatch.StartNew();
    int successCount = 0;
    int failureCount = 0;
    var failedOrders = new List<(string OrderId, string Error)>();
    
    foreach (var order in orders)
    {
        try
        {
            await ProcessOrder(order);
            successCount++;
        }
        catch (Exception ex)
        {
            failureCount++;
            failedOrders.Add((order.Id, ex.Message));
            
            // 只记录Error级别的失败日志
            _logger.LogError(ex, "订单处理失败: OrderId={OrderId}", order.Id);
        }
    }
    
    stopwatch.Stop();
    
    // 汇总统计（Information级别）
    _logger.LogInformation(
        "批量处理完成: 总数={Total}, 成功={Success}, 失败={Failure}, 耗时={ElapsedMs}ms",
        orders.Count, successCount, failureCount, stopwatch.ElapsedMilliseconds);
    
    // 如果有失败，记录摘要（Warning级别）
    if (failureCount > 0)
    {
        _logger.LogWarning("失败的订单ID: {FailedOrderIds}", 
            string.Join(", ", failedOrders.Select(x => x.OrderId)));
    }
}
```

**优势**：
- 日志数量从2000+减少到2-3条
- 包含汇总统计，更容易分析问题
- 保留了关键的Error日志用于排查

### 案例2：配置加载

#### ❌ 错误做法
```csharp
public void LoadConfigs()
{
    var configs = _configRepository.GetAll();
    
    foreach (var config in configs)
    {
        if (config == null)
        {
            _logger.LogWarning("配置为空: {ConfigType}", config?.GetType().Name);  // ❌ 逻辑错误
        }
        else
        {
            _logger.LogDebug("加载配置: {ConfigType}", config.Type);
        }
    }
}
```

**问题**：
- `config == null` 时 `config?.GetType()` 永远为null
- 每个配置都记录日志
- Warning级别过高

#### ✅ 正确做法
```csharp
public void LoadConfigs()
{
    var configs = _configRepository.GetAll();
    int loadedCount = 0;
    int skippedCount = 0;
    
    foreach (var config in configs)
    {
        if (config == null)
        {
            skippedCount++;  // 只计数
            continue;
        }
        
        _configCache[config.Type] = config;
        loadedCount++;
    }
    
    // 只在有跳过时记录一条Debug日志
    if (skippedCount > 0)
    {
        _logger.LogDebug("配置加载完成: 成功{Loaded}个, 跳过{Skipped}个空配置", 
            loadedCount, skippedCount);
    }
    else
    {
        _logger.LogDebug("配置加载完成: 共{Count}个", loadedCount);
    }
}
```

## 监控和告警

### 不要对正常日志设置告警

```csharp
// ❌ 错误：对正常业务日志设置告警
_logger.LogWarning("用户登录失败: UserId={UserId}", userId);  // 这可能很频繁

// ✅ 正确：只对异常模式告警
// 在监控系统层面设置规则：
// - 1分钟内登录失败超过100次 → 告警（可能是攻击）
// - 数据库连接失败 → 立即告警
// - 订单处理成功率低于95% → 告警
```

### 使用指标而不是日志进行监控

```csharp
// 使用Metrics而不是日志
_metrics.Counter("orders.processed.total").Increment();
_metrics.Counter("orders.processed.failed").Increment();
_metrics.Histogram("orders.processing.duration").Observe(elapsedMs);
```

## 总结

### 日志性能优化 Checklist

- [ ] 避免在循环中记录日志（使用汇总统计）
- [ ] 选择合适的日志级别（Debug用于调试，Warning/Error用于异常）
- [ ] 避免在高频路径中记录日志
- [ ] 记录有价值的信息（关键标识、上下文）
- [ ] 使用结构化日志（占位符而非字符串拼接）
- [ ] 生产环境调整日志级别（默认Warning或Error）
- [ ] 限制日志文件大小和保留数量
- [ ] 考虑使用异步日志
- [ ] 定期审查和清理无用日志

### 性能影响对比

| 场景 | 错误做法 | 正确做法 | 性能提升 |
|------|---------|---------|---------|
| 批量处理1000条 | 2000条日志 | 2-3条日志 | ~99% |
| 高频检查（1000次/秒） | 每次记录 | 采样或移除 | ~100% |
| 配置加载（10个） | 10条Warning | 1条Debug | ~90% |

**记住**：日志是为了帮助你排查问题，不是为了让日志文件变大。
