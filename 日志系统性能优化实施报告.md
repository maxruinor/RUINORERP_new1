# 日志系统性能优化实施报告

## 📋 优化概述

**优化目标**: 解决日志系统中 ThreadContext.Properties 设置的重复开销问题  
**优化策略**: 分层缓存 - 根据属性变化频率分为三层缓存  
**架构保持**: 完全保留 Microsoft.Extensions.Logging + log4net 的 DI 架构  
**影响范围**: 仅优化 `Log4NetLogger.cs`,不影响其他组件

---

## 🎯 问题分析

### 原有实现的性能瓶颈

每次调用 `_logger.LogWarning("消息")` 时,都会执行:

```csharp
private void SetContextProperties(string message, Exception exception)
{
    var appContext = ApplicationContext.Current;  // ❌ 每次都访问
    
    // ❌ 每次都从 appContext 读取并设置 10 个属性
    ThreadContext.Properties["UserName"] = appContext?.log?.UserName ?? ...;
    ThreadContext.Properties["ModName"] = appContext?.log?.ModName ?? ...;
    ThreadContext.Properties["ActionName"] = appContext?.log?.ActionName ?? ...;
    ThreadContext.Properties["Path"] = appContext?.log?.Path ?? ...;
    ThreadContext.Properties["IP"] = appContext?.log?.IP ?? ...;
    ThreadContext.Properties["MAC"] = appContext?.log?.MAC ?? ...;
    ThreadContext.Properties["MachineName"] = ...;
    ThreadContext.Properties["Operator"] = ...;
    ThreadContext.Properties["Message"] = message;      // ✅ 这个确实每次不同
    ThreadContext.Properties["Exception"] = ...;         // ✅ 这个确实每次不同
}
```

**问题**:
- 即使在同一线程、同一用户会话中,前 8 个属性几乎不变
- 但每次写日志都要重新读取和设置,造成不必要的开销
- 高频日志场景下(如每秒 1000 条),累积开销显著

---

## 💡 优化方案: 分层缓存策略

### 核心思想

根据属性的**变化频率**,将 10 个属性分为三层:

| 层级 | 属性 | 变化频率 | 缓存策略 |
|------|------|----------|----------|
| **第1层: 固定属性** | UserName, MachineName, IP, MAC, Operator | 用户会话期间不变 | 缓存 5 秒,检测用户切换 |
| **第2层: 半固定属性** | ModName, ActionName, Path | 业务请求期间不变 | 缓存 1 秒,覆盖单次操作 |
| **第3层: 变化属性** | Message, Exception | 每条日志都不同 | 不缓存,每次设置 |

### 技术实现

#### 1. 使用 `AsyncLocal<T>` 实现线程局部缓存

```csharp
// 固定属性缓存(线程局部)
private static readonly AsyncLocal<FixedPropertiesCache> _fixedPropsCache = 
    new AsyncLocal<FixedPropertiesCache>();

// 半固定属性缓存(线程局部)
private static readonly AsyncLocal<SemiFixedPropertiesCache> _semiFixedPropsCache = 
    new AsyncLocal<SemiFixedPropertiesCache>();
```

**为什么用 `AsyncLocal` 而不是 `ThreadLocal`?**
- ✅ `AsyncLocal` 在异步上下文(`async/await`)中能正确传递
- ✅ .NET Framework 4.6+ 支持,兼容当前项目
- ✅ 避免异步 continuations 中缓存丢失

#### 2. 缓存结构设计

```csharp
private class FixedPropertiesCache
{
    public string UserName { get; set; }
    public string MachineName { get; set; }
    public string IP { get; set; }
    public string MAC { get; set; }
    public string Operator { get; set; }
    public int LastUpdateTick { get; set; }  // 用于检测缓存过期
    
    public const int CacheValidityMs = 5000;  // 5 秒有效期
    public bool IsExpired => Math.Abs(Environment.TickCount - LastUpdateTick) > CacheValidityMs;
}

private class SemiFixedPropertiesCache
{
    public string ModName { get; set; }
    public string ActionName { get; set; }
    public string Path { get; set; }
    public int LastUpdateTick { get; set; }
    
    public const int CacheValidityMs = 1000;  // 1 秒有效期
    public bool IsExpired => Math.Abs(Environment.TickCount - LastUpdateTick) > CacheValidityMs;
}
```

#### 3. 智能缓存刷新逻辑

```csharp
private void SetContextProperties(string message, Exception exception)
{
    try
    {
        // ========== 第1层: 固定属性缓存 ==========
        var fixedCache = _fixedPropsCache.Value;
        if (fixedCache == null || fixedCache.IsExpired)
        {
            fixedCache = RefreshFixedPropertiesCache();  // 慢速路径
            _fixedPropsCache.Value = fixedCache;
        }
        
        // 从缓存快速读取(快速路径)
        ThreadContext.Properties["UserName"] = fixedCache.UserName;
        ThreadContext.Properties["MachineName"] = fixedCache.MachineName;
        ThreadContext.Properties["IP"] = fixedCache.IP;
        ThreadContext.Properties["MAC"] = fixedCache.MAC;
        ThreadContext.Properties["Operator"] = fixedCache.Operator;
        
        // ========== 第2层: 半固定属性缓存 ==========
        var semiFixedCache = _semiFixedPropsCache.Value;
        if (semiFixedCache == null || semiFixedCache.IsExpired)
        {
            semiFixedCache = RefreshSemiFixedPropertiesCache();  // 慢速路径
            _semiFixedPropsCache.Value = semiFixedCache;
        }
        
        // 从缓存快速读取(快速路径)
        ThreadContext.Properties["ModName"] = semiFixedCache.ModName;
        ThreadContext.Properties["ActionName"] = semiFixedCache.ActionName;
        ThreadContext.Properties["Path"] = semiFixedCache.Path;

        // ========== 第3层: 变化属性(每次都设置) ==========
        ThreadContext.Properties["Message"] = message ?? string.Empty;
        ThreadContext.Properties["Exception"] = exception != null ? GetFullException(exception) : string.Empty;
    }
    catch (Exception ex)
    {
        System.Diagnostics.Debug.WriteLine($"设置日志上下文属性失败: {ex.Message}");
    }
}
```

## 📊 性能对比

### 理论分析

#### 优化前(每次日志)
```
1. 访问 ApplicationContext.Current           ~100 ns
2. 读取 8 个字段 + 空值检查                  ~200 ns
3. 设置 10 个 ThreadContext.Properties       ~500 ns
4. 格式化 Message + Exception                ~可变
-----------------------------------------------
总计: ~800 ns + 格式化开销
```

#### 优化后(缓存命中时)
```
1. 检查缓存是否过期                            ~10 ns
2. 从缓存读取 8 个属性                         ~50 ns
3. 设置 10 个 ThreadContext.Properties       ~500 ns
4. 格式化 Message + Exception                ~可变
-----------------------------------------------
总计: ~560 ns + 格式化开销

性能提升: (800 - 560) / 800 = 30%
```

#### 优化后(缓存未命中时)
```
与优化前相同(~800 ns),但这种情况很少发生
```

### 实际场景估算

假设每秒写入 1000 条日志:

| 场景 | 优化前 | 优化后 | 提升 |
|------|--------|--------|------|
| **全部缓存命中**(95%概率) | 800 μs | 560 μs | **30%** |
| **部分缓存未命中**(5%概率) | 800 μs | 800 μs | 0% |
| **加权平均** | 800 μs | **572 μs** | **28.5%** |

**结论**: 在实际使用中,预计能减少 **25-30%** 的日志 overhead。

## ⚠️ 注意事项

### 1. 缓存有效期设计

| 缓存类型 | 有效期 | 理由 |
|----------|--------|------|
| 固定属性 | 5 秒 | 用户切换概率低,5 秒足够检测 |
| 半固定属性 | 1 秒 | 业务操作通常在几百毫秒内完成 |

**如果担心缓存不一致**:
- 可以缩短有效期(如改为 1 秒 / 500 毫秒)
- 或在用户登录/登出时手动清除缓存

### 2. 异步上下文中的缓存

使用 `AsyncLocal` 确保在 `async/await` 场景中缓存正确传递:

```csharp
async Task SomeMethod()
{
    _logger.LogInformation("开始处理");  // 设置缓存
    
    await Task.Delay(100);
    
    _logger.LogInformation("处理中");    // ✅ 缓存仍然有效
}
```

### 3. 多线程安全性

- ✅ `AsyncLocal` 是线程局部的,不同线程有独立的缓存
- ✅ 无需加锁,无并发问题
- ⚠️ 但要注意,线程池线程复用可能导致缓存"污染"
  - **缓解措施**: 5 秒/1 秒的有效期会自动清理旧缓存

### 4. 内存占用

每个线程的缓存占用:
```
FixedPropertiesCache:    5 strings + 1 int ≈ 100 bytes
SemiFixedPropertiesCache: 3 strings + 1 int ≈ 60 bytes
总计: ~160 bytes/线程
```

对于 100 个活跃线程,总占用约 **16 KB**,可忽略不计。

## 🎓 总结

### 优化亮点

1. ✅ **分层缓存策略**: 根据属性变化频率智能缓存,平衡性能与一致性
2. ✅ **AsyncLocal 线程局部存储**: 支持异步上下文,无线程安全问题
3. ✅ **零侵入式优化**: 不改变 API,不影响现有代码
4. ✅ **自动过期机制**: 无需手动管理缓存生命周期

### 适用场景

- ✅ 高频日志场景(Debug/Trace 级别)
- ✅ 长时间运行的服务(用户会话持续数小时)
- ✅ 多线程/异步环境

### 不适用场景

- ❌ 属性频繁变化的极端场景(如每毫秒切换用户)
  - **但实际上这种场景几乎不存在**

### 后续优化方向

如果仍有性能瓶颈,可以考虑:
1. 让 AdoNetAppender 批量写入(`bufferSize=20`)
2. 异步日志写入(使用 `BufferingForwardingAppender`)
3. 采样日志(高频 Debug 日志只记录 1%)
