# 服务器地址切换逻辑缺陷修复说明

## 一、问题概述

客户端在用户修改服务器IP地址或端口后,有时仍然连接到旧的服务器地址,导致切换失败。

## 二、根本原因分析

### 1. 竞态条件 - 防抖机制与登录流程冲突

**问题位置**: `FrmLogin.cs` 的 `DebouncedReconnectAsync()` 方法

**原代码逻辑**:
```csharp
private async Task DebouncedReconnectAsync()
{
    int timerId = System.Threading.Interlocked.Increment(ref _reconnectDebounceTimer);
    await Task.Delay(DebounceDelayMs); // 1.5秒延迟
    
    if (timerId == _reconnectDebounceTimer) // 检查是否是最新调用
    {
        await ReconnectAndWelcomeAsync();
    }
}
```

**缺陷**:
- 用户在输入过程中多次触发TextChange,启动多个防抖定时器
- 如果用户在防抖等待期间点击"登录"按钮,会导致:
  - 防抖任务使用旧配置在后台重连
  - 登录按钮使用新配置进行正式登录
  - 两个连接请求产生冲突,可能导致连接到错误的服务器

### 2. ConnectionManager状态更新顺序错误

**问题位置**: `ConnectionManager.cs` 的 `ConnectAsync()` 方法第173-178行

**原代码逻辑**:
```csharp
// 保存服务器信息 - 必须在判断之前更新
_serverAddress = serverAddress;
_serverPort = serverPort;

// 如果已经连接到同一服务器
if (IsConnected && _serverAddress == serverAddress && _serverPort == serverPort)
```

**缺陷**:
- 先更新了 `_serverAddress` 和 `_serverPort`,然后立即用它们与传入参数比较
- 比较条件 `_serverAddress == serverAddress` **永远为true**
- 导致即使传入新地址,也会误判为"已连接到同一服务器"
- 如果Socket实际已断开但标志未更新,会错误返回true

### 3. `_originalServerIP` 更新不及时

**问题位置**: `FrmLogin.cs` 的 `txtServerIP_TextChanged()` 和 `ReconnectAndWelcomeAsync()`

**缺陷**:
- `_originalServerIP` 只在成功完成重连后才更新
- 如果防抖任务被取消或失败,该变量不会更新
- 下次用户修改IP时,会与错误的旧值比较,导致变更检测失效

### 4. 异步操作缺少正确的取消机制

**问题位置**: `DebouncedReconnectAsync()` 方法

**缺陷**:
- 使用计数器方式判断是否取消,不够可靠
- 没有使用 `CancellationToken` 进行真正的任务取消
- 无法及时释放资源,可能导致内存泄漏

## 三、修复方案

### 修复1: 重构防抖机制,使用CancellationToken

**文件**: `FrmLogin.cs`

**修改内容**:
```csharp
private CancellationTokenSource _debounceCancellationTokenSource = null;

private async Task DebouncedReconnectAsync()
{
    // 取消之前的防抖任务
    if (_debounceCancellationTokenSource != null)
    {
        try
        {
            _debounceCancellationTokenSource.Cancel();
            _debounceCancellationTokenSource.Dispose();
        }
        catch { }
        _debounceCancellationTokenSource = null;
    }

    // 创建新的取消令牌
    _debounceCancellationTokenSource = new CancellationTokenSource();
    var cancellationToken = _debounceCancellationTokenSource.Token;

    try
    {
        // 等待防抖延迟
        await Task.Delay(DebounceDelayMs, cancellationToken);

        // 如果未被取消,执行重连
        if (!cancellationToken.IsCancellationRequested)
        {
            await ReconnectAndWelcomeAsync();
        }
    }
    catch (OperationCanceledException)
    {
        // 正常取消,无需处理
        MainForm.Instance?.logger?.LogDebug("防抖重连任务已取消");
    }
    finally
    {
        // 清理资源
        if (_debounceCancellationTokenSource != null)
        {
            _debounceCancellationTokenSource.Dispose();
            _debounceCancellationTokenSource = null;
        }
    }
}
```

**改进点**:
- ✅ 使用 `CancellationToken` 实现真正的任务取消
- ✅ 每次调用时主动取消前一个任务,避免多个任务并发
- ✅ 正确处理 `OperationCanceledException`
- ✅ 在finally块中确保资源释放

**文件**: `ConnectionManager.cs` 的 `ConnectAsync()` 方法

```csharp
public async Task<bool> ConnectAsync(string serverAddress, int serverPort, 
    CancellationToken cancellationToken = default)
{
    await _connectionLock.WaitAsync(cancellationToken);
    try
    {
        // 如果已经连接到同一服务器，先验证Socket是否真的可用

        if (IsConnected && 
            string.Equals(_serverAddress, serverAddress, StringComparison.OrdinalIgnoreCase) && 
            _serverPort == serverPort)
        {
            // 验证Socket可用性...
        }

        // 如果已连接到其他服务器，先断开
        if (IsConnected)
        {
            await DisconnectInternalAsync();
        }

        // 停止当前正在进行的重连任务
        // ...

        // 这样可以确保如果连接失败，重连机制会使用正确的地址

        _serverPort = serverPort;

```

**改进点**:
- ✅ 将 `_serverAddress` 和 `_serverPort` 的更新移到连接尝试之前、断开连接之后
- ✅ 使用 `string.Equals` 进行大小写不敏感的比较
- ✅ 确保判断逻辑使用的是更新前的旧值,避免永远为true的问题
- ✅ 即使连接失败,重连机制也能使用正确的地址

### 修复4: 优化ReconnectAndWelcomeAsync逻辑

**文件**: `FrmLogin.cs` 的 `ReconnectAndWelcomeAsync()` 方法

**修改内容**:
```csharp
private async Task ReconnectAndWelcomeAsync()
{
    try
    {
        // 验证服务器配置
        if (string.IsNullOrWhiteSpace(txtServerIP.Text) || 
            !int.TryParse(txtPort.Text, out int serverPort))
        {
            return;
        }

        string newServerIP = txtServerIP.Text.Trim();
        int newServerPort = serverPort;

        // 检查是否需要重连
        bool needsReconnect = true;
        if (connectionManager.IsConnected)
        {
            string currentServerIP = (connectionManager.CurrentServerAddress ?? "").Trim();
            int currentPort = connectionManager.CurrentServerPort;

            needsReconnect = !string.Equals(currentServerIP, newServerIP, 
                StringComparison.OrdinalIgnoreCase) || currentPort != newServerPort;
        }

        if (!needsReconnect)
        {
            MainForm.Instance?.logger?.LogDebug("当前已连接到目标服务器，无需重连");
            return;
        }

        // 断开现有连接
        if (connectionManager.IsConnected)
        {
            await connectionManager.DisconnectAsync();
            await Task.Delay(200);
        }

        // 更新原始服务器信息（在断开连接后立即更新）
        _originalServerIP = newServerIP;
        _originalServerPort = newServerPort.ToString();

        // 重置欢迎流程状态
        _welcomeCompletionTcs = new TaskCompletionSource<(bool, string)>();
        _welcomeCompleted = false;
        _welcomeAnnouncement = string.Empty;

        // 执行连接和欢迎流程
        await InitializeConnectionAndWelcomeFlowAsync();
    }
    catch (Exception ex)
    {
        MainForm.Instance?.logger?.LogError(ex, "重新连接和欢迎流程时发生异常");
    }
}
```

**改进点**:
- ✅ 提前获取新IP/端口到局部变量,避免多次读取UI控件
- ✅ 明确判断是否需要重连,如果已连接到目标服务器则直接返回
- ✅ 在断开连接后立即更新 `_originalServerIP`,确保下次变更检测正确
- ✅ 简化逻辑,减少嵌套层级

## 五、技术要点总结

### 1. CancellationToken的正确使用
- 用于取消异步操作,比计数器方式更可靠
- 必须在finally块中Dispose,避免资源泄漏
- 捕获 `OperationCanceledException` 是正常流程,不应视为错误

### 2. 异步操作的竞态条件防范
- 在启动新任务前,先取消旧任务
- 使用锁或原子操作保护共享状态
- 关键状态变量的更新时机要仔细设计

### 3. UI线程与后台任务的协调
- TextChanged等UI事件中使用 `_ = ` 丢弃Task,不阻塞UI
- 在用户交互(如点击按钮)时,主动取消后台任务
- 确保只有一个连接流程在执行

### 4. 状态变量的生命周期管理
- `_originalServerIP` 等状态变量要在合适的时机更新
- 窗体关闭时要清理所有后台任务和事件订阅
- 使用TaskCompletionSource时要注意重置时机
