# 系统日志问题修复说明

## 问题概述

根据2026-04-11的系统日志，发现了以下三个主要问题：

### 1. 性能数据上传授权失败（ERROR级别）
```
Token附加失败: CommandId=0x0012:PerformanceDataUpload, TokenInfo=已初始化, TokenManager=可用
发送请求失败: 没有合法授权令牌, 指令：0x0012:PerformanceDataUpload
```

**影响**：频繁的错误日志，可能影响系统性能监控数据的收集

### 2. 单据解锁警告（WARN级别）
```
解锁失败，已恢复本地锁状态: 单据ID=2042809454268911616, 原因: 单据未被锁定或不存在
```

**影响**：虽然是正常边界情况，但使用WARN级别可能导致运维人员误判

### 3. 欢迎流程验证超时（WARN级别）
```
欢迎流程验证超时,但连接已建立
```

**影响**：可能是网络波动或服务器繁忙导致的临时问题

---

## 根本原因分析

### 问题1：性能数据上传Token授权失败

**根本原因**：
在 `ClientCommunicationService.cs` 的 `SendPacketCoreAsync` 方法中：
1. `AutoAttachTokenAsync` 方法在某些情况下不会设置Token（例如TokenStorage未初始化、GetTokenAsync返回null等）
2. 但后续代码对所有非登录命令都强制要求Token必须存在，否则抛出异常
3. 对于性能数据上报这类**非关键命令**，Token缺失时不应该抛出异常，而应该静默跳过

**调用链**：
```
ClientPerformanceMonitorService.OnPerformanceDataUpload()
  → SendPerformanceDataAsync()
    → ClientCommunicationService.SendCommandWithResponseAsync()
      → SendPacketCoreAsync()
        → AutoAttachTokenAsync()  // 可能不设置Token
        → 检查 Token == null  // 如果为null则抛出异常 ❌
```

### 问题2：单据解锁警告日志级别不当

**根本原因**：
在 `ClientLockManagementService.cs` 的 `UnlockBillAsync` 方法中：
1. 当服务器返回"单据未被锁定或不存在"时，这通常是正常的边界情况
   - 可能是重复解锁操作
   - 可能是锁已过期自动释放
   - 可能是本地状态与服务器状态短暂不一致
2. 但代码统一使用 `LogWarning` 记录，导致产生大量不必要的警告日志

### 问题3：欢迎流程验证超时

这是网络连接层面的问题，可能与网络波动、服务器负载等因素有关，属于偶发性问题，不需要代码修复。

## 修复方案

### 修复1：优化Token附加逻辑（ClientCommunicationService.cs）

#### 修改位置1：AutoAttachTokenAsync 方法增强日志

**文件**：`RUINORERP.UI\Network\ClientCommunicationService.cs`  
**行号**：约1508-1555行

**修改内容**：
```csharp
protected virtual async Task AutoAttachTokenAsync(CommandContext executionContext)
{
    try
    {
        // 使用null条件运算符简化检查
        if (_tokenManager?.TokenStorage == null)
        {
            _logger?.LogDebug("TokenManager或TokenStorage未初始化，跳过Token附加");
            return;
        }

        // 获取令牌并验证有效性
        var tokenInfo = await _tokenManager.TokenStorage.GetTokenAsync();

        // ✅ 检查Token是否存在且未过期
        if (tokenInfo != null && !string.IsNullOrEmpty(tokenInfo.AccessToken))
        {
            // ... 原有的Token刷新逻辑 ...
            
            // 使用现有Token(可能已过期,由服务端处理)
            executionContext.Token = tokenInfo;
            _logger?.LogDebug("Token附加成功");  // ✅ 新增：成功日志
        }
        else
        {
            _logger?.LogDebug("TokenInfo为空或AccessToken为空，无法附加Token");  // ✅ 新增：失败原因日志
        }
    }
    catch (Exception ex)
    {
        _logger?.LogWarning(ex, "自动附加Token失败");
    }
}
```

**改进点**：
- ✅ 增加详细的调试日志，便于排查Token附加失败的具体原因
- ✅ 区分不同的失败场景（TokenStorage未初始化 vs TokenInfo为空）

**修改后**：
```csharp
await AutoAttachTokenAsync(packet.ExecutionContext);
//除登录登出命令，其他命令都需要附加令牌
if (packet.CommandId != AuthenticationCommands.Login)
{
    if (packet.ExecutionContext.Token == null)
    {
        // ✅ 对于非关键命令（如性能数据上报），Token缺失时静默跳过而不是抛出异常
        bool isNonCriticalCommand = packet.CommandId == SystemCommands.PerformanceDataUpload ||
                                   packet.CommandId == SystemCommands.Heartbeat ||
                                   packet.CommandId == SystemCommands.PerformanceMonitorStatus;
        
        if (isNonCriticalCommand)
        {
            _logger?.LogDebug("非关键命令 {CommandId} Token未就绪，静默跳过执行", commandId.ToString());
            // 对于非关键命令，直接返回而不发送，避免抛出异常
            return;
        }
        
        // 记录详细日志以便排查问题
        _logger?.LogWarning("Token附加失败: CommandId={CommandId}, ...", ...);
        
        // 对于关键命令，仍然抛出异常
        throw new Exception($"发送请求失败: 没有合法授权令牌, 指令：{commandId.ToString()}");
    }
}
```

**改进点**：
- ✅ 识别非关键命令（性能数据上报、心跳、监控状态等）
- ✅ 非关键命令Token缺失时静默跳过，不抛出异常
- ✅ 关键命令仍然保持严格的Token验证
- ✅ 减少错误日志噪音，提升系统稳定性

```csharp
else
{
    // 边界条件：解锁失败，恢复本地状态
    if (removedLock != null)
    {
        _activeLocks.TryAdd(billId, removedLock);
        _clientCache.UpdateCacheItem(removedLock);
        
        // ✅ 根据失败原因区分日志级别
        bool isLockNotFound = response.Message?.Contains("未被锁定") == true || 
                             response.Message?.Contains("不存在") == true;
        
        if (isLockNotFound)
        {
            // 如果是因为锁不存在而失败，这通常是正常的（可能是重复解锁或锁已过期）
            _logger.LogDebug("解锁时锁已不存在: 单据ID={BillId}, 原因: {Message}",
                billId, response.Message);
        }
        else
        {
            // 其他失败原因记录为警告
            _logger.LogWarning("解锁失败，已恢复本地锁状态: 单据ID={BillId}, 原因: {Message}",
                billId, response.Message);
        }
    }
    else
    {
        // 没有移除的锁，说明本地也没有这个锁的记录
        _logger.LogDebug("解锁时本地无锁记录: 单据ID={BillId}, 服务器响应: {Message}",
            billId, response.Message);
    }
}
```

**改进点**：
- ✅ 智能识别"锁不存在"这类正常的边界情况
- ✅ 正常边界情况使用 `LogDebug` 而非 `LogWarning`
- ✅ 真正的异常情况仍然使用 `LogWarning`
- ✅ 增加本地无锁记录的处理分支
- ✅ 减少误报警告，提升日志质量
