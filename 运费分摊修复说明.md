# 运费分摊多次出库问题修复说明

## 问题描述

系统在销售订单分多次出库时，存在以下问题:
- **首次出库**:能按数量比例正确分摊运费 (如 100 个产品收 10 元运费，首次出库 80 个分摊 8 元) ✅
- **第二次出库**:剩余 20 个出库时运费计算为零，而非预期的 2 元 ❌

## 根因分析

### 原因 1:缺少累计出库数量校验
原代码只考虑当前出库单的数量，没有检查该订单之前已经出库了多少，可能导致:
- 多次出库运费总和超过订单总运费
- 超量出库时财务数据混乱

### 原因 2:未处理最后一次出库的剩余运费
原代码每次都按固定比例计算，导致四舍五入误差累积:
- 第一次：10 × (80/100) = 8.00 元
- 第二次：10 × (20/100) = 2.00 元
- 但如果第一次是 33.33 元，第二次也是 33.33 元，第三次也是 33.33 元，总和就变成了 99.99 元 ≠ 100 元

### 原因 3:容差阈值可能误判小额运费
当运费非常小 (如 0.005 元) 时，可能被容差阈值 (默认 0.0001) 归零，导致正常的小额运费丢失。

## 解决方案

### 修改文件 1: `tb_SaleOrderControllerPartial.cs`

#### 修改位置：第 1777-1842 行

**核心改进:**

1. **添加超量出库检查和自动调整**
```csharp
// 检查是否超量出库
if (totalDeliveredQty + currentOutQty > totalOrderQty)
{
    // 自动调整出库数量，防止超发
    currentOutQty = totalOrderQty - totalDeliveredQty;
    // 如果已无货可出，直接返回
    if (currentOutQty <= 0)
    {
        tipsMsg.Add($"订单:{entity.SaleOrderNo}已全部出库，请检查是否正在重复出库!");
        return entity;
    }
    // 按比例调整明细行数量
    foreach (var detail in NewDetails)
    {
        detail.Quantity = detail.Quantity * (currentOutQty / entity.TotalQty);
        detail.Quantity = Math.Round(detail.Quantity, 0, MidpointRounding.AwayFromZero);
    }
    entity.TotalQty = currentOutQty;
}
```

2. **查询已累计出库的运费**
```csharp
// 查询该订单已审核出库单的累计运费
decimal deliveredFreightIncome = 0;
if (saleorder.tb_SaleOuts != null && saleorder.tb_SaleOuts.Count > 0)
{
    deliveredFreightIncome = saleorder.tb_SaleOuts
        .Where(o => o.DataStatus >= (int)DataStatus.确认)
        .Sum(o => o.FreightIncome);
}
```

3. **区分是否为最后一次出库**
```csharp
// 判断是否为最后一次出库 (剩余数量全部出库)
bool isLastOutbound = (totalDeliveredQty + currentOutQty >= totalOrderQty);

if (isLastOutbound)
{
    // 最后一次出库：使用剩余运费，避免四舍五入误差累积
    entity.FreightIncome = remainingFreight;
    if (entity.FreightIncome < 0) entity.FreightIncome = 0; // 防止负数
}
else
{
    // 非最后一次出库：按当前出库数量比例分摊
    decimal currentRatio = totalOrderQty > 0 ? currentOutQty / totalOrderQty : 0;
    entity.FreightIncome = saleorder.FreightIncome * currentRatio;
}
```

4. **添加最后一次出库的运费差额调整**
```csharp
// 如果是最后一次出库，确保分摊总和等于主表运费
if (isLastOutbound)
{
    decimal allocatedTotal = NewDetails.Sum(d => d.AllocatedFreightIncome);
    decimal difference = entity.FreightIncome - allocatedTotal;
    
    // 如果存在差额且绝对值大于容差，调整最后一行
    if (Math.Abs(difference) > authorizeController.GetAmountCalculationTolerance() && NewDetails.Count > 0)
    {
        var lastDetail = NewDetails[NewDetails.Count - 1];
        lastDetail.AllocatedFreightIncome += difference;
        lastDetail.AllocatedFreightIncome = lastDetail.AllocatedFreightIncome.ToRoundDecimalPlaces(authorizeController.GetMoneyDataPrecision());
    }
}
```

### 修改文件 2: `SaleOutCoordinator.cs`

#### 修改位置：第 153-180 行

**优化容差处理逻辑:**

```csharp
// 原来的逻辑：只要小于容差就归零
if (Math.Abs(allocatedValue) <= tolerance)
{
    allocatedValue = 0m;
}

// 修改后的逻辑：同时检查是否小于 0.01 元，避免误判正常的小额运费
if (Math.Abs(allocatedValue) <= tolerance && Math.Abs(allocatedValue) < 0.01m)
{
    allocatedValue = 0m;
}
```

**改进点:**
- 只有当值既在容差范围内 **又** 小于 0.01 元时才归零
- 避免了如 0.005 元这样的正常小额运费被错误归零

## 测试用例验证

### 场景 1:标准多次出库
**测试数据:**
- 订单：100 个产品，运费 10 元
- 第一次出库：80 个
- 第二次出库：20 个

**预期结果:**
- 第一次：分摊运费 = 10 × (80/100) = 8.00 元
- 第二次：分摊运费 = 剩余运费 = 10 - 8 = 2.00 元 ✅
- 总和：8 + 2 = 10 元 ✅

### 场景 2:三次出库
**测试数据:**
- 订单：100 个产品，运费 10 元
- 第一次：30 个
- 第二次：40 个  
- 第三次：30 个

**预期结果:**
- 第一次：10 × (30/100) = 3.00 元
- 第二次：10 × (40/100) = 4.00 元
- 第三次：剩余运费 = 10 - 3 - 4 = 3.00 元
- 总和：3 + 4 + 3 = 10 元 ✅

### 场景 3:超量出库
**测试数据:**
- 订单：100 个产品，运费 10 元
- 已出库：90 个，已分摊运费 9 元
- 本次尝试出库：20 个

**预期结果:**
- 系统自动调整为出库 10 个
- 运费 = 剩余运费 = 10 - 9 = 1.00 元
- 提示警告信息 ✅

### 场景 4:小额运费
**测试数据:**
- 订单：1000 个产品，运费 1 元
- 第一次出库：1 个

**预期结果:**
- 分摊运费 = 1 × (1/1000) = 0.001 元
- 不会被视为零处理 ✅

## 注意事项

### 1. 数据精度
- 所有金额计算都使用 `ToRoundDecimalPlaces()` 方法，确保精度一致
- 容差阈值通过 `GetAmountCalculationTolerance()` 动态获取

### 2. 状态判断
- 只统计已审核 (`DataStatus >= DataStatus.确认`) 的出库单
- 草稿状态的出库单不计入累计运费

### 3. 边界场景
- 当剩余运费为负数时，自动归零保护
- 当订单已全部出库时，阻止重复出库操作
