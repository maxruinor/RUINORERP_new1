# 运费重复计算问题修复说明

## 问题描述

在销售订单→销售出库单→应收款单的业务流程中，**运费被重复计算**，导致应收款单的未核销金额虚增。

### 问题表现

1. **销售订单**有运费收入（`FreightIncome`）
2. **销售出库单**继承运费收入，并增加运费成本（`FreightCost`）
3. **生成应收款单**时：
   - 明细行金额累加 = 产品交易金额（已包含运费收入）
   - 但又额外添加了 `ShippingFee`（运费成本）
   - 导致**总金额和未核销金额重复计算了运费**

### 多次出库场景

当一个销售订单分多次出库时：
- **预期逻辑**：只有第一次出库包含运费，后续出库的运费应为 0
- **实际情况**：代码没有正确判断多次出库的情况，导致运费处理混乱

## 根本原因分析

### 原因 1：多次出库运费判断逻辑错误

**原始代码**（`tb_SaleOutControllerPartial.cs` 第 944-949 行）：

```csharp
//运费检测  如果一个订单有运费。多次出库时。运费也默认加到了每次的出库单。
//多次出库第一次加上运费。第二次起都是为 0. 代码要在财务前面处理。因为应收明细中会将运费为一行添加。
if (entity.tb_saleorder.tb_SaleOuts.Count > 1)
{
    entity.FreightIncome = 0;
}
```

**问题**：
- `entity.tb_saleorder.tb_SaleOuts.Count > 1` 这个判断不准确
- 它只是判断订单关联的出库单数量大于 1，但没有判断**当前出库单是否为第一次出库**
- 即使只有 1 个出库单，只要 `Count > 1` 就成立，就会错误地将运费设为 0

### 原因 2：应收款单总金额重复计算运费

**原始代码**（`tb_FM_ReceivablePayableControllerPartial.cs` 第 2408-2416 行）：

```csharp
if (entity.FreightCost > 0 && !IsProcessCommission)
{
    payable.ShippingFee = entity.FreightCost;
}

payable.TotalLocalPayableAmount = payable.tb_FM_ReceivablePayableDetails.Sum(c => c.LocalPayableAmount) + payable.ShippingFee;
```

**问题**：
- 出库单的 `TotalAmount = 产品金额 + FreightIncome`（已包含运费收入）
- 应收款单明细的 `LocalPayableAmount = TransactionPrice * Quantity`
- 这些明细累加后应该等于出库单的 `TotalAmount`
- 但是代码又**额外添加了 `ShippingFee`**，导致运费重复计算

## 修复方案

### 修复 1：正确处理多次出库的运费分配

**文件**：`tb_SaleOutControllerPartial.cs`

**修复后代码**：

```csharp
//运费检测  如果一个订单有运费。多次出库时。运费也默认加到了每次的出库单。
//多次出库第一次加上运费。 第二次起都是为 0. 代码要在财务前面处理。因为应收明细中会将运费为一行添加。
//关键修复：需要判断当前出库单是否为该订单的第一次出库
if (entity.tb_saleorder != null && entity.tb_saleorder.tb_SaleOuts != null)
{
    // 获取所有已审核的出库单（排除当前正在审核的这张）
    var auditedSaleOuts = entity.tb_saleorder.tb_SaleOuts
        .Where(o => o.SaleOut_MainID != entity.SaleOut_MainID 
                 && o.DataStatus >= (int)DataStatus.确认 
                 && o.ApprovalStatus == (int)ApprovalStatus.审核通过)
        .OrderBy(o => o.Created_at)
        .ToList();
    
    // 如果已经有审核过的出库单，则当前出库单的运费收入应该为 0
    if (auditedSaleOuts.Count > 0)
    {
        entity.FreightIncome = 0;
        // 同时调整总金额，扣除运费收入
        entity.TotalAmount = entity.TotalAmount - entity.FreightIncome;
        MainForm.Instance.uclog.AddLog($"销售出库单{entity.SaleOutNo}不是第一次出库，运费收入已调整为 0，调整后总金额：{entity.TotalAmount}");
    }
    else
    {
        MainForm.Instance.uclog.AddLog($"销售出库单{entity.SaleOutNo}是第一次出库，保留运费收入：{entity.FreightIncome}");
    }
}
```

**修复要点**：
1. ✅ 检查**已审核的出库单**（而不是所有出库单）
2. ✅ 排除当前正在审核的这张出库单
3. ✅ 按创建时间排序，确保第一次出库的优先级
4. ✅ 如果是第一次出库，保留运费收入；否则设为 0 并调整总金额
5. ✅ 添加日志记录，便于追踪和审计

### 修复 2：移除应收款单中的重复运费计算

**文件**：`tb_FM_ReceivablePayableControllerPartial.cs`

**修复后代码**：

```csharp
#region 单独处理运费 ,这里是应收。意思是收取客户的运费。应该以运费成本为标准。佣金不需要
// 关键修复：出库单的 TotalAmount 已经包含了 FreightIncome（运费收入）
// 应收款单明细的 LocalPayableAmount 累加后应该等于 TotalAmount
// 因此不应该再额外添加 ShippingFee，否则会导致运费重复计算
// ShippingFee 字段仅用于记录运费成本信息，不参与总金额计算
if (entity.FreightCost > 0 && !IsProcessCommission)
{
    payable.ShippingFee = entity.FreightCost;
    // 注释说明：ShippingFee 只是信息记录，已在 TotalAmount 中体现，无需重复累加
    // payable.TotalLocalPayableAmount 不应该再加 payable.ShippingFee
}

#endregion

payable.tb_FM_ReceivablePayableDetails = details;
// 关键修复：移除重复计算的运费
// 错误写法：payable.TotalLocalPayableAmount = payable.tb_FM_ReceivablePayableDetails.Sum(c => c.LocalPayableAmount) + payable.ShippingFee;
// 正确写法：明细金额累加已经包含在 TotalAmount 中（含运费收入），不应再额外添加运费
payable.TotalLocalPayableAmount = payable.tb_FM_ReceivablePayableDetails.Sum(c => c.LocalPayableAmount);
payable.TaxTotalAmount = payable.tb_FM_ReceivablePayableDetails.Sum(c => c.TaxLocalAmount);
payable.UntaxedTotalAmont = payable.TotalLocalPayableAmount - payable.TaxTotalAmount;
```

**修复要点**：
1. ✅ 保留了 `ShippingFee` 字段（用于信息记录和统计）
2. ✅ 但**不再将其累加到总金额中**
3. ✅ `TotalLocalPayableAmount` 只计算明细金额的累加
4. ✅ 添加详细注释说明修复原因

## 数据流验证

### 修复前（错误）

```
销售订单:
  TotalAmount = 10000 (产品) + 500 (运费) = 10500

销售出库单:
  TotalAmount = 10000 (产品) + 500 (FreightIncome) = 10500

应收款单:
  明细金额累加 = 10000 (产品 TransactionPrice * Quantity)
  ShippingFee = 500 (FreightCost)
  ❌ TotalLocalPayableAmount = 10000 + 500 = 10500
  ❌ LocalBalanceAmount (未核销金额) = 10500
  
  问题：明细金额 10000 已经是产品金额，再加 500 运费就重复了！
```

### 修复后（正确）

```
销售订单:
  TotalAmount = 10000 (产品) + 500 (运费) = 10500

销售出库单 (第一次出库):
  TotalAmount = 10000 (产品) + 500 (FreightIncome) = 10500

销售出库单 (第二次出库):
  FreightIncome = 0
  TotalAmount = 8000 (产品) + 0 (运费) = 8000

应收款单:
  明细金额累加 = 10000 (产品 TransactionPrice * Quantity，已含运费)
  ShippingFee = 500 (仅用于记录，不参与计算)
  ✅ TotalLocalPayableAmount = 10000
  ✅ LocalBalanceAmount (未核销金额) = 10000
  
  验证：与出库单 TotalAmount 一致！✅
```

## 影响范围

### 受影响的业务流程

1. ✅ 销售订单 → 销售出库单 → 应收款单
2. ✅ 多次出库场景的运费分配
3. ✅ 应收款单的未核销金额计算
4. ✅ 对账单生成（基于应收款单余额）

### 不受影响的部分

1. ✅ 佣金应付单据（`IsProcessCommission = true` 时不处理运费）
2. ✅ 其他业务类型生成的应收款单（采购入库、退货等）
3. ✅ `ShippingFee` 字段的存储和查询（仍然保留）

**预期结果**：
- 出库单 `FreightIncome = 500`，`TotalAmount = 10500`
- 应收款单 `TotalLocalPayableAmount = 10500`，`LocalBalanceAmount = 10500`

1. 创建销售订单，产品金额 10000 元，运费 500 元

3. 审核出库单 A
4. 第二次出库 40% 数量，生成出库单 B
5. 审核出库单 B，自动生成两张应收款单

## 总结

本次修复解决了**运费重复计算**的核心问题：

1. ✅ **多次出库运费分配**：只有第一次出库包含运费，后续出库自动扣除运费
2. ✅ **应收款单总额计算**：移除了重复累加的 `ShippingFee`，确保与出库单总金额一致
3. ✅ **未核销金额准确性**：消除了虚增的运费部分，保证财务数据准确
4. ✅ **审计可追溯**：添加详细日志，记录运费调整过程

修复后，应收款单的未核销金额将准确反映实际应收金额，避免财务对账错误。
