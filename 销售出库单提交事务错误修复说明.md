# 销售出库单提交事务错误修复说明

## 问题描述

**错误信息**:
```
类型: System.InvalidOperationException
消息: 此 SqlTransaction 已完成；它再也无法使用。
堆栈: 
   在 System.Data.SqlClient.SqlTransaction.ZombieCheck()
   在 System.Data.SqlClient.SqlTransaction.Rollback()
   在 SqlSugar.AdoProvider.RollbackTran()
   在 RUINORERP.Repository.UnitOfWorks.UnitOfWorkManage.ForceRollback(ISqlSugarClient dbClient)
```

**发生场景**: 销售出库单提交时偶发出现

---

## 根本原因分析

### 1. 核心问题

在 [tb_SaleOutControllerPartial.cs](file://e:/CodeRepository/SynologyDrive/RUINORERP/RUINORERP.Business/tb_SaleOutControllerPartial.cs) 的审核流程中,**存在多处提前调用 `RollbackTran()` 并直接返回的情况**,但**没有正确清理事务上下文状态**。

具体问题点:
- **第520、529行**: 库存检查失败时调用 `RollbackTran()` 后直接 throw/return
- **第597、632行**: 负库存检查失败时调用 `RollbackTran()` 后返回
- **第745、758、785、801行**: 订单数量校验失败时调用 `RollbackTran()` 后返回  
- **第828、841、863行**: 其他业务校验失败时调用 `RollbackTran()` 后返回
- **第954行**: 订单更新失败时调用 `RollbackTran()` 后返回

这些地方的问题是:**手动回滚后,事务上下文的 Depth 可能未正确归零,或者 Transaction 对象已被 ADO.NET 标记为完成,但上下文仍认为事务活跃**。当后续操作(如UI层的提交后处理、异常捕获中的再次回滚)尝试访问或回滚事务时,就会抛出 "此 SqlTransaction 已完成" 异常。

### 2. 事务生命周期管理缺陷

原有的 `UnitOfWorkManage` 实现存在以下问题:

1. **缺少防御性检查**: 在调用 `RollbackTran()` 或 `CommitTran()` 时,没有验证事务对象是否仍然有效
2. **状态清理不及时**: `ResetTransactionState()` 只在特定条件下调用,可能导致事务上下文残留
3. **异常处理不完善**: 没有专门处理 "事务已完成" 异常的逻辑,导致这类正常情况被当作错误抛出
4. **日志不够详细**: 缺少事务ID、深度、调用栈等关键信息,难以追踪问题根源

### 主要改进

#### 1. RollbackTran() 方法增强

**新增功能**:
- ✅ **防御性检查**: 在回滚前验证事务对象和连接状态
- ✅ **异常过滤**: 专门捕获 `InvalidOperationException` (包含"已完成"或"Zombie"),将其视为正常情况
- ✅ **强制状态重置**: 在 finally 块中无条件调用 `ResetTransactionState()`,确保事务上下文被清理
- ✅ **详细日志**: 记录事务ID、深度、调用栈、操作结果等关键信息

**关键代码片段**:
```csharp
// 防御性检查：验证事务对象状态
if (dbClient.Ado.Transaction == null)
{
    _logger.LogWarning($"[Transaction-{context.TransactionId}] 事务对象已为空，跳过回滚");
    context.Status = TransactionStatus.RolledBack;
}
else
{
    // 检查事务是否已经完成（提交或回滚）
    var transactionConnection = dbClient.Ado.Transaction.Connection;
    if (transactionConnection == null || transactionConnection.State != System.Data.ConnectionState.Open)
    {
        _logger.LogWarning($"[Transaction-{context.TransactionId}] 事务连接已关闭或无效，跳过回滚");
        context.Status = TransactionStatus.RolledBack;
    }
    else
    {
        dbClient.Ado.RollbackTran();
        context.Status = TransactionStatus.RolledBack;
        _logger.LogInformation($"[Transaction-{context.TransactionId}] 事务回滚成功");
    }
}

// 捕获 "事务已完成" 异常
catch (InvalidOperationException invEx) when (invEx.Message.Contains("已完成") || invEx.Message.Contains("Zombie"))
{
    _logger.LogWarning(invEx, $"[Transaction-{context.TransactionId}] 事务已完成（可能已被其他地方回滚），忽略此异常");
    context.Status = TransactionStatus.RolledBack;
}
finally
{
    // 关键修复：无论成功与否，都要重置事务状态
    ResetTransactionState();
}
```

## 如何排查问题

### 1. 启用详细日志

确保日志配置中包含 `Information` 及以上级别:

```json
{
  "Logging": {
    "LogLevel": {
      "Default": "Information",
      "RUINORERP.Repository.UnitOfWorks": "Debug"
    }
  }
}
```

### 2. 搜索关键日志

当日志中出现错误时,搜索以下关键字:

- `[Transaction-` - 所有事务相关日志的前缀
- `事务已开启` - 事务开始
- `事务提交成功` / `事务回滚成功` - 正常结束
- `事务已完成` - 已处理的异常情况
- `事务对象已为空` / `事务连接已关闭` - 防御性拦截
- `长事务警告` - 耗时超过60秒的事务
- `RollbackTran 外层异常` / `CommitTran 外层异常` - 未预期的异常

### 3. 追踪完整事务生命周期

通过事务ID追踪完整流程:

```bash
# 示例：搜索特定事务ID的所有日志
grep "Transaction-abc123" logfile.txt
```

典型输出应该类似:
```
[时间] INFO  [Transaction-abc123...] 事务已开启，线程 ID: 12，调用栈：xxx
[时间] DEBUG [Transaction-abc123...] 回滚操作：Depth=1, ShouldRollback=False
[时间] DEBUG [Transaction-abc123...] 回滚操作：Depth=0, ShouldRollback=True
[时间] INFO  [Transaction-abc123...] 事务回滚成功
```

### 4. 识别问题模式

**模式1: 多次回滚**
```
[时间] INFO  [Transaction-xxx] 事务回滚成功
[时间] WARN  [Transaction-xxx] 事务对象已为空，跳过回滚  ← 第二次回滚被拦截
```
✅ 这是正常的,说明修复生效

**模式2: 提交后再次回滚**
```
[时间] INFO  [Transaction-xxx] 事务提交成功
[时间] WARN  [Transaction-xxx] 事务连接已关闭或无效，跳过回滚  ← 提交后的回滚被拦截
```
✅ 这是正常的,说明修复生效

**模式3: 深度不匹配**
```
[时间] WARN  [Transaction-xxx] 提交请求但事务深度为 -1，跳过提交
```
⚠️ 需要检查代码中是否有不平衡的 BeginTran/RollbackTran 调用

## 后续优化建议

### 1. 代码层面

**建议1: 统一异常处理**

在 Controller 层使用统一的 try-catch-finally 模式:

```csharp
public async Task<ReturnResults<T>> ApprovalAsync(T entity)
{
    try
    {
        _unitOfWorkManage.BeginTran();
        
        // 业务逻辑...
        
        _unitOfWorkManage.CommitTran();
        return ReturnResults<T>.Success(entity);
    }
    catch (Exception ex)
    {
        _unitOfWorkManage.RollbackTran(); // 即使已回滚,再次调用也是安全的
        _logger.LogError(ex, "审核失败");
        return ReturnResults<T>.Fail(ex.Message);
    }
}
```

**建议2: 减少手动回滚**

避免在业务逻辑中间手动调用 `RollbackTran()` 并返回,改为抛出异常让外层统一处理:

```csharp
// ❌ 不推荐
if (inventory.Quantity < requiredQty)
{
    _unitOfWorkManage.RollbackTran();
    return ReturnResults.Fail("库存不足");
}

// ✅ 推荐
if (inventory.Quantity < requiredQty)
{
    throw new BusinessException("库存不足");
}
```

### 2. 架构层面

**建议3: 引入 Saga 模式**

对于跨多个聚合根的复杂业务流程,考虑使用 Saga 模式管理分布式事务,避免长时间持有数据库事务。

**建议4: 读写分离**

将查询操作从写事务中分离,减少事务持有时间和锁竞争。
