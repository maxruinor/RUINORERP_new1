# 领料单审核死锁问题 - 最终修复方案

## 🔴 问题现状

即使经过多次优化，领料单审核仍然出现**双重事务超时**和**表级死锁**：

```
[Transaction-db3c3747] ⚠️ 事务超时! 配置超时=60秒, 实际运行=92.0秒
  调用方: <ApprovalAsync>d__39.MoveNext (领料单审核)
  
[Transaction-80f0ea6d] ⚠️ 事务超时! 配置超时=60秒, 实际运行=60.0秒
  调用方: <BatchRecordTransactions>d__39.MoveNext (库存流水记录)

🔒 关键发现：
- ApprovalAsync 在第 134 行开启事务后，仅 2 毫秒就停止活动（Last Activity）
- 但事务持续持有 92 秒才超时
- BatchRecordTransactions 在 121 秒后又尝试开启新事务
```

## 🔍 根本原因深度分析

### 1. 嵌套事务导致的死锁

#### 问题代码（修复前）

```csharp
// tb_InventoryTransactionControllerPartial.cs - BatchRecordTransactions
public async Task<bool> BatchRecordTransactions(...)
{
    bool inTransaction = _unitOfWorkManage.TranCount > 0;
    
    // ❌ 错误逻辑：如果不在事务中，就开启新事务
    if (useTransaction && !inTransaction)
    {
        await _unitOfWorkManage.BeginTranAsync();  // ← 开启新事务
        try
        {
            var rs = await dbClient.Insertable(validTransactions)
                .ExecuteReturnSnowflakeIdListAsync();
            await _unitOfWorkManage.CommitTranAsync();  // ← 提交新事务
            return rs.Count > 0;
        }
        catch (Exception ex)
        {
            await _unitOfWorkManage.RollbackTranAsync();  // ← 回滚新事务
            throw;
        }
    }
    else
    {
        // ✅ 在当前事务中执行
        var rs = await dbClient.Insertable(validTransactions)
            .ExecuteReturnSnowflakeIdListAsync();
        return rs.Count > 0;
    }
}
```

#### 问题分析

虽然代码检查了 `TranCount`，但存在以下问题：

1. **TranCount 可能不准确**：基于 `AsyncLocal<int>`，在某些异步场景下可能丢失上下文
2. **SqlSugar 底层事务对象残留**：即使 `TranCount == 0`，`dbClient.Ado.Transaction` 可能仍不为 null
3. **嵌套事务冲突**：如果外层事务已经锁定了某些表，内层事务再尝试锁定会导致死锁

### 2. 事务超时 92 秒的真正原因

从日志可以看出：

```
Created At: 2026-04-29 23:10:07.737
Last Activity: 2026-04-29 23:10:07.739  ← 仅 2 毫秒后就停止了！
Duration: 92.02 seconds                  ← 但持有了 92 秒
```

这说明：
- 事务在第 134 行开启后，执行到某个数据库操作就**卡住了**
- 最可能的原因是：**等待其他会话释放表锁**
- 可能是另一个并发操作也在访问 `tb_Inventory` 或 `tb_MaterialRequisition` 表

### 3. 死锁形成机制

```
时间线分析：

T0 (23:10:07.737): ApprovalAsync 开启事务
                   ├─ 锁定 tb_MaterialRequisition (MR_ID)
                   └─ 准备更新 tb_Inventory

T1 (23:10:07.739): 执行到某个数据库操作时卡住
                   └─ 等待 tb_Inventory 的锁被释放

T2 (23:10:xx.xxx): 另一个并发操作也在访问 tb_Inventory
                   └─ 形成循环等待 → 💀 死锁

T3 (23:11:47.737): ApprovalAsync 事务超时 (60秒)
                   └─ UnitOfWorkManage 强制回滚

T4 (23:12:08.843): BatchRecordTransactions 尝试开启新事务
                   ├─ 检测到残留事务对象 (Ado.Transaction != null)
                   └─ 再次超时...
```

## ✅ 最终修复方案

### 核心策略

**彻底禁止嵌套事务**：`BatchRecordTransactions` 永远不应该自己开启事务，必须依赖调用方的事务。

### 修复后的代码

```csharp
// tb_InventoryTransactionControllerPartial.cs - BatchRecordTransactions
public async Task<bool> BatchRecordTransactions(
    List<tb_InventoryTransaction> transactions, 
    bool useTransaction = true)
{
    // 验证输入
    if (transactions == null || !transactions.Any())
        return false;

    var validTransactions = transactions.Where(t => t.QuantityChange != 0).ToList();
    if (!validTransactions.Any())
        return false;

    // 设置交易时间
    var now = DateTime.Now;
    foreach (var tran in validTransactions)
    {
        tran.TransactionTime = now;
    }

    // ✅ P0修复：强制使用当前事务，禁止嵌套事务
    var dbClient = _unitOfWorkManage.GetDbClient();
    bool inTransaction = _unitOfWorkManage.TranCount > 0;
    
    if (!inTransaction)
    {
        // ⚠️ 警告：不在事务中执行批量插入
        _logger.LogWarning($"[BatchRecordTransactions] 警告：当前不在事务中，直接插入 {validTransactions.Count} 条库存流水记录");
        
        // 直接插入，不使用事务
        var rs = await dbClient.Insertable(validTransactions)
            .ExecuteReturnSnowflakeIdListAsync();
        return rs.Count > 0;
    }
    else
    {
        // ✅ 在当前事务中执行（推荐方式）
        var rs = await dbClient.Insertable(validTransactions)
            .ExecuteReturnSnowflakeIdListAsync();
        return rs.Count > 0;
    }
}
```

### 关键改进点

#### 1. 移除 `BeginTranAsync()` 调用

**修复前**：
```csharp
if (useTransaction && !inTransaction)
{
    await _unitOfWorkManage.BeginTranAsync();  // ❌ 开启新事务
    // ...
    await _unitOfWorkManage.CommitTranAsync();
}
```

**修复后**：
```csharp
if (!inTransaction)
{
    // ⚠️ 只记录警告，不开启新事务
    _logger.LogWarning("当前不在事务中，直接插入");
    // 直接执行插入
}
```

#### 2. 简化逻辑分支

**修复前**：两个分支（独立事务 vs 当前事务）
**修复后**：统一逻辑，只在当前事务中执行

#### 3. 添加警告日志

如果调用方没有开启事务，记录警告但不抛出异常，便于排查问题。

```csharp
// ✅ 在 UnitOfWorkManage 中添加长事务监控
private void InitializeTransactionTimeout(TransactionContext context, int? timeoutSeconds)
{
    var timeout = timeoutSeconds ?? _options.DefaultTransactionTimeoutSeconds;
    
    context.CancellationTokenSource = new CancellationTokenSource(TimeSpan.FromSeconds(timeout));
    context.CancellationTokenSource.Token.Register(() =>
    {
        _logger.LogError($"[Transaction-{context.TransactionId}] ⚠️ 事务超时! " +
                        $"配置超时={timeout}秒, 实际运行={context.Duration.TotalSeconds:F1}秒, " +
                        $"调用方={context.CallerMethod}");
        
        // ✅ 新增：记录详细的锁信息
        LogLockInformation(context);
    });
}

private void LogLockInformation(TransactionContext context)
{
    try
    {
        var dbClient = GetDbClient();
        var locks = dbClient.Ado.SqlQuery<dynamic>(@"
            SELECT 
                request_session_id,
                resource_type,
                resource_description,
                request_mode,
                request_status
            FROM sys.dm_tran_locks
            WHERE request_session_id = @@SPID
        ");
        
        _logger.LogWarning($"[Transaction-{context.TransactionId}] 当前持有的锁: {JsonConvert.SerializeObject(locks)}");
    }
    catch (Exception ex)
    {
        _logger.LogError(ex, "记录锁信息失败");
    }
}
```

## 📝 经验总结

### 避免嵌套事务的黄金法则

1. **单一事务原则**：一个业务操作只应该有一个事务边界
2. **事务传递**：子方法应该使用调用方的事务，而不是创建新事务
3. **明确职责**：
   - Controller 层：开启和提交事务
   - Service 层：在事务中执行业务逻辑
   - Repository 层：执行数据库操作

### 事务设计最佳实践

```csharp
// ✅ 推荐的事务模式
public async Task<Result> BusinessOperation()
{
    // 1. 事务外：预加载数据、验证参数
    var data = await PreloadData();
    
    // 2. 开启事务（Controller 层）
    await unitOfWork.BeginTranAsync();
    
    try
    {
        // 3. 事务内：执行业务逻辑（Service 层）
        var result = await ProcessBusiness(data);
        
        // 4. 提交事务
        await unitOfWork.CommitTranAsync();
        return Success(result);
    }
    catch
    {
        // 5. 异常回滚
        await unitOfWork.RollbackTranAsync();
        throw;
    }
}

// ✅ Service 层：不管理事务，只执行业务逻辑
public async Task<Result> ProcessBusiness(Data data)
{
    // 直接使用 unitOfWork.GetDbClient() 执行操作
    // 不需要 BeginTran/CommitTran
    await repository.Update(data);
    return Success();
}
```
