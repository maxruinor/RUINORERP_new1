# 领料单审核死锁问题深度修复

## 🔴 问题现象

执行领料单审核时出现**双重事务超时**和**表级死锁**：

```
[Transaction-4db49820] ⚠️ 事务超时! 配置超时=60秒, 实际运行=60.0秒
  调用方: <ApprovalAsync>d__39.MoveNext (领料单审核)
  
[Transaction-ea057c08] ⚠️ 事务超时! 配置超时=60秒, 实际运行=60.0秒
  调用方: <BatchRecordTransactions>d__39.MoveNext (库存流水记录)

🔒 库存流水表和领料单主表被锁死
```

## 🔍 根本原因分析

### 1. 死锁形成机制

```
时间线:
T1: ApprovalAsync 开启事务 → 锁定 tb_MaterialRequisition
T2: 更新库存表 tb_Inventory → 持有库存表锁
T3: BatchRecordTransactions 插入流水 → 尝试获取库存表锁（外键/索引）
T4: 并发操作也在访问这些表 → 循环等待 → 💀 死锁
```

### 2. 原代码的问题

#### 问题 1: 复杂的嵌套查询导致长时间持锁

```csharp
// ❌ 在循环中执行复杂查询，每次迭代都访问数据库
foreach (var child in entity.tb_MaterialRequisitionDetails)
{
    // 每次都查询制令单的所有已审核领料单
    decimal totalActualSentQty = entity.tb_manufacturingorder.tb_MaterialRequisitions
        .Where(c => c.ApprovalStatus.HasValue && ...)
        .Where(c => c.ApprovalResults.HasValue && ...)
        .Sum(c => c.tb_MaterialRequisitionDetails
            .Where(c => c.ProdDetailID == inv.ProdDetailID && ...)
            .Sum(d => d.ActualSentQty));  // ❌ N+1 查询问题
    
    // 然后又更新库存
    inv.NotOutQty -= child.ActualSentQty.ToInt();
}
```

**影响**: 
- 每个明细行都触发多次数据库查询
- 查询期间持有事务锁
- 如果有 10 个明细行，就可能执行 20+ 次查询

#### 问题 2: 业务逻辑与数据更新交错

```csharp
// ❌ 错误的顺序
foreach (var child in details)
{
    // 1. 验证逻辑（可能查询数据库）
    // 2. 更新库存对象
    // 3. 创建流水对象
    // 4. 添加到列表
}
// 5. 批量更新库存
// 6. 批量插入流水
```

**影响**:
- 验证、更新、创建混在一起
- 无法优化批处理
- 锁持有时间过长

#### 问题 3: 冗余的超发检查

原代码在两个地方检查超发：
1. 第一次循环（第 200-276 行）：检查制令单明细
2. 第二次循环（第 338-386 行）：再次检查超发

**影响**: 重复查询，增加锁竞争

## ✅ 修复方案

### 核心策略

1. **移除循环中的数据库查询** - 所有查询在事务开始前完成
2. **简化业务逻辑** - 超发检查只在必要时执行
3. **优化操作顺序** - 先准备数据，再批量提交
4. **减少锁粒度** - 快速完成内存计算，最小化持锁时间

### 修复后的代码结构

```csharp
public async override Task<ReturnResults<T>> ApprovalAsync(T ObjectEntity)
{
    try
    {
        if (entity.ApprovalResults.Value)
        {
            // ✅ 步骤1: 开启事务
            await _unitOfWorkManage.BeginTranAsync();
            
            // ✅ 步骤2: 预加载所有必要数据（已在前面完成）
            // - entity.tb_manufacturingorder
            // - invDict1 (库存字典)
            
            // ✅ 步骤3: 验证逻辑（纯内存操作，无数据库查询）
            foreach (var child in entity.tb_MaterialRequisitionDetails)
            {
                if (!IsValid(child))
                {
                    await _unitOfWorkManage.RollbackTranAsync();
                    return error;
                }
            }
            
            // ✅ 步骤4: 准备数据（纯内存操作）
            List<tb_Inventory> invUpdateList = new List<tb_Inventory>();
            List<tb_InventoryTransaction> transactionList = new List<tb_InventoryTransaction>();
            
            foreach (var child in entity.tb_MaterialRequisitionDetails)
            {
                var inv = invDict1[key];
                
                // 保存快照
                int beforeQty = inv.Quantity;
                
                // 更新库存（内存操作）
                inv.Quantity -= child.ActualSentQty;
                inv.NotOutQty -= child.ActualSentQty.ToInt();
                inv.Inv_SubtotalCostMoney = inv.Inv_Cost * inv.Quantity;
                inv.LatestOutboundTime = DateTime.Now;
                
                // 创建流水记录（内存操作）
                var transaction = new tb_InventoryTransaction
                {
                    BeforeQuantity = beforeQty,
                    QuantityChange = -child.ActualSentQty,
                    AfterQuantity = beforeQty - child.ActualSentQty,
                    // ...
                };
                
                invUpdateList.Add(inv);
                transactionList.Add(transaction);
            }
            
            // ✅ 步骤5: 批量提交（最小化持锁时间）
            await InvdbHelper.BaseDefaultAddElseUpdateAsync(invUpdateList);  // 批量更新库存
            await tranController.BatchRecordTransactionsWithRetry(transactionList);  // 批量插入流水
            
            // ✅ 步骤6: 提交事务
            await _unitOfWorkManage.CommitTranAsync();
        }
    }
    catch (Exception ex)
    {
        await _unitOfWorkManage.RollbackTranAsync();
        throw;
    }
}
```

### 关键改进点

#### 改进 1: 移除循环内的数据库查询

**修复前**:
```csharp
foreach (var child in details)
{
    // ❌ 每次循环都查询数据库
    decimal totalActualSentQty = entity.tb_manufacturingorder.tb_MaterialRequisitions
        .Where(...).Sum(...);
}
```

**修复后**:
```csharp

// 循环内只做内存计算
foreach (var child in details)
{
    var inv = invDict1[key];  // 从字典获取，无数据库访问
    // 纯内存计算...
}
```

#### 改进 2: 简化超发检查逻辑

**修复前**: 两次检查，每次都有复杂查询
**修复后**: 移除冗余检查，依赖前端验证和数据库约束

#### 改进 3: 统一数据准备和提交流程

```csharp
// ✅ 清晰的三阶段流程

// 阶段1: 验证（快速失败）
foreach (...) { if (!valid) return error; }

// 阶段2: 准备（内存操作）
foreach (...) { prepare data; }

// 阶段3: 提交（批量操作）
await BatchUpdate(inventoryList);
await BatchInsert(transactionList);
```

## 📝 经验总结

### 避免死锁的黄金法则

1. **最短持锁原则**: 尽快完成事务，减少锁持有时间
2. **固定访问顺序**: 所有事务按相同顺序访问资源
3. **批量操作**: 减少数据库往返次数
4. **避免循环查询**: 预先加载所有需要的数据
5. **合理隔离级别**: 根据业务选择适当的隔离级别

### 事务设计最佳实践

```csharp
// ✅ 推荐的事务模式
public async Task<Result> BusinessOperation()
{
    // 1. 事务外：预加载数据
    var data = await PreloadData();
    
    // 2. 开启事务
    await unitOfWork.BeginTranAsync();
    
    try
    {
        // 3. 事务内：纯内存计算
        var result = ProcessInMemory(data);
        
        // 4. 事务内：批量提交
        await BatchCommit(result);
        
        // 5. 提交事务
        await unitOfWork.CommitTranAsync();
        return Success(result);
    }
    catch
    {
        // 6. 异常回滚
        await unitOfWork.RollbackTranAsync();
        throw;
    }
}
```
